# Folder Template — Reuse This Handbook Structure in Any Project

> **TL;DR:** Copy this folder skeleton into any new project, fill the files in numbered order, keep
> every file ≤ ~250 lines with a TL;DR block on top, and link each summary back to its deep-dive
> source. The result: anyone can understand the system in 30 minutes and develop a page that fits.

---

## The Skeleton (copy this)

```
docs/system/
├── README.md                      ← THE INTERFACE — brief summary, folder map, reading paths by role, non-negotiable rules
├── 00_overview/
│   ├── SYSTEM_OVERVIEW.md         ← what it is, actors, features, architecture diagram
│   └── TECH_STACK.md              ← stack tables per layer + ports + repo layout
├── 01_flow/
│   ├── FLOW_INDEX.md              ← flow list + intersection map
│   ├── <ACTOR>_FLOW.md            ← one file per user journey (sequence diagram + step table)
│   ├── <ENTITY>_STATE_MACHINE.md  ← one per core entity with statuses
│   └── PAYMENT_FLOW.md            ← (or any external-integration flow)
├── 02_spec/
│   ├── API_SPEC.md                ← every endpoint, table format only
│   ├── DB_SCHEMA.md               ← tables + ER diagram
│   ├── ERROR_SPEC.md              ← error format + code table + FE mapping
│   └── BUSINESS_RULES.md          ← RBAC + domain rules summary (links to authority doc)
├── 03_be/
│   ├── BE_TECH_SUMMARY.md         ← layers, patterns, one worked request example
│   ├── BE_CODE_SUMMARY.md         ← package map, route table, "add endpoint" checklist
│   ├── REDIS_CACHE.md             ← keys, TTLs, invalidation triggers (or CACHING.md)
│   └── REALTIME_SSE.md            ← realtime architecture (SSE/WS/polling)
├── 04_fe/
│   ├── FE_TECH_SUMMARY.md         ← stack, folder conventions, route map
│   ├── FE_CODE_SUMMARY.md         ← store/hook/component inventory tables
│   ├── STATE_MANAGEMENT.md        ← strict decision rules: server/global/local/form/URL state
│   ├── LOADING_PATTERNS.md        ← loading/error/empty conventions
│   ├── DESIGN_SYSTEM.md           ← tokens + shared component catalog
│   └── DATA_COMMUNICATION.md      ← FE↔BE, page↔page, realtime, storage keys
└── 05_dev_guide/
    ├── NEW_PAGE_GUIDE.md          ← phased guide to build a page that fits
    └── FOLDER_TEMPLATE.md         ← this file
```

## Rules That Make It Work

| Rule | Why |
|---|---|
| **README is the only entry point** | One door in; it routes by role and task |
| **TL;DR blockquote on top of every file** | 30-second understanding before committing to read |
| **Tables + mermaid > prose** | Scannable; diffs stay readable |
| **≤ ~250 lines per file** | A file that fits one screen-scroll gets read; a 1000-line doc doesn't |
| **Every file ends with "Deep Dive Sources"** | Summary here, authority elsewhere — no duplication drift |
| **One fact, one home** | If two files state the same rule, one links to the other |
| **Code wins** | When summary and code disagree, fix the summary — verify against code when writing |
| **Numbered folders** | Forces reading order: overview → flows → contracts → implementation → dev guide |

## How to Fill It for a New Project (order matters)

1. **00_overview** — write SYSTEM_OVERVIEW from the BRD/requirements; TECH_STACK from the repo.
2. **01_flow** — one file per user journey. Draw the sequence diagram first; the step table falls out of it.
3. **02_spec** — generate from contracts/schema if they exist; otherwise write them here first (this becomes the contract).
4. **03_be / 04_fe** — write TECH_SUMMARY (rules) before CODE_SUMMARY (inventory). Inventory must be generated by reading real code, not memory.
5. **05_dev_guide** — write NEW_PAGE_GUIDE last; it references everything above.
6. **README** — write last of all; it indexes what actually exists.

**Maintenance:** whenever a flow, contract, or convention changes, update the matching file here in the same PR. A stale handbook is worse than none.

## Adapting to Different Stacks

- No Redis? → rename `REDIS_CACHE.md` to `CACHING.md` (or delete and remove from README map).
- No realtime? → drop `REALTIME_SSE.md`.
- Different state library? → `STATE_MANAGEMENT.md` keeps the same shape: a strict "data kind → home" decision table is stack-agnostic.
- Mobile app instead of web FE? → `04_fe/` becomes `04_app/`; DESIGN_SYSTEM + STATE_MANAGEMENT + DATA_COMMUNICATION still apply 1:1.

The invariant across projects: **overview → flows → contracts → BE summary → FE summary → dev guide**, each ≤ 250 lines, each with TL;DR + deep-dive links.
