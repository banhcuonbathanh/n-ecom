## 🔴 RAISE-MY-VOICE headline findings

**No 🔴 doc-vs-code contradiction was found** — the doc-set matches the code. That is the honest
headline: this is a clean, source-traced doc-set.

**However, two real CODE bugs (already documented in `COMBO_BUGS.md`) persist on this branch and are
re-confirmed by this audit.** They are code defects, *not* doc drift — the doc describes them
correctly:

1. **Bug 1 — unavailable-combo UI is dead code.** The page renders a "Hết hàng" badge
   (`page.tsx:122-126`) and a disabled "Combo tạm hết" CTA (`page.tsx:180-185`), but `GET /combos` is
   served by `ListCombosAvailable` whose SQL is `WHERE is_available = 1 AND deleted_at IS NULL`
   (`be/internal/db/products.sql.go:387`). So an unavailable combo never reaches the wire → the combo
   that renders **always** has `is_available === true` → both branches can never fire. A sold-out combo
   shows "Không tìm thấy combo." instead. Severity 🟡 (dead code / misleading not-found).

2. **Bug 2 — a combo sub-item shows its raw UUID as the name.** Sub-item names/prices are resolved by
   joining `combo_items` against `GET /products` (`page.tsx:33`), but that endpoint
   (`ListProductsAvailable`, `products.sql.go:469`) also excludes unavailable products. When a combo
   references a since-unavailable dish, the FE map misses and falls back to the raw `product_id`:
   `product_name: productMap.get(ci.product_id)?.name ?? ci.product_id` (`page.tsx:45`), with
   `unit_price` undefined. The user sees a UUID line in "Gồm có". Severity 🟡 (user-visible but
   conditional).

> Both are tracked in [`COMBO_BUGS.md`](COMBO_BUGS.md) and logged in the
> `07_business_logic` Decision Log. They are **not** on `MASTER_TASK.md` yet — a fix must be registered
> + ALIGNed first (CLAUDE.md).
