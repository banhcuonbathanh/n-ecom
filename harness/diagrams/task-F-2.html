Target architecture — what exists when F-2 is done

Five containers on one compose network. Only two ports ever reach your machine: :80 (Caddy, the one entry door) and :3306 (MySQL, so you can poke the database with a local client). Backend and frontend are hidden behind Caddy — the same shape production will have.

Solid arrows = live request paths in F-2. Dashed grey arrows = connections that exist in compose but carry no traffic yet — F-3 (migrations) is the first task to actually talk to MySQL. Green ✓ lines are the compose healthchecks that gate startup order (section 4).

Startup order — healthchecks gate the chain

depends_on … condition: service_healthy means services come up in a guaranteed order instead of racing each other. The same green ✓ checks from the architecture diagram drive this chain:

1 · databasesmysql + redis start,
healthchecks must pass
→
2 · appsbe + fe start
(be waits for the DBs)
→
3 · door openscaddy starts once
be + fe are healthy
→
4 · proofcurl /api/health → 200
localhost/ renders

Why bother in F-2: from F-3 on, the backend will connect to MySQL at boot. Getting the ordering right now means F-3 never sees a "connection refused because MySQL was still starting" flake.

5

Hot reload — the edit loop we live in every day after this

The dev images do not bake the code in — they bind-mount the source folders. Saving a file on your machine is instantly visible inside the container, and a watcher restarts/reloads the app:

you save a filebe/…/main.go
or fe/src/…/page.tsx
→
bind mountthe container sees the
change immediately
→
watcher reactsGo: air recompiles + restarts
Next: HMR patches the page
→
browserrefresh (BE) or
auto-update (FE) — seconds

No docker compose build in the daily loop — rebuilds are only needed when dependencies change (a new Go module, a new npm package).

8

Decisions taken — defaults; say so if you want different

  • MySQL + Redis join compose now, even though nothing connects to them until F-3 — so F-3 starts from a running, healthchecked database instead of more plumbing.
  • Dev-only images this task. Production builds (multi-stage, small images) arrive with CI in F-4.
  • Only Caddy is reachable from the browser — BE and FE hide behind it, same shape as production. MySQL's 3306 is the one convenience exception, for your local DB client.
  • Pinned versions: golang 1.25 · node 22 LTS · mysql 8 · redis 7 — no latest tags, so the stack behaves the same on any machine and on any day.
  • Health endpoint lives at /api/health (not /health) so it travels through Caddy like every real API route — the check proves the whole path, not just the process.
9

Risks I'm watching — each with its fallback

⚠️ First start is slow. The first compose up downloads ~1–2 GB of images.Fallback: none needed — one-time cost, not a failure. Later starts are seconds.
⚠️ Port collisions. Port 80 or 3306 may already be busy on your machine.Fallback: alternate host ports, documented in ENVIRONMENT.md so there is exactly one place to look.
⚠️ Docker bind mounts on macOS can make hot reload sluggish.Fallback: run BE/FE natively and keep compose for the databases only — documented, and you'd get a report line if we switch.
