Backend State & Data Design

How every piece of backend data gets exactly one owner, how a request flows handler → service → repository → MySQL, where transactions begin, and how failures become the one error envelope. Source of truth: harness/BE_STATE.md — this page is the visual companion. Stack: Go · Gin · sqlc · MySQL · Redis.

01Four kinds of state, four owners — and one banned

The BE twin of the FE's five-kinds rule. Data living in two places (stock in a struct and in MySQL) is how overselling happens — banned by rule.

BUSINESS

Business state

MySQL via repository products · carts · orders · users · stock touch from a handler; keep in Redis
CACHE

Cache state

Redis via platform/cache catalog cache · rate counters (A-1) store what MySQL can't rebuild
REQUEST

Request state

context.Context request id · auth claims · deadline stash in a global / outlive the request
CONFIG

Config state

platform/config at boot DSNs · JWT secret · ports · TTLs os.Getenv elsewhere; mutate after boot
BANNED

In-process mutable

— no owner, no home package vars holding carts / stock / sessions the API is stateless — kill -9 loses nothing
Would losing it lose a business fact?→ MySQL via repository
no ↓
Rebuildable, and only there to make reads fast?→ Redis via platform/cache
no ↓
Scoped to one request (who / trace / deadline)?→ context.Context
no ↓
Known at boot, never changes?→ platform/config
no ↓
Otherwise it's local→ a local variable
Allowed in-process state is immutable wiring only — router, parsed config, injected deps. A package var that changes after boot and would be missed by a customer belongs in MySQL or Redis.

02Request flow — one straight line

A handler calls exactly ONE service method; orchestration of several repos is a service's job (checkout is the worked example). sqlc row types never leak past the service.

03Transactions & consistency

Services own Begin/Commit; repositories never do. One tx per request, never nested. Stock is only ever mutated inside a locked transaction.

OperationTxInside the transaction
place order (CC-4)✅lock lines SELECT … FOR UPDATE → check stock → decrement → insert order + lines w/ price snapshot → clear cart · any line short ⇒ rollback, INSUFFICIENT_STOCK
cancel order (O-2)✅re-check status ∈ {placed, confirmed} → set cancelled → restore stock · moved on ⇒ rollback, CANCEL_WINDOW_CLOSED
cart merge on login (A-3)✅sum quantities per product, cap at stock → delete guest cart (rule 5)
cart add / update qty (CC-2)✅read stock → upsert line capped at stock — check + write atomically
all reads (lists, details)—single statements; never open a tx to read

04Error codes — the envelope's producing side

Services return typed domain errors → platform/errs writes {"error":{code,message,details}}. The FE branches on exactly these codes (FE_STATE.md §4) — this table is the other half of that contract. Codes live in domain/constants.go, the ONE constants file.

codeHTTPRaised byWhen
VALIDATION_FAILED400 / 422handler · service400 unparseable · 422 field rules; details[] = {field, issue}
UNAUTHORIZED401auth middlewaremissing / invalid / expired JWT
FORBIDDEN403serviceauthenticated but not the owner
NOT_FOUND404servicefrom sql.ErrNoRows — entity doesn't exist
CONFLICT409serviceduplicate email; generic state conflict
INSUFFICIENT_STOCK409servicestock cap / atomic decrement failure (rule 3)
CANCEL_WINDOW_CLOSED409servicecancel after shipped (rule 2)

Never in the envelope: SQL strings, stack traces, Go error text. Internals go to the request log with the request id. 5xx log at error level with stack; 4xx at info — expected traffic, not incidents.

05Validation & identity — who checks what

TIER 1 · HANDLER

Shape validation

  • Gin binding + validator tags: types, required, min/max, enum membership
  • fails fast: VALIDATION_FAILED + details[] before any service call
  • same limits as FE Zod (min qty 1 ⇔ min=1)
TIER 2 · SERVICE

Business validation

  • anything needing state or rules: stock caps, cancel window, duplicate email
  • never in handlers, never in SQL triggers
  • BE is the authority — FE validation is UX, BE validation is law
IDENTITY

Who is asking

  • JWT (httpOnly cookie) verified by middleware → claims into context — the only writer
  • handlers read via auth.UserID(ctx), pass userID to services as an explicit parameter
  • guest cart token cookie: same pattern — handler reads, service receives a parameter
  • bcrypt in the accounts service; plaintext never logged
Cache-aside lives in the service (policy: ARCHITECTURE.md §4): Get → miss ⇒ repository → Set w/ TTL. Writes Del their keys in the same method. Redis down ⇒ log + fall through to MySQL — never a 5xx.

07Hard BE rules — violations are bugs

  1. Layering is strict: handler → service → repository → db — no skips, no reverse imports.
  2. Business facts live in MySQL only; no package-level mutable business state — FLUSHALL and kill -9 must never lose a fact.
  3. Every handler error exits via platform/errs; codes from domain/constants.go; internals never leak into the envelope.
  4. Transactions begin/commit in services only; stock mutations always inside a locked tx.
  5. Handlers validate shape, services validate business — never the other way.
  6. os.Getenv only in platform/config; the redis client only in platform/cache.
  7. Identity (userID, cart token) reaches services as explicit parameters — never dug out of context.
  8. A response-DTO change updates fe/src/lib/api/types.ts in the same scope contract; a new error code updates the §04 table too.
