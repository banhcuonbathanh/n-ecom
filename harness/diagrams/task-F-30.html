3

The centrepiece — combo expansion

The one algorithm nobody can improvise. Two request lines become five order_items rows; every name and price is snapshotted server-side.

product line  → 1 standalone row      name/price snapshot · toppings_snapshot = real values

combo line    → 1 header row          combo_id set · unit_price = 0 (label row, prevents double-count)
                + N sub-item rows     from the DB template, never the body
                                      quantity = template.qty × line.qty      ← multiply, don't copy
                                      unit_price = product price (the real charge lives here)
                                      combo_ref_id = header.id

canh          → never inside a combo. FE sends it standalone; seeded templates carry no canh row.

Trust boundary first. source and table_id come from the guest JWT — the body’s copies are assertions to check (403 on disagreement), never inputs. status is BE-set, created_by stays NULL, and the order number is allocated in-transaction.

What the reconciliation found

🚨 F27 — online orders are unownable. The table-less guest JWT identifies a guest, but orders has no column to store that identity (DB_SCHEMA §4.3), so every ownership gate fails for table_id IS NULL orders — a guest cannot read the order they just placed. This is exactly the old-system bug OVERALL_PLAN §3.4 promises to fix. Plan requests orders.guest_id CHAR(36) NULL in migration 004; the amendment is not written here — DB_SCHEMA.md owns it. Blocks the online path of the order-create slice, nothing before it.
