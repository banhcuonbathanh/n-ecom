The components — and how they stay aligned

Five containers, four backend layers, five domains, one API client — this page names every component, what it may and may not do, how the frontend and Redis fit in, and the eight gates that keep the structure from drifting as the build grows. Source of truth: harness/ARCHITECTURE.md.

Runtime map — 5 containers, one entry door

Everything the browser can reach goes through Caddy. Behind it, only the Go API touches the data stores; the frontend's entire world is /api/* over HTTP.

ComponentResponsibilityTalks toNever does
1Caddy :80Single public entry; routes /api/* → Go, the rest → Next.jsGo API, Next.jsbusiness logic, auth
2Go API :8080ALL business logic, validation, auth, the error envelopeMySQL, Redisrendering HTML
3Next.js :3000Rendering + UX state; SSR fetches from the Go APIGo API onlytouching MySQL/Redis, business rules
4MySQL 8 :3306The only source of truth — every business fact——
5Redis 7 :6379Disposable accelerator: cache + ephemeral counters—holding data not rebuildable from MySQL

Two hard consequences: the frontend never talks to MySQL or Redis — and redis-cli FLUSHALL must never lose a business fact. If losing a key loses data, the data was in the wrong store: that's a bug.

Inside the backend — 4 layers × 5 domains + platform packages

Every request crosses the same four layers, top to bottom, no skipping. A lower layer importing a higher one — or a handler reaching straight into a repository — is a bug by definition (PLAN.md §Architecture rules).

handlerMay: parse/validate request shape, call ONE service, write response/envelope. Never: business rules, SQL, Redis.
serviceMay: business rules (stock, cancel window, merge), orchestrate repos, use the Cache interface. Never: parse HTTP, write SQL.
repositoryMay: sqlc-generated + hand-written MySQL access. Never: business rules, HTTP, Redis.
dbSQL sources: db/queries/*.sql + numbered migrations.

FE ↔ BE — one client, one envelope, mirrored types

RuleDetail
One API clientfe/src/lib/api/ is the only module calling fetch. Components import typed functions, never raw fetch.
Two URLs, one switchBrowser → relative /api/* (Caddy routes it); SSR → internal http://be:8080. The switch lives inside the client, nowhere else.
One error shapeThe client parses {"error":{code,message,details}} in one place and throws a typed ApiError. UI switches on code, never on message strings.
Mirrored DTOsfe/src/lib/api/types.ts mirrors the Go DTOs 1:1. A BE task that changes a DTO updates this file in the same scope contract — that is the sync mechanism.
State handlingcache keys, loading/error policy, stores → owned by FE_STATE.md (F-5), not repeated here.

Worked example — “Add to cart” on a cached catalog

  1. Product page calls the API client: POST /api/cart/items. Caddy hands it to the Go API.
  2. handler checks the JSON shape → cart service checks stock (business rule) → repository writes the cart line to MySQL. Redis is not involved — carts are business facts.
  3. On failure the handler answers through platform/errs: {"error":{"code":"OUT_OF_STOCK",…}} — the FE client turns it into a typed error, the UI switches on the code.
  4. Meanwhile a product-detail read goes: catalog service → Cache.Get("v1:catalog:product:42") → hit? return · miss? repository → MySQL → Cache.Set (60 s TTL) → return.
  5. TanStack Query updates its cart cache; badge and mini-cart re-render by themselves (see FE_STATE.md).

Redis policy — disposable accelerator, never a record proposal

Redis is in the stack since F-1 but its job was never pinned down. This is the proposal — say so before C-2 starts if you want it different.

Redis IS for

v1 — nothing else

  • Catalog read cache — cache-aside in the catalog service: product detail + list pages, 60 s TTL (lands with C-2)
  • Rate limiting on auth endpoints — arrives with A-1, not before

Redis is NOT for

explicit non-uses

  • Cart — MySQL with a cookie token (PLAN rule 5)
  • Sessions — JWT is stateless
  • Orders / stock — transactional, MySQL only
RuleDetail
Accessonly platform/cache imports the redis client; services use the Cache interface; handlers and repositories never see Redis
Keysv1:{domain}:{entity}:{id} — e.g. v1:catalog:product:42, v1:catalog:list:{filterhash}
ExpiryTTL-first (60 s catalog) + explicit Del on any write path that mutates the cached entity
DegradationRedis down ⇒ log + fall through to MySQL. A cache outage slows the shop, never breaks it — every cached path must work with the cache stubbed out.

Alignment — 8 gates, each firing at a fixed moment

Alignment is not good intentions; it is gates wired into the task loop (READ → PLAN → REPORT → IMPLEMENT → SELF-REVIEW → VERIFY → CHECKPOINT). Skipping one is a Hard-Rule violation, not a style choice.

Per-task checklist (run at SELF-REVIEW, every task)

  • Every changed file was in the scope contract.
  • No import crosses a layer arrow in §2 (grep handlers for repository|redis, repos for gin|redis).
  • Errors leave handlers only via platform/errs.
  • New enum/constant landed in the ONE constants file per side, nowhere else.
  • If a DTO changed → types.ts changed in the same diff.
  • If code contradicted a harness doc → doc fixed, noted in STATE.md.
