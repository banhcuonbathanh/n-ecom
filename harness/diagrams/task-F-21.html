01

Objective

Reconcile the reference implementation of /admin/ingredients against our harness rules and emit a build-ready plan set per PAGE_PLAN_GUIDE.md. The reference BE doc is a trace of the old system's actual code — including five flaws its own authors flagged. The value of this task is the reconciliation, not the transcription.

Why this page matters. stock_movements is an append-only ledger and ingredients.current_stock is its running total. If those two ever disagree, inventory is fiction. The reference let them disagree in three separate ways. This plan closes all three.

05

Reference defects designed out

The big one (D-1). The reference inserted the movement row and then issued a separate UPDATE on current_stock — no transaction. It also wrote initial_stock straight into current_stock at create time with no movement row at all. Both mean the ledger and the count can disagree, and the second one guarantees they disagree from day one. Our version runs one tx with the ingredient row locked, and creates an opening 'in' movement. The invariant Σ movements == current_stock becomes testable.
#Reference findingCountermeasure

D-2current_stock reachable as a plain update fieldRejected key — stock has one door

D-4Delete had no in-use guard → MySQL 1451 → 500Service checks product_ingredients → clean 422. Load-bearing: the FK is CASCADE, so without the guard a delete silently destroys recipe rows

06

Flags for the boss

⚠️ F-1 — schema amendment requested. Add UNIQUE(name) to ingredients, with the soft-delete rename rule from DB_SCHEMA.md §4.4 (name → "<name>#deleted-<id>"). DB_SCHEMA.md owns tables, so this plan requests rather than declares it. Until it is ratified, the 409 duplicate-name path does not exist and the reference's duplicate toast must be cut.
⚠️ F-2 — over-draw default is flippable. Ruling 8 rejects a below-zero out with VALIDATION_FAILED and says "lock in when AD opens". This plan builds the reject. If real kitchen practice is "count later, log now", say so before AD-INV-4 and it becomes a clamp + warning instead.
⚠️ F-3 — admin palette still unresolved. The live admin shell used orange/dark while our design tokens are neutral. This plan's mockup uses the neutral F-7 tokens per PAGE_PLAN_GUIDE §7. Same open question as the other admin plans — one answer should cover all of them.
❓ C-1 — batch vs ingredient dating. The schema carries one import_date per ingredient row, so re-importing overwrites it and silently moves expiry_date for stock that is physically older. Real batch tracking would need per-movement dating. Out of scope here, but it should be ruled on before /admin/storage designs expiry reporting.
