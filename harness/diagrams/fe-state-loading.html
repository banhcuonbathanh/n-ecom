Frontend State & Loading Design

How every piece of FE state gets exactly one owner, how data flows through the one API client, and what users see while it loads or fails. Source of truth: harness/FE_STATE.md — this page is the visual companion. Stack: Next.js App Router · TanStack Query · Zustand · RHF/Zod.

01Five kinds of state, five owners

Every piece of state lives in exactly one place. Duplicating it (cart badge in a store and in the query cache) is the classic source of stale-UI bugs — banned by rule.

SERVER

Server state

TanStack Query products · categories · cart · orders · me copy into Zustand
URL

URL state

searchParams / route params category filter · page · search query · product id mirror in a store
FORM

Form state

React Hook Form + Zod address · login/register · quantity stepper control inputs via Zustand
CLIENT/UI

Client state

Zustand (shared) · useState (local) cart drawer · toasts · mobile nav put fetched data here
SESSION

Session state

httpOnly JWT cookie + useMe() "who am I" · logged-in header JWT in localStorage/store

04Loading & error — three tiers, never stacked spinners

TIER 1 · ROUTE

loading.tsx / error.tsx

Every segment ships both. Skeleton mirrors the real layout (grid skeleton for /products, detail skeleton for /products/[id]). Never a centered global spinner. error.tsx offers retry via reset().

TIER 2 · QUERY

Component states

  • isPending (no data yet) → in-place skeleton
  • isFetching (has data) → keep old data visible, subtle hint, no layout jump
  • empty result → dedicated empty state + clear-filter action
  • error → inline message + Retry (refetch()), shell stays alive
TIER 3 · MUTATION

Inline pending

  • firing button: disabled + spinner, label kept ("Adding…")
  • failure → toast with envelope message
  • field errors: envelope details[] → RHF setError — same UI as Zod errors
  • branch on ApiError.code, never message text
Optimistic updates: cart only. onMutate snapshots → apply locally → onError rollback + toast → onSettled invalidate. Checkout and orders are always pessimistic — stock decrement is atomic on the server (rule 3). Retry: queries 2× (never on 4xx) · mutations 0×.

06Hard FE rules — violations are bugs

  1. Server data is read only via a hook in queries/ — no fetch in components, no API data in Zustand.
  2. Filter / page / search live in the URL — back button and shared links reproduce the exact view.
  3. Every route segment ships loading.tsx + error.tsx with layout-shaped skeletons.
  4. Optimistic updates for cart mutations only; checkout/orders always pessimistic.
  5. Branch on ApiError.code, never on message text.
  6. Query keys come from queries/keys.ts only.
  7. A new Zustand slice requires a note in FE_STATE.md first (expected v1: ui only).
Snapshot, not a source of truth — on any conflict harness/FE_STATE.md and PLAN.md win. Server/client split: RSC prefetch + HydrationBoundary for catalog SEO; 'use client' pushed to the leaves (buttons, forms, badge) — a page is never wholly client.
