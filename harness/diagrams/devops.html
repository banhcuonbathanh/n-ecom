05

Two-stage go-live

Stage A — owner's Mac, shop LAN

  • Real stack, real phones, http://<mac-ip> through Caddy
  • DHCP reservation for the Mac; firewall must allow Docker
  • Proves: full order loop, seeds, smoke tests
  • Skips: public TLS, CI deploy path (webhooks are P-phase anyway)

Stage B — VPS + domain + HTTPS

  • 2 vCPU / 2 GB, Singapore region · DNS A record before first start
  • Fresh secrets on the VPS — never reuse Stage A values
  • Pipeline-only deploys (rule D8) · nightly backups + restore drill
  • Prove rollback once with a deliberately broken commit before announcing
06

The 8 hard rules (D-rules)

IDRule (short form — full text in DEVOPS.md §10)
D1NEXT_PUBLIC_API_BASE_URL is baked at FE build time and points through Caddy, never :8080. Changing it = rebuild fe.
D2ACME_EMAIL never empty — empty crash-loops Caddy.
D3MySQL creds freeze into mysql_data at first init — change = backup, wipe, re-migrate, re-seed.
D4No down -v / volume delete in prod without a same-day verified backup.
D5DB/Redis ports never public on the VPS — only Caddy 80/443.
D6Env vars have one canonical list (ENVIRONMENT.md); .env never in git; no secrets in images or logs.
D7No deploy without a live rollback path: previous tag in GHCR + .env.deployed current.
D8Production only ships via the pipeline — no hand-built images on the VPS (except executing a rollback).
