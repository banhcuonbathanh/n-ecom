The system — five containers, one entry door

Everything runs in Docker Compose on your machine. You open http://localhost and Caddy decides where the request goes: anything starting with /api/ goes to the Go backend, everything else goes to the Next.js frontend.

One concrete example: you click “Add to cart”

  1. The product page calls our single API client: POST /api/cart/items {productId, qty: 2}. No component ever calls fetch directly.
  2. Caddy sees /api/ and passes the request to the Go backend.
  3. Backend: handler checks the JSON shape → service checks the business rule (is there stock? cap at available) → repository writes the cart line to MySQL.
  4. Backend answers with the full updated cart — or with the error envelope: {"error":{"code":"OUT_OF_STOCK", …}}.
  5. TanStack Query updates its cached cart, and every component that shows the cart (the header badge, the mini-cart) re-renders by itself.
2

Who does what — backend vs frontend

The one rule that decides everything: the backend is the source of truth. The frontend never calculates prices, stock, totals, or order status — it displays what the backend returns. If the FE ever needs math on money, that math belongs in the BE.

Backend owns

Go + Gin + sqlc · the truth

  • Data — the only thing that reads/writes MySQL and Redis
  • Business rules — stock, price snapshot, cancel window, cart merge (PLAN.md §Business rules)
  • Real validation — the check that counts; FE validation is only comfort
  • Auth — passwords, JWT signing, who may see which order
  • The error envelope — every endpoint, same shape, always

Frontend owns

Next.js + TS + Tailwind · the experience

  • Pages & navigation — routes, layout, SEO for the catalog
  • Server data — fetched and cached via TanStack Query, never copied into a store
  • UI state — “is the cart drawer open” — Zustand, and only this kind of state
  • Forms — React Hook Form + Zod for instant feedback before submit
  • One API client — fe/src/lib/api/, the single door to the BE

Inside the backend: four layers, no skipping

handlerHTTP only — parse request, call service, format response. Zero business logic.
serviceBusiness rules live here — “stock decrements at placement”, “cancel only before shipped”.
repositorySQL only, generated by sqlc from written queries. No rules, no HTTP.
dbMySQL 8. Schema changes only through migrations, never by hand.

Why so strict: when a bug appears, its location tells you where to look. A wrong total is always the service. A broken query is always the repository. This is what keeps the codebase maintainable when it grows.

Design system — decide once, reuse everywhere proposal

A design system is not a tool or a library. It is a small set of visual decisions we make once — colors, text sizes, spacing, corner radius — plus a small set of reusable components built from those decisions. Every page is then assembled from the same parts, so the whole shop looks like one product and new pages get faster to build, not slower.

6

Words we keep using — so they mean the same thing

Harness
The harness/ folder: plan, tasks, state, receipts. It is the project's memory — every session resumes from these files, not from chat history.
Task · AC · receipt
A task is one session of work with an acceptance criterion (AC) — the sentence that defines “done”. A receipt is the proof (test output, curl transcript, screenshot) logged in VERIFICATION.md. No receipt, no ✅.
Vertical slice
Building one feature through all layers (schema → API → page) before starting the next feature. Opposite of “all backend first”.
Migration
A numbered SQL file that changes the database schema, with an up (apply) and down (undo). The only allowed way to change the schema.
Error envelope
The one JSON shape every API error uses: {"error":{"code","message","details"}}. The FE handles errors in one place because they always look the same.
Price snapshot
At checkout, the order copies the product's current price into the order line. Later price changes never affect existing orders.
Guest cart
A cart identified by a random cookie token instead of a user — shopping works without an account; on login it merges into the account's cart.
COD
Cash on delivery — v1 takes no online payment, so an order is confirmed manually and paid on arrival.
Design token
One named visual decision (“accent = #0F766E”, “radius = 8px”) that components reference instead of hard-coding values.

Lessons learned

Files beat memory
Every session resumes from STATE.md, not from chat history — the one time state wasn't written down (repo not initialized), it surfaced as drift the next session. Checkpoint discipline is what caught it.
Cut scope early, in writing
Deferring payment / admin / AI in Session 0 — recorded as decisions, not vague intentions — means no session re-litigates them.
One fact, one home
When the same fact lives in two files it drifts (Hard Rule 5). Same idea on this page: the progress bars in section 0 are computed from the task list in section 3 instead of being a second copy of it.
