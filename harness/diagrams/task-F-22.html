F-22 — The admin task board, planned with its four bugs removed

You asked for the task-board page. The reference has one, fully implemented — and fundamentally broken: a task's status is written once as pending and no endpoint, no job, nothing in the codebase ever changes it. So three of the four KPIs, the completion rate, the quality stars and the overdue highlight are permanently zero on any live database. This task produces the 4-doc page-plan set, and the plan's job is to fix that before a line is built.

01

Objective

Produce the four-document page-plan set for /admin/staff/task-board per PAGE_PLAN_GUIDE.md, reconciling the reference implementation against our harness rules — and design out every defect the reference documented about itself.

The one thing this plan exists to fix. The reference board is a dashboard that cannot show progress. Its staff_tasks.status column has four values; exactly one is reachable. Its stats queries, its KPI row, its completion percentages and its overdue highlight are all built on the three that are not.

Our correction is two decisions: add a status-transition endpoint so status can move, and stop storing overdue — derive it from due_at < NOW() at read time, so it is correct by construction with no sweeper job to write, deploy, or forget.

Core loop: pick a date → read the KPIs → scan the staff table → expand a row to see that person's tasks → assign new work. The date is a server filter (it re-keys both queries); role, status and search are client filters over data already in hand.

04

Decisions taken

#DecisionWhy
1Add PATCH /admin/tasks/:id/status with a validated transition machineWithout it the board is decorative. This is the root fix
2overdue is derived, never stored; the stored enum is 4 valuesA stored overdue needs a sweeper job to ever be true. Deriving it needs nothing and cannot drift
3qualityScore is dropped entirelyIt was completionRate / 20 dressed up as an independent metric. No quality signal is captured anywhere. We do not ship invented data
4Bad staffId → 422 VALIDATION_FAILED, not 500Caller error, not server error. Pre-check the staff row instead of letting the FK constraint blow up
5Stats DTO carries per-status counts (byStatus)It is what makes the status filter functional instead of decorative
6A write invalidates both the task's date and the filtered dateThe reference invalidates only the task's date, so creating for another day leaves the visible KPIs stale
7v1 ships create + status-change. Edit and delete deferredThe reference's "edit" silently inserts a duplicate row. Better to ship neither than half of one
8No Redis in the task domainARCHITECTURE §4 applied: Redis is catalog cache + auth rate-limit only
9Neutral F-7 tokens, not the customer dark/orange shellAdmin surface rule, PAGE_PLAN_GUIDE §7
10This plan owns the /admin/tasks contract; the sibling admin_todo_list plan links to itSame table, same endpoints — one fact, one home
05

The status model

                 ┌──────────────► cancelled ◄──────────┐
                 │                                      │
   (create) → pending ──────► in_progress ──────► completed
                 │                                  │
                 └──────────────────────────────────┘   direct complete allowed

   stored    pending · in_progress · completed · cancelled
   derived   overdue  =  status IN ('pending','in_progress') AND due_at < NOW()
                        ↑ computed in the read query — no job, no write, never stale

completed → cancelled and anything out of cancelled are rejected 409 CONFLICT. A transition to the same status is an idempotent 200, so a double-click is harmless. Validation lives in the service layer — handlers shape, services decide (BE_STATE).

💡 SUGGESTION — the assignee cannot see their own task. Both pages that read staff_tasks are manager+ gated. A manager can assign work that the staff member has no screen to read. Out of scope for this page, but it wants a ruling before the AD phase closes — cheapest fix is a GET /me/tasks plus a panel on the staff surface.

Risk — shared working tree. Parallel sessions are editing TASKS.md, CONTEXT_MAP.md and README.md concurrently. Mitigation: every edit here is strictly additive (append a row, never rewrite a block), and the commit stages only this task's paths.
