01

Why this task, why now

F-5 fixed how the frontend holds state; F-6 fixed the component map and layer contracts. What is still undecided is the backend's dynamics: who owns which kind of data at runtime, where a transaction begins and ends, which layer validates what, and how a service failure turns into the Session-0 error envelope. Every BE task (C-2 → A-3) needs those answers; writing them once now is cheaper than re-deciding them five times.

One-fact-one-home: this doc does not repeat the layer table or Redis policy — those stay in ARCHITECTURE.md §2/§4 and are linked. BE_STATE.md owns only what no other file owns.

03

What the doc decides

  • Five kinds of backend state, five owners — business facts → MySQL · cache → Redis via platform/cache · request scope → context.Context · config → platform/config at boot · in-process mutable business state → banned (the API stays stateless).
  • Request flow, one straight line — Caddy → Gin router → middleware (recover → log → auth) → handler (bind + shape-check, ONE service call) → service (rules, tx, cache) → repository (sqlc) → MySQL; errors exit only through platform/errs.
  • Transaction policy — services own Begin/Commit; repositories accept a querier; the three atomic flows (place order, cancel, cart merge) mapped explicitly; reads never open transactions.
  • Error-code enum — the stable SCREAMING_SNAKE codes with their HTTP statuses and which layer raises them; FE branches on these codes (FE_STATE.md §4), so this table is the contract's other half.
  • Validation tiers — handler = shape only (binding tags) · service = business rules; mirrors the FE's Zod-vs-server split.
  • Hard BE rules — 8 numbered rules with the same weight as PLAN.md rules.
