Realtime, built right this time

   service writes MySQL ──▶ publish Redis ──▶ fan-out to every subscribed BE instance
                                                    │
        channels: order:{id} · orders:kds · orders:admin · queue/tables:broadcast
                                                    │
             ┌──────────────────────────────────────┼──────────────────────┐
             ▼ SSE (customers, admin popup)         ▼ WS (staff boards)    ▼
      snapshot-on-connect FIRST, then deltas · ownership + role gates on every stream
      one event-contract file (Go + TS mirror) · exhaustive consumer switches
      separate Redis clients for cache vs pub/sub · SSE retry: directive

The four lines above each kill a documented defect of the old system: lost events on reconnect, ignored event types, ungated streams, pool exhaustion.

04

Frontend plan — 3 shells on decided rules

shell

Customer

Mobile-first, fixed bottom tab bar (Menu · Orders · Favourites · Tracking · Settings).

shell

Dashboard

Wraps /kds /pos /cashier /admin — owns the ONE WebSocket provider per browser session.

shell

Admin

Tab nav + AuthGuard/RoleGuard (manager+), theme toggle; nested in dashboard shell.

  • Cart is client-only (Zustand, persisted) — first server write is POST /orders; no BE cart endpoints exist.
  • Sessions in httpOnly cookies (F-5 decision stands) — beats the reference's memory-token: SSE/WS auth rides the cookie, no ?token= in logs, no "F5 wipes the session, re-scan the QR" bug.
  • FE renders only what BE provides: writes return full DTOs; FE role-gates every button it shows; error branches exist only for codes BE actually emits.
  • Favourites = device-local store (account sync waits for the online phase). POS cart = component state, deliberately separate.
