The gap, measured

The question was “do we have enough to build FE and BE?” Answering it honestly meant reading both plans against the docs they lean on. The asymmetry was the finding:

SideBeforeMissing
BE~90 % — F-30 gives migrations, every sqlc method, package tree, tx body, the expansion algorithm, receiptsone schema amendment (F27), unregistered rows, TASKS.md ACs still generic-shop
FE~60 % — §4 of the page plan gives the file map, state ownership, loading tiers, 12 behaviorsthe whole build spine: slices, token/copy layer, store contracts, prop contracts, the payload algorithm, receipts

Why the asymmetry mattered. A behavior spec tells a session what to check when it’s done. It doesn’t tell it what to type first, which file owns the shell, or that its types.ts is blocked on a curl transcript that doesn’t exist yet. That last edge — FE slices wait on BE receipts — is invisible in a behavior list and is now drawn in the spine.

What the FE reconciliation found

🚨 F34 — canh is identified by name on the frontend. The page plan filters soup out of lists, search and combo payloads with isSoupName(), a string match — while the backend plan seeds a real CANH category and deliberately refuses name-matching in Go. So one business rule is keyed two different ways, and renaming a dish in /admin/products would silently break the canh gate, the double-canh guard and the payload builder at once. Plan default: isCanhProduct(p) = p.category_id === CANH_CATEGORY_ID — both sides then key off the same seeded fact.

#WhatPlan default
F35 ⚠️keys.ts + FE cache policy are pre-pivot: params on the products key, a cart query that no longer exists, 60 s staleness, no combos keyparams-free factory + a combos key + 5-min staleTime; amend FE_STATE §2/§3 at C-4a (the FE twin of F29)
