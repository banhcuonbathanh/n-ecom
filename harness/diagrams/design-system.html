The design system — every component, specified

One visual language for the storefront: the tokens everything is built from, a deep-dive on the button (anatomy, six variants, three sizes, six states), and the full component inventory — forms, feedback, overlays, navigation, and the commerce patterns the Catalog/Cart/Orders phases will ship. Stack per F-1: Tailwind classes map 1:1 to these tokens; each specimen becomes a React component in components/ui/.

02

Buttons — the deep dive

The button is the most-used component in the store — add to cart, checkout, pay, cancel. One React component, <Button variant size>, renders every specimen on this page. Anything clickable that is not a text link is this component.

2.1 Anatomy

① leading icon 16–18px, gap 8px ② label — verb first, sentence case, ≤ 3 words ③ container — height fixed per size, radius 8/10px ④ focus ring — 2px offset ring, never removed

Fixed heights (32 / 40 / 48) — never let content stretch a button. Horizontal padding scales with size; icon-only buttons are perfect squares.

2.2 Variants — 6, no more

VariantUse forRule
primaryThe one main action of a view — Add to cart, Place order, Sign inMax one per view/section. Two primaries = no primary.
secondarySupporting action next to a primary — View details, Apply couponTinted, never competes with primary for attention.
outlineNeutral actions — Continue shopping, Edit address, filtersDefault choice when nothing above fits.
ghostLow-emphasis, dense UIs — table row actions, Clear, toolbar iconsOnly inside surfaces that already have structure.
destructiveIrreversible: Cancel order, Remove item, Delete addressAlways behind a confirm step (modal §5) for data loss.
linkNavigation styled as text inside prose or card footersNever for mutations — links navigate, buttons act.

Mobile: interactive targets must hit ≥ 44×44px — sm buttons get extra tap area via a pseudo-element, or use md.

2.4 States — every variant × every state

DefaultHoverFocusDisabledLoading Primary Secondary Outline Ghost Destructive

The default-column buttons are live — hover and Tab through them. The other columns force each state for side-by-side comparison. Active (pressed) darkens one more step than hover.

StateBehaviour contract
hoverBackground shifts one step (primary → primary-hover); 120ms ease. Never changes size or moves the label.
activeOne step darker than hover. No translate/scale tricks — color only.
focus-visible2px ring offset by 2px of surface, ring color = focus-ring. Keyboard only (:focus-visible), never suppressed.
disabledopacity: 0.5 + cursor: not-allowed. Prefer hiding an action over disabling it with no explanation nearby.
loadingSpinner replaces the leading icon, label switches to present-progressive ("Placing…"), aria-busy="true", pointer events off. Width must not jump — reserve icon space. Every mutation button (TanStack isPending) uses this.

2.5 Compositions

With icons — leading (action) or trailing (direction)
Button group — segmented siblings (sort, view toggle)
Full width — checkout funnel & mobile sheets only
Action pair — primary on the right, always

2.6 Do / Don't

Do

These pass review

  • One primary per view; everything else steps down a variant
  • Verb-first labels: Add to cart, Place order, Save address
  • Loading state on every mutation — wire to isPending
  • Destructive variant + confirm modal for anything irreversible
  • aria-label on every icon-only button
  • Price in the label when it's the final commit: "Place order · $172.49"

Don't

These get bounced in self-review

  • Invent variant #7 — restyle nothing per-page
  • Vague labels: "OK", "Submit", "Yes", "Click here"
  • Disable a button as the only error feedback
  • Use btn-link for a mutation, or a real <a> styled as primary for POSTs
  • Let button width jump when the spinner appears
  • Remove the focus ring "because it looks cleaner"

03

Form controls

Forms run through React Hook Form + Zod (F-1 decision). Every field = label + control + one line of help or error, never both. Error copy comes from Zod messages; API errors map to fields via the {"error":{details}} envelope.

Select · textarea

Checkbox · radio · switch

Checkbox = independent yes/no. Radio = pick one of few, all visible. Switch = instant effect, no form submit. Select = pick one of many.

Quantity stepper — cart's signature control

2 5 Only 5 in stock

"+" disables at the stock cap (PLAN.md business rule) with the reason shown beside it. Qty 0 = remove, always via a confirm on the cart page. Updates are optimistic (FE_STATE.md: cart is the only optimistic domain).

Loading — spinner vs skeleton

Skeleton — first paint of any list/detail (tier 1)
Spinner — in-button (tier 2) or tiny inline refetch (tier 3)

Skeletons mirror the real layout to prevent shift. Never both a page spinner and skeletons; never a full-screen overlay for a refetch.

Overlays — modal, menu, tooltip

Confirm modal — the destructive-action gate

Cancel this order?

Order #10248 will be cancelled and stock returned. This can't be undone after the order is confirmed.

Rules: dimmed backdrop at 45% ink · focus trapped inside · Esc + backdrop click = safe dismiss · the safe action is a plain outline, the destructive one carries the color · titles are questions, buttons answer them (never "OK / Cancel" — "Cancel" is ambiguous when the action is cancelling).

Dropdown menu — account & row actions

👤 My account 📦 My orders 📍 Addresses
↩ Sign out

Menus: shadow-2, 6px padding, destructive item last after a divider. Tooltips: labels only (≤ 4 words), 300ms delay, never contain actions, never the sole label for anything.

Hard rules (self-review checklist for every FE task)

  • No raw hex / px values in feature code — tokens and scale steps only.
  • No new button variant, badge tone, or shadow without updating this page first.
  • Every mutation button shows its loading state; every list has a skeleton; every empty list has an empty state.
  • Icon-only controls carry aria-label; focus rings are never suppressed; contrast ≥ 4.5:1 for text.
  • Feature components import from components/ui/ — they compose, they never fork.
  • Both themes checked before a task's VERIFY receipt — one screenshot each, light and dark.

💡 Proposal (silence = accepted, locks in when C-4 starts): brand primary = the harness blue #2B59D9, danger doubles as the sale color, and star-gold is the only color outside the token set's semantic roles. Owner may swap the primary hue any time before C-4 — it's a one-line token change.
