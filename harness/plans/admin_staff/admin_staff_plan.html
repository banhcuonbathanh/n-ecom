Admin Staff — /admin/staff

The staff-account roster: the one screen where a manager creates, edits, deactivates and deletes staff logins. Six manager+ gated endpoints, plain CRUD over the staff table with three business guards — and the account-lifecycle origin of the whole platform: every staff JWT, every POS/KDS session and every assignee dropdown downstream begins with a row created here.

source of truthadmin_staff_PLAN.md

Why a write here outlives the page. It changes two server hubs: the staff MySQL row (single source of truth) and the auth:staff:<id> Redis key the auth middleware reads on every authenticated request.

Deactivating a cashier logs them out of the POS on their next API call — lockout is immediate, not TTL-delayed, and their refresh tokens are deleted in the same transaction.

Not in scope: staff login (staff_login), per-staff KPIs (admin_summary · GET /admin/staff-performance), task assignment (F-22 / F-20), training progress (F-20).

BE contract — endpoints, schema, cache map, wire shapes

Endpoints the page consumes (§3.1 — all under /api/v1)

Route group /staff, authMW + AtLeast("manager") on the whole group; the DELETE sits in a nested group additionally requiring AtLeast("admin").

#RouteAuthTaskBehavior
1GET /staffmanager+S-3 List, server-filtered + server-paged: role, search, is_active, page, limit. Returns {data:[Staff], meta:{page,limit,total}}

3POST /staffmanager+S-4 Create. Guards: valid role → hierarchy → username unique → bcrypt → INSERT. 201 {data:Staff}
4PATCH /staff/:idmanager+S-4 Partial update. Only keys present in the body change (§3.7). Hierarchy guard on both current and new role. 200 {data:Staff}
5PATCH /staff/:id/statusmanager+S-4 {is_active:bool}. Self-toggle blocked, hierarchy guard, + auth-cache invalidation + refresh-token revoke. Returns the full row

Stats are derived, not fetched — and that breaks under paging. The reference computed the stats bar client-side from the full list. With server-side paging, page 2 of 5 can't count the roster. So GET /staff returns a meta.counts block computed in the same query as the total. This is the one place we add to the reference contract rather than trim it.

Every staff column is read; writes cover all columns except id / created_at. No migration is owned by this plan — staff lands with the S-phase auth task (the same task that seeds the first admin).

Cache & side-effect map (§3.3 — the S-4 acceptance criteria)

Per ARCHITECTURE.md §4, Redis is catalog-cache + auth-rate-limit only. The staff roster is never read-cached — every GET /staff* hits MySQL. The only Redis touch is a write.

WriteMySQL effectRedis effectrefresh_tokens effect
POST /staffINSERT, is_active=1——
PATCH /staff/:idUPDATE changed cols——
PATCH /staff/:id/statusis_active flipped DEL auth:staff:<id>DELETE all rows when deactivating
DELETE /staff/:iddeleted_at=NOW() + username rename DEL auth:staff:<id>DELETE all rows

The DEL forces the auth middleware's next IsStaffActive check to re-read MySQL, so lockout is immediate, not TTL-delayed. Deleting the refresh-token rows closes the reference's unrevoked-session hole.

⚠️ Fail-open is inherited, deliberately. IsStaffActive returns true when Redis is down (ARCHITECTURE §4), so a Redis blip briefly weakens the just-disabled lockout. Availability-over-strictness, consistent with the rest of the platform — but now bounded by the token revoke, which is a MySQL write and therefore blip-proof.

PATCH semantics (§3.7) — a real design constraint

PATCH /staff/:id must distinguish three cases per field: omitted (leave), null (clear), value (set). A plain struct-bind cannot — a missing string and an empty string both arrive as "". The reference decoded into map[string]json.RawMessage and built *string pointers; we keep that approach (it is correct). AC: PATCH {"job_title": null} clears the column; PATCH {} is a no-op that still returns 200 with the unchanged row.

The four guards — all in the service layer

BE_STATE §5: business rules never in handlers, never in SQL.

GUARD 1Hierarchy

You may only act on a target strictly below your level, and may only assign a role strictly below your level. Consequence: no admin can be created or promoted through this page (admin 5 ≮ admin 5). Adopted from the reference deliberately — admins are seeded by migration, never minted from the UI.

GUARD 2Username uniqueness

Checked before insert; the UNIQUE index is the backstop. Surfaces as a field-level 409 on the username input, modal stays open (behavior 8).

GUARD 3Last admin

Deleting the final active, non-deleted admin is refused — "Không thể xóa admin cuối cùng", and the row stays (behavior 16).

GUARD 4No self-harm

You cannot toggle your own status or delete your own account. The FE disables the buttons with a tooltip; the 403 is the backstop, not the UX (behavior 13).

What a write leaves behind (§4.5) — cross-surface effects

SurfaceEffect
Auth middleware (every authed request) DEL auth:staff:<id> → next request re-reads MySQL → deactivated/deleted staff gets 401 immediately
Login / refresh a created account can log in at once; a deactivated one is refused; its refresh tokens are gone

Multi-device: two managers' tabs do not sync live. Manager B sees A's change on tab-focus refetch. The affected staff is cut off immediately (auth middleware). That asymmetry is intended. No SSE/WS — every downstream surface is pull-only.

06

Loading branches (§4.3) — five named, mutually exclusive

Four layers, and five named render branches — the reference had four layers but collapsed three of its states into two surfaces. Named branches are the countermeasure.

BRANCH 1loading

When: first load, no data. Renders: a table-shaped skeleton — header + 10 ghost rows + stats-tile ghosts. Never a bare "Đang tải…".

BRANCH 2error

When: list query failed. Renders: inline error panel + Thử lại → refetch(); the filters stay usable.

BRANCH 3empty-roster

When: meta.counts.total === 0. Renders: "Chưa có nhân viên nào" + a primary + Thêm nhân viên CTA.

BRANCH 4empty-filtered

When: total > 0 but this filter matched 0. Renders: "Không có nhân viên khớp bộ lọc" + a Xóa bộ lọc CTA.

BRANCH 5rows

When: data present. Renders: the table.

The key fix: empty-roster vs empty-filtered. This split is the single highest-value fix on this page. The reference showed one EmptyState for both, so a manager could not tell "we have no staff" from "your filter is too narrow." Two branches, two CTAs, two different next actions.

Other layer rules

RuleWhy
Stats bar reserves its height while loading (ghost tiles) so it does not pop in and shift the table (ref loading-Flag 3)
Detail drawer has an error branch — a failed GET /staff/:id shows "Không tải được. Thử lại / Đóng" never a stuck spinner when the row was deleted on another device (ref loading-Flag 1)
Background refetch is visible — a subtle top progress bar while a focus-triggered refetch is in flight so silently-stale rows are not mistaken for fresh ones (ref loading-Flag 4)
Search is debounced 300 ms and drives a server query; the table dims rather than unmounting (placeholderData: keepPreviousData) unlike the reference there is an in-flight state — keep the previous page visible
07

Behaviors spec (§4.4) — the twenty acceptance criteria

B1Role gate, no flash

Page renders only for role ≥ manager; a lower role is redirected by the shell guard, never shown a flash of content.

B2Deep link reproduces the view

List loads server-filtered and server-paged from the URL params; a deep link reproduces filters, page and search exactly.

B3Stats survive paging

Stats bar shows roster-wide counts from meta.counts and does not change when you page.

B4Debounced search

Search debounces 300 ms, resets to page 1, and keeps the previous rows visible (dimmed) while in flight.

B5Filters reset the page

Role and status filters reset to page 1 and are reflected in the URL.

B6Five branches, each reachable

The five render branches (§06) are mutually exclusive and each is individually reachable.

B7Add mode offers only lower roles

+ Thêm nhân viên opens the modal in add mode; the role <select> offers only roles strictly below the caller's level.

B8Taken username, field-level

Create with a taken username surfaces "Tên đăng nhập đã tồn tại" on the username field, and the modal stays open with input intact.

B9Create success

Closes the modal, toasts, and the new row appears after the list invalidate.

B10Edit mode, no password field

Row ✎ opens the modal in edit mode, prefilled; the password field is absent in edit mode (password change is a separate S-phase concern — ❓-1).

B11PATCH sends only changed keys

Clearing an optional field sends explicit null and the column is cleared (§3.7).

B12Toggle names its consequence

Row ⏻ toggles status; re-activating asks for confirmation, deactivating asks for confirmation and names the consequence ("… sẽ bị đăng xuất khỏi mọi thiết bị").

B13Your own row is disabled

Your own row's ⏻ and 🗑 are disabled with a tooltip explaining why — the 403 is the backstop, not the UX.

B14Rows at or above your level

Have ✎/⏻/🗑 disabled with a tooltip; 🗑 is hidden entirely for managers (admin-only route).

B15Typed delete confirmation

Row 🗑 opens a typed-confirmation dialog (not a native confirm()), naming the account.

B16Last admin refused

Deleting the last admin is refused with "Không thể xóa admin cuối cùng" and the row stays.

B17Drawer has both branches

Row 👁 opens the detail drawer, which fetches its own detail and has loading and error branches.

B18Delete closes an open drawer

Deleting a staff whose drawer is open closes the drawer and evicts its detail cache.

B19Every mutation is pessimistic

The button shows a pending state; the table does not change until the server confirms.

B20Lockout is real

A deactivated or deleted staff is locked out on their next API call, and their refresh tokens are gone (§04) — verified by the S-4 receipt, not by this page's UI.

⚠️-3 — Fail-open weakens lockout. Inherited from ARCHITECTURE §4. Now bounded by the token revoke, but a Redis outage still leaves a live access token valid until expiry. Accept, or shorten access-token TTL at S-2.
