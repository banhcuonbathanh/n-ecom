Admin Staff — how it works

Not what we will build — what actually happens when it runs. This page traces /admin/staff end to end: the four loading layers and five render branches on first paint, what a keystroke costs now that filtering is server-side, the four guards a POST /staff passes through in order, why PATCH needs three-way field semantics, and — the part that outlives the page — how one deactivate click reaches MySQL, Redis and refresh_tokens, then fans out to four downstream surfaces.

01

Big picture — the whole machine on one screen

One manager, one browser tab, six REST endpoints. There is no realtime channel — no SSE, no WebSocket, no polling loop. Every downstream surface is pull-only (PLAN §4.5). On the client there is no Zustand slice and no localStorage: nothing on this page is cross-page client state (PLAN §4.2, FE_STATE rule 7). Only two things are durable, and both live on the server: the staff MySQL row (single source of truth) and the auth:staff:<id> Redis key the auth middleware reads on every authed request.

The three traffic kinds, and what each costs

Reads — always MySQL
GET /staff · GET /staff/:id

The staff roster is never read-cached in Redis (ARCHITECTURE §4 — Redis is catalog-cache + auth-rate-limit only). Every list and every detail fetch hits MySQL. Server-filtered, server-paged.

Writes — MySQL only
POST /staff · PATCH /staff/:id

Create and edit touch nothing but the staff table. No cache to invalidate, no sessions to revoke — the account keeps working exactly as before.

Writes with a blast radius
PATCH /staff/:id/status · DELETE /staff/:id

These three-touch writes hit MySQL + Redis DEL + refresh-token DELETE, and are felt on four other surfaces. That is all of §06.

The machine in one line: a thin pull-only client (TanStack list key + detail key, filters in the URL) over six REST endpoints whose guards all live in the service layer — and exactly two writes that reach past this page into every other staff session on the platform.

02

First load — four loading layers, five render branches

A manager clicks Nhân viên in the admin shell (or opens a deep link with filters already in the query string). Four independent loading layers fire, in this order — and the table area resolves to exactly one of five named branches (PLAN §4.3).

Layers 1–3 run on every load, left to right. Layer 4 is lazy — the drawer query is enabled only while the drawer is open.

Step by step, first paint to rows

  1. 1
    next server — the route segment starts rendering; loading.tsx paints a table-shaped skeleton: header row, ten ghost rows, and ghost stat tiles. Not a bare "Đang tải…". The skeleton has the shape of what is coming, so nothing jumps when data lands (FE_STATE rule 3).
  2. 2
    next server — the admin shell's auth + role guard runs. A staff JWT below manager is redirected here, before any roster markup exists. Behavior 1: a lower role is never shown a flash of content — the guard is upstream of the render, not a conditional inside it.
  3. 3
    next server — page.tsx reads searchParams (search, role, is_active, page, limit), prefetches the list with exactly those params, and wraps the tree in a HydrationBoundary. This is why a deep link reproduces filters, page and search exactly — the URL is the input to the very first fetch, not something applied afterwards.
  4. 4
    gin api — GET /staff passes authMW + AtLeast("manager"), and the service runs one filtered, paged query plus the roster-wide counts.
  5. 5
    mysql — MySQL answers. No Redis is consulted on this path at all: the roster is never read-cached.
  6. 6
    gin api — responds {data:[Staff], meta:{page,limit,total,counts}}. meta.counts is roster-wide — it ignores page/limit so the stats bar does not change as you page (behavior 3).
  7. 7
    browser — the client hydrates the same cache entry the RSC filled: ['admin','staff','list',{filters}]. No second fetch, no flash of skeleton after hydration.
  8. 8
    browser — the table area resolves to one of the five branches below, and the stats bar swaps ghost tiles for real numbers in the height it already reserved.
  9. 9
    browser — layer 4 stays idle. The detail query fires only when a row's 👁 is clicked, and it has its own loading and error branch — a failed GET /staff/:id (the row was deleted on another device) shows "Không tải được. Thử lại / Đóng", never a stuck spinner.

The five render branches — mutually exclusive, each individually reachable

This is the highest-value fix on the page. The reference collapsed three states into two surfaces, so a manager could not tell "we have no staff" from "your filter is too narrow". Five named branches are the countermeasure, and behavior 6 asserts that exactly one is ever on screen.

Filter & search at runtime — what a keystroke costs

The reference filtered on the client: it fetched ?limit=100 once and did all searching, role-filtering and paging in memory. Zero network per keystroke — and a roster of more than 100 people silently truncated. We reversed it (PLAN §3.4-b): the FE sends role, search, is_active, page and limit from the URL, and the server does the work. A keystroke now costs a round trip, so the runtime picture is completely different.

The keystroke, step by step

  1. 1
    browser — the manager types a, n in the search box. The input value is local and updates on every character; nothing else moves yet.
  2. 2
    browser — a 300 ms debounce timer runs. Each new character restarts it. Typing "nguyen" at normal speed produces one request, not six.
  3. 3
    browser — the timer fires and the debounced value is written to the URL, together with page=1. The URL is the owner of filter state (FE_STATE rule 2), so this write is the single trigger for everything downstream. Because it went through the URL, the back button walks the search history and the link is shareable — a colleague opening it sees the same filtered view.
  4. 4
    browser — the params change ⇒ the query key changes ⇒ TanStack treats it as a different query. There is no cache entry for the new key yet.
  5. 5
    browser — placeholderData: keepPreviousData hands back the previous key's rows while the new fetch is in flight. The table dims rather than unmounting — no skeleton flash, no layout collapse, no scroll jump (behavior 4).
  6. 6
    gin api → mysql — the server filters and pages in SQL and returns the matching page plus roster-wide meta.counts.
  7. 7
    browser — rows swap in and the dimming lifts. The stats bar does not move: its numbers come from meta.counts, which ignores the filter's paging and stays roster-wide.
  8. 8
    browser — if the new page has zero rows, the branch decision from §02 reruns: total > 0 and this page empty ⇒ empty-filtered, with a Xóa bộ lọc CTA — not the "no staff at all" message.

Why we changed it

Reference (client-side)Us (server-side, PLAN §3.4-b)
Per keystrokezero network — filter in memoryone debounced round trip (300 ms)
Roster size?limit=100 hard-coded; person 101 is invisible with no warningno client cap — the server pages
BE filter paramsimplemented but never sent — dead contractsent from the URL on every request
In-flight statenone existed, so none was designeda real one: previous rows kept, dimmed (keepPreviousData)
Deep linkfilters live in component state — a shared URL loses themURL owns them — the view reproduces exactly

The trade, stated plainly: we bought correctness at scale (no silent 100-row truncation) and shareable views, and paid for them with one network round trip per settled keystroke. The 300 ms debounce bounds the cost; keepPreviousData hides the latency.

04

The write path — POST /staff through every layer

A manager fills the add modal and hits save. The request crosses five layers, and in the service layer it must pass four guards in a fixed order. Order matters: each guard is cheaper than the next, and each produces a different error code.

Blue = the request/response the manager sees. Amber = handler→service. Green = the persistence path. Dashed red = the error rail: each guard exits to a different code, and the modal stays open with input intact.

Layer by layer

  1. 1
    browser — React Hook Form + Zod validate the form shape before anything leaves. The role <select> only ever offered roles strictly below the caller's level, so the common hierarchy violation is prevented at the widget, not discovered at the server (behavior 7).
  2. 2
    gin api — the /staff route group has already run authMW + AtLeast("manager"). The handler does shape validation only and passes the caller's identity and role to the service as explicit params — never business rules, never reaching into request context deep in the stack (BE_STATE §5, §6).
  3. 3
    gin api — the service runs the guards, in order: ① valid role — is the requested role in the enum at all? → 422 VALIDATION_FAILED, details: role: invalid. ② hierarchy — may this caller assign this role? You may only assign a role strictly below your own level (chef/cashier 2 · staff 3 · manager 4 · admin 5). → 403 FORBIDDEN, details: role: hierarchy. Consequence: no admin can ever be created through this page (admin 5 ≮ admin 5) — admins are seeded by migration. ③ username unique — checked before the insert so the manager gets a clean field-level message. → 409 CONFLICT, details: username: taken. ④ bcrypt — the password is hashed. Not a rejection gate; it is where the request spends its time, and it happens only after everything cheap has passed.
  4. 4
    sqlc → mysql — a generated query INSERTs the row with is_active=1. No hand-written SQL layer (BE_PLAYBOOK §1 golden rule). The UNIQUE index on username is the backstop behind guard ③, not the primary check.
  5. 5
    gin api — 201 {"data": Staff}. password is write-only and never echoed; password_hash is 🔒 never serialized — enforced by the DTO, not by hoping.
  6. 6
    browser — the mutation is pessimistic: the save button showed a pending state and the table did not change until this 201 arrived. Now the modal closes, a toast fires, and ['admin','staff','list'] is invalidated across all filter combos — the new row appears on the refetch. Optimistic updates are reserved for cart mutations (FE_STATE rule 4). An account change is not a place to guess.

⚠️ Two different 409s share one code. Both username taken and last admin map to CONFLICT, so at runtime the FE must branch on code plus details[0].issue. That still satisfies "branch on structured data, never message text" — but it is a real widening of the contract, and it is why the issue discriminator is load-bearing rather than decorative.

Two backstops that look like dead code but are not. The row actions disable ⏻/🗑 on your own row and on rows at or above your level, with a tooltip explaining why. The 403s above still exist and are still tested — the disabled button is the UX, the guard is the security. A hand-crafted request must not get further than a click would.

05

PATCH semantics — omitted vs. null vs. value

This is subtle enough to deserve its own picture. PATCH /staff/:id must distinguish three cases per field, and a plain Go struct-bind can only distinguish two — a missing string and an empty string both arrive as "". Getting this wrong means every edit silently wipes every field the manager did not touch.

One question per field, three outcomes. The distinction between the middle and left branches is the entire reason this endpoint is a PATCH and not a PUT.

How it is actually implemented

  1. 1
    browser — the edit modal was prefilled from the row. On save it diffs against the original and sends only changed keys. Clearing an optional field sends an explicit null, not "" (behavior 11). The password field is absent in edit mode entirely — password change is a separate S-phase concern, and today's accepted gap is that a manager cannot reset a forgotten password from here (❓-1).
  2. 2
    gin api — the handler decodes into map[string]json.RawMessage, not a struct. Key presence is the signal; a struct-bind would have destroyed it before the service ever saw the body.
  3. 3
    gin api — each present key becomes a *string (or typed pointer): non-nil pointer to a value = SET, non-nil pointer to nil-inner = CLEAR, absent = nil pointer = LEAVE.
  4. 4
    gin api — the hierarchy guard runs on both the current and the new role. You may not edit someone at or above your level, and you may not promote anyone to at-or-above your level. Both directions, one endpoint.
  5. 5
    mysql — the UPDATE names only the columns that were present. Untouched columns are not in the statement at all, so they cannot be accidentally overwritten by a default.
  6. 6
    gin api — 200 {"data": Staff} with the full row, so the client can replace its cached row wholesale rather than hand-patching it. Then the list key and ['admin','staff','detail',id] are invalidated.

The acceptance criteria that pin this down: PATCH {"job_title": null} clears the column; PATCH {} is a no-op that still returns 200 with the unchanged row. Nullable columns serialize back as real null — never "" — so a round trip through the modal is idempotent. shifts is a JSON array or null, never a string.

06

The side-effect fan-out — where one click is actually felt

This is the most important section on the page, because it is the part that is not visible on this page. POST and PATCH /staff/:id touch one table and stop. PATCH /staff/:id/status (deactivating) and DELETE /staff/:id do three things at once, and are then felt on four other surfaces.

What each write actually touches

WriteMySQL effectRedis effectrefresh_tokens
POST /staffINSERT, is_active=1——
PATCH /staff/:idUPDATE changed cols——
PATCH /staff/:id/statusis_active flippedDEL auth:staff:<id>DELETE all rows when deactivating
DELETE /staff/:iddeleted_at=NOW() + username renamed to <username>#deleted-<id>DEL auth:staff:<id>DELETE all rows

The username rename is not cosmetic: without it a soft-deleted row keeps occupying the UNIQUE index, and re-hiring someone as chef1 fails forever. The token DELETE closes a genuine auth hole — the reference's "revoke sessions" was an aspirational comment that revoked nothing.

End to end: create a cashier, then cut them off

The single most useful runtime story on this page — the full lifecycle of one account, from the row being created to the moment their POS stops working.

t0row createdPOST /staff · is_active=1
t0 + 1 minthey log inJWT + refresh token issued
t0 → t+6hthey workevery request warms auth:staff:<id>
t+6hmanager deactivates3 writes, one transaction
t+6h + secondsnext call 401sno re-login possible
  1. 1
    browser — a manager creates cashier cash02. The row lands with is_active=1. Nothing is cached, nothing is revoked; the account simply exists.
  2. 2
    gin api — cash02 logs in at the POS. Login checks the staff row, issues an access JWT and writes a refresh_tokens row. A created account can log in at once — there is no propagation delay to wait out.
  3. 3
    redis — on each authed request the middleware calls IsStaffActive, which reads auth:staff:<id>. For six hours of shift this is a warm Redis hit and MySQL is never touched for auth.
  4. 4
    browser — end of shift, the manager clicks ⏻. The confirm dialog names the consequence: "… sẽ bị đăng xuất khỏi mọi thiết bị". Re-activating asks for confirmation too, but without that warning — only deactivation cuts someone off.
  5. 5
    gin api — the service runs its guards (you cannot toggle your own status; you cannot toggle someone at or above your level), then performs the three writes: flip is_active, DEL auth:staff:<id>, DELETE FROM refresh_tokens WHERE staff_id=?.
  6. 6
    browser — the manager's tab gets the full row back (not a thin {id,is_active,updated_at}), so the cache replaces the row wholesale instead of hand-patching, then invalidates the list and detail keys.
  7. 7
    redis — cash02's POS makes its very next API call. The cache key is gone, so IsStaffActive falls through to MySQL, reads is_active=0, and the request 401s. Lockout is immediate, not TTL-delayed — that is precisely what the DEL buys.
  8. 8
    gin api — the POS tries to refresh its token. The refresh_tokens rows were deleted, so there is nothing to refresh with. A fresh login is refused by the same is_active check. Every door is closed.

The multi-device asymmetry — and it is intended

Two managers watching the same roster do not see each other's changes live. The person being deactivated is cut off in seconds. Same write, two completely different latencies — because one path is pull and the other is a gate on every request.

⚠️ Consumers must opt in to the filter. Surface ③ — the assignee dropdowns on the task board and to-do list — call the same GET /staff. is_active is a real query param, but it is a param: a consumer that forgets to send is_active=true will happily keep offering a deactivated cashier as an assignee. The list filters deleted_at for you; it does not filter is_active for you.

Nothing on this page uses localStorage and nothing uses Zustand. The only client-side thing that outlives a refresh is the URL — which is exactly the point of putting filters there.

Failure mode 1 — Redis is down

⚠️ IsStaffActive fails OPEN
  • →When Redis is unreachable, IsStaffActive returns true — availability over strictness, inherited deliberately from ARCHITECTURE §4 and consistent with the rest of the platform.
  • →Consequence: during a blip, a just-deactivated staff member's still-valid access token keeps working until it expires.
  • →The alternative — fail closed — would turn a Redis blip into a total platform outage for every logged-in staff member. That is the trade we accepted.
Reads are unaffected: the roster is never read-cached, so GET /staff works normally with Redis down.
✅ but the refresh-token revoke BOUNDS it
  • →The token DELETE is a MySQL write, so it is blip-proof — it lands whether or not Redis is up.
  • →So the exposure window is not "until someone notices" — it is at most one access-token lifetime. The moment that token expires there is nothing to refresh with and no login to be had.
  • →Open lever if that window is still too wide: shorten the access-token TTL at S-2. Cost is more refresh traffic, no design change.
This is why the revoke is not redundant with the DEL: one makes lockout fast, the other makes it certain.

The durability rule in one line: the browser holds nothing that matters — rows are refetched, filters ride in the URL, everything else is memory. The staff MySQL row is the only truth; Redis holds a fast, revocable opinion about it, and the refresh-token table is the backstop that makes revocation stick even when that opinion cannot be reached.
