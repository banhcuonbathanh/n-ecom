Three kinds of traffic, and only three

1 · the read
GET /orders/:id
→ ['order', id]

Fires on mount and on every structural event. Never cached anywhere on the server; MySQL is read each time. The FE seeds from ORDER_CACHE first so the paint is instant.

2 · the stream
GET /sse/order-monitor/:id
→ connected → snapshot → deltas

Exactly one per page, owned by the orchestrator. It never holds order data — it patches or triggers a refetch. Four surfaces hang off it.

3 · the two writes
DELETE /orders/items/:id
DELETE /orders/:id

Both pessimistic, both through the one CancelConfirmModal, both gated by the one canCancel predicate. A refusal shows the envelope message in place — the dish never flickers away first.

The whole page in one sentence: one pivot (activeOrderId), one read (GET /orders/:id into the Query cache), one stream (four surfaces, exhaustive switch), two writes (both cancels) — and a device-local cache that makes the first paint instant and the history list possible without any server endpoint.

02

First load — instant paint, then the network, then the stream

The device cache paints before the network is even touched; the stream connects after the read, and its first act is to hand back a full snapshot — so nothing that happened while the page was closed is missed.

  1. 1
    browser — Tier 1, route. loading.tsx mirrors the real layout (top bar + banner + card + list stubs), never a centered spinner. The reference shipped a bare spinner for the whole group.
  2. 2
    browser — the orchestrator reads activeOrderId from the persisted cart slice and seeds the active order from ORDER_CACHE. A warm return renders real content with zero skeleton. This is why the pivot lives in a persisted store rather than the URL — a cold return has the id before a single byte of network.
  3. 3
    gin api — GET /orders/:id checks the guest JWT and ownership by table_id, reads MySQL (there is no cache to consult), and returns the bare order object — identical in shape to the POST /orders 201, so one TS type serves both.
  4. 4
    browser — the response reconciles into ['order', id]. Items carry a server-derived item_status (pending | preparing | done) — the FE does not re-derive it.
  5. 5
    browser — the orchestrator opens the one EventSource. The httpOnly cookie rides the handshake automatically, so no token ever appears in the URL.
  6. 6
    gin api — event: connected, then the snapshot (queue.update + tables.status) read from MySQL, then deltas from Redis. The pill flips to ● Đang theo dõi.
  7. 7
    browser — live changes are reconciled back into ORDER_CACHE, so the history card and the live card can never disagree. That was the reference's stale-list bug.

The six render branches, as the decision rail

Evaluated top to bottom; the first match wins. Only branch 1 is fatal — every other branch keeps the floor queue and your history on screen.

1. 401/403 on the stream or the GET? — yes → FULL-SCREEN · "Phiên làm việc hết hạn — quét lại mã QR tại bàn" + Về trang menu
↓ no
2. activeOrderId is null? — yes → SCOPED · friendly CTA in the active-order zone only — history + floor queue still render
↓ no
3. GET returned 404/403? — yes → SCOPED · "Đơn hàng không tồn tại" card + clears the orphaned pointer
↓ no
4. GET pending and no cached copy? — yes → layout-shaped skeleton (no shift)
↓ no
5. !sseConnected? — yes → COSMETIC ONLY · pill flips + ConnectionErrorBanner · last-known queue with a stale marker · page fully usable
↓ no
6. default → the live screen
Branch 3 is the deliberate upgrade. In the reference, a missing or foreign order wedged the whole screen — and inside the detail overlay copy it wedged as an infinite spinner, because isNotFound was returned by the hook and never read by the component. Ours scopes the failure to one card and clears the orphaned pointer, so losing your order does not take the floor queue and your history down with it.

Tier 3 — the mutation branch

Cancels are pessimistic: confirm → button disabled + inline "Đang huỷ…" → 204 → refetch. Never optimistic, because a cancel the BE refuses (CANCEL_THRESHOLD) must not flicker the dish away first. The full path is §06.

Frames are event: <type> followed by a JSON data: line. The server subscribes three Redis channels on the guest's behalf and multiplexes them onto one connection:

A published type without a consumer case fails CI. This single rule is the countermeasure that kills four reference bugs at once.

The reference's 🔴 bug, exactly. The FE listened for order.status; the BE published order_status_changed. Nothing threw, nothing logged — the live badge simply never advanced. Three more events (item_progress, items_added, item_cancelled) were published to a hook that had no case for any of them. One generated event-contract file plus an exhaustive switch makes all four impossible to reintroduce.

Reconnect — why snapshot-on-connect means nothing is lost

The snapshot is current state, not an event replay — which is why it heals a gap of any length. The reference had no replay at all: a transition during a disconnect was lost for good.

  • 15 s : keep-alive — a comment frame; it carries no data but stops proxies and mobile radios from killing an idle connection.
  • retry: — the server tells the browser how long to wait before reconnecting; the reconnect is the browser's job, not the app's.
  • 401/403 are permanent — the ownership gate failing is not a transient error. The hook stops retrying and hands control to branch 1 (§02), the only fatal branch.
  • Any other drop is cosmetic — branch 5: the pill flips to ○ Mất kết nối, ConnectionErrorBanner appears, the last-known queue stays on screen with a stale marker, and the page remains fully usable.
04

Queue + ETA — one payload, two surfaces, no disagreement

The banner ("Vị trí hàng chờ: #2 trong 5 đơn", "~8 phút") and the whole-floor list ("Hàng chờ phục vụ [3 bàn]") look like two independent features. They are one payload rendered twice — which is the entire point.

Because the BE holds the ordered active list at snapshot time, it is the only place that can compute position and ETA correctly. The FE renders; it never calculates.

  1. 1
    mysql — any active-order change floor-wide (a new order, a status transition, a cancel) makes the ordered active list move. Active = pending·confirmed·preparing·ready, ordered by created_at.
  2. 2
    gin api — the BE recomputes position and estimated_minutes and publishes one queue.update to queue:broadcast. A NULL table_id serializes as table_label: "Mang về".
  3. 3
    browser — the single switch case stores the whole payload once. TableInfoBanner takes position/estimated_minutes; WholeFloorPrepList takes queue[]. Two components, one object, one render pass — they are structurally incapable of showing different numbers.
  4. 4
    browser — the ETA line pulses when you are at #1; your row in the floor list is ringed in orange and labelled "(bàn bạn)". Only active statuses appear there, and it carries no progress bars — that detail belongs to your order alone.

Why this is a contract, not a convenience. "The banner says #2 but the list shows me third" is the kind of bug users screenshot. Making the two surfaces read one BE-computed payload means the failure mode does not exist — and it retires the reference's 🟡 finding where the wire sent 0 placeholders and the FE quietly made numbers up.

05

The rollup math — "TỔNG SỐ MÓN (7 loại)"

The rollup answers a question the raw item list cannot: how many distinct dishes are actually coming to this table? Combos hide their contents, and the same dish arrives from several sources. lib/order-rollup.ts runs three steps, in this order:

step 1FLATTEN
every combo → its component rows (combo_ref_id set)
→
step 2DROP the header
combo header row: combo_id set, combo_ref_id NULL, unit_price 0 — the 0 đ row
→
step 3MERGE + SUM
rows sharing dish name AND nhân set collapse; SL adds up
→
resultn = distinct rows
Bàn 04 → 7 loại
The header-row exclusion is load-bearing twice. The combo header is excluded from every visible dish list and from all progress math. Leave it in the list and the guest sees a phantom "0 đ" line; leave it in the math and Σ qty_served / Σ quantity is diluted by a row that can never be served.

The Bàn 04 worked example, step by step

1 · raw order_items rows
  • Suất Đầy Đủ Trứng Chín ×1 · 0 đ · combo HEADER
  • ↳ Bánh Cuốn ×3 (Nhân thịt · Mộc nhĩ) · combo_ref_id set
  • ↳ Canh có rau ×1 · from a combo
  • ↳ Canh có rau ×2 · from a combo
  • Canh có rau ×4 · individual
  • Bánh Trứng Chín · from the combo
  • Bánh Trứng Vàng · individual
  • … the remaining individual rows
2 · after flatten + drop
  • combo header — gone from lists AND math
  • components promoted to top level
  • each row now carries: name + nhân set (from toppings_snapshot) + quantity + qty_served
  • 3 separate "Canh có rau" rows are now visible side by side: 1 + 2 + 4
3 · after merge → 7 loại
  • Canh có rau ×7  ← 1 + 2 + 4 merged
  • Bánh Cuốn ×3 (Nhân thịt · Mộc nhĩ)
  • Bánh Trứng Chín
  • Bánh Trứng Vàng  ← stays separate
  • … + the remaining distinct dish+nhân rows
  • n = 7 distinct rows → "TỔNG SỐ MÓN (7 loại)"
  • Tổng cộng 103.000 đ

The two merge rules that produce 7.

Merge on name AND nhân set. Canh có rau arrives three times — ×1 and ×2 from combos plus ×4 ordered individually — and because name and nhân match, all three collapse into one row of ×7. That is the merge doing its job.

Do not merge on name alone. Bánh Trứng Vàng stays a separate row from the combos' Bánh Trứng Chín: different dishes, and even where a name repeats, a different nhân set is a different row. Merging too eagerly would understate n and mislabel what the kitchen is actually cooking.

The same flattened list drives the progress math

bar        = Σ qty_served / Σ quantity     over DISPLAY items only (header excluded)
DishRow    = tổng ×n · ra ×n · còn ×n      + "↳ Nhân thịt · 'ít hành'" caption
             a fully served row shows ✓ in CÒN
footer     = "3/6 phần đã ra"
Card C     = Đã dùng / Còn lại / Tổng cộng     money split three ways

money      = the ONE formatVND() — same function, same glyph as the menu page
             a zero-priced canh renders "0 đ", NEVER blank

⚠️ Money glyph flag. This page's DESIGN_PROMPT specified 105.000 ₫ (U+20AB, space before); the menu plan locked 30.000 đ (U+0111). FE_STATE.md allows exactly one formatVND(), so the app cannot ship both. Plan default: the menu's đ — used throughout this page and its 103.000 đ worked example. One constant, one file, flips everywhere at once.

The nhân set comes from toppings_snapshot (₫0 toppings), never from a filling column — that column does not exist, which is what makes the reference's "OrderItem has no filling field" bug structurally impossible here.

06

The cancel write path — DELETE /orders/items/:id, all the way down and back

One tap on "Huỷ" traverses five layers and comes back up as a broadcast to every client watching the order. This is the page's only structural write, and both cancel scopes share it.

The delete and the total recalculation are in one transaction — a total_amount that disagrees with its items is not a state the database is allowed to reach. The publish happens only after commit, so no client ever sees an event for a write that rolled back.

  1. 1
    browser — "Huỷ" opens the shared CancelConfirmModal. On confirm the button disables and shows inline "Đang huỷ…". Pessimistic by rule — nothing is removed from the UI yet.
  2. 2
    gin api — DELETE /orders/items/:id. The guest JWT rides the httpOnly cookie; the handler parses and delegates. Handlers do not contain business rules.
  3. 3
    gin api — gate 1, ownership. Not your table's order → 403.
  4. 4
    gin api — gate 2, active check via the single canCancel predicate — the same predicate the FE uses to decide whether the button renders at all, so the BE and the button can never disagree. ⚠️ The cancel rule itself is still open: the reference code enforced < 30 % served, the owner's stated target is "cancel any time before payment". Plan default = the owner's rule; it must be locked before O-2 builds this gate. One predicate, one file, one-line flip.
  5. 5
    gin api — gate 3, already-served reject. qty_served >= quantity → 422 CANCEL_THRESHOLD. You cannot un-cook a dish that is already on the table.
  6. 6
    mysql — inside one transaction: DeleteOrderItem then RecalculateTotalAmount. total_amount is denormalized, so every item mutation must recalculate it in the same tx — both or neither.
  7. 7
    redis — after commit, publish item_cancelled to order:{id}. Redis is transport only here — it is never read as a cache for order data.
  8. 8
    browser — 204 No Content comes back to the caller and the item_cancelled event comes down the stream to every client watching that order — including a POS or a second device at the same table. The switch case refetches ['order', id]; the row disappears, the rollup drops a loại if that was its last row, and Tổng cộng updates.

             "message": "Không thể huỷ món đã phục vụ", "details": [] } }

The whole-order cancel (DELETE /orders/:id) is the same rail with a wider scope: same modal, same canCancel predicate, soft-delete, → 204, publishes order_cancelled. On the receiving end that is a terminal branch — the action row is replaced rather than re-rendered.

07

Two-client walkthrough — the chef taps, this screen moves

This is the R-1 acceptance receipt, drawn: a write on one client must move the other client's screen with no refresh. Client A is the kitchen display; client B is the guest's phone on Bàn 04.

Zero refreshes, zero polls. The only thing B did was keep its one stream open.

  1. 1
    kds — the chef marks one more portion of Bánh Cuốn as served on Bàn 04's ticket. qty_served goes 2 → 3 against quantity: 3.
  2. 2
    mysql — the write commits. qty_served is the entire progress model — there is no per-item status column to keep in sync.
  3. 3
    redis — the service publishes item_progress to order:{id}. Redis is pure transport; nothing about this order is cached anywhere.
  4. 4
    this screen — B's useOrderMonitorSSE receives the frame and hits its item_progress case. Plan default: a local cache patch of that item's qty_served, not a refetch — the difference between a smooth bar and a flicker. ❓ Confirm at R-1. Structural events (items_added, item_cancelled) still refetch, because rows change.
  5. 5
    this screen — the DishRow moves: tổng ×3 · ra ×3 · còn ✓. The card's bar advances (Σ qty_served / Σ quantity over display items — the 0 đ combo header excluded), the footer reads "4/6 phần đã ra", and Card C's Đã dùng / Còn lại split shifts.
  6. 6
    gin api — when the last item completes and the order transitions to ready, two more events fire: order_status_changed on order:{id} and a fresh queue.update on queue:broadcast.
  7. 7
    this screen — the StatusBadge advances live, StatusNotificationModal pops "Món của bạn đã sẵn sàng!" with one "Đã hiểu" button, and — from the same queue.update payload — the banner shows #1 with its ETA line pulsing while the floor list row advances. Reconciled back into ORDER_CACHE, so the history card below shows the same progress as the live card above.

The receipt this walkthrough becomes: a two-client transcript for R-1 — client A increments qty_served, client B's stream receives item_progress and the queue advances; plus a reconnect transcript proving snapshot-on-connect replays a status changed while dropped, and a 403 proving the ownership gate. FE rows add mobile-viewport screenshots per behavior, including the rollup showing 7 loại for the Bàn 04 example and all six render branches.

What the reference did instead. It published item_progress and had no consumer case, so cooking progress never moved. It listened for order.status while the BE published order_status_changed, so the badge never advanced. It opened a second SSE connection per opened detail sheet. And it never checked ownership on the stream, so any authed client could watch any order. Each of those is closed structurally here — one gated stream per page, one event-contract file, one exhaustive CI-gated switch.
