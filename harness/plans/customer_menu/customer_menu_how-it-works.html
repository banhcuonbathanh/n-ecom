The three kinds of traffic — drawn

The page's entire network life in three lanes: reads are double-cached, the cart is silent, the single write bypasses Redis.

Catalog reads — public, cached twice
browser → GET /products·/categories·/combos
→ Gin → Redis (hit? done) → MySQL → Redis SET

Cached in Redis (5-min TTL) and in the browser's Query cache (staleTime 5 min — deliberately the same number). Most guests never trigger a MySQL read.

The cart — never leaves the browser
tap → Zustand cart store → subscribers re-render
network: NONE

Add, remove, nhân pills, canh stepper, note — all pure client state. The server first hears about the cart at confirm time.

Order write — never cached
confirm → POST /orders (cookie JWT)
→ Gin → service TX → MySQL → 201 full order

One pessimistic write. Redis is not involved — orders are never cached. Prices/names are snapshotted server-side inside one transaction.

The one-sentence version: the menu is a read-mostly page with a client-side cart — catalog data flows down through two caches (Redis, Query), the cart lives entirely in Zustand, and exactly one write (POST /orders) crosses back, carrying ids + quantities only.

02

First load — the 3-tier loading strategy, step by step

Guest opens /menu (QR path or direct). This is Tier 1 — route (PLAN.md §4.3), drawn first, then narrated:

Tier 1. The skeleton (step 2) mirrors the real layout — banner + nav + rail + card grid, never a centered spinner. Data rides inside the HTML (step 7) — no second round-trip.

  1. 1
    browser requests /menu. QR path: the guest JWT cookie (with table_id) is already set from /table/:id. Online path: a table-less JWT gets minted silently.
  2. 2
    next server starts rendering page.tsx (a React Server Component) and immediately streams loading.tsx to the browser. The skeleton mirrors the real layout — a deliberate upgrade over the reference's centered spinner.
  3. 3
    next server prefetches all three catalog queries in parallel: GET /products, GET /categories, GET /combos.
  4. 4
    gin api handles each read: service asks redis for products:list / categories:list / combos:list. Warm (normal case): Redis hit → JSON straight back, MySQL untouched. Cold: miss → mysql SELECT → Redis SET with 5-min TTL → JSON back. Redis down? Fail-open: serve from MySQL anyway (§07).
  5. 5
    next server renders the page HTML and serializes the three query results into a HydrationBoundary.
  6. 6
    browser hydrates: client components mount and their useProducts() / useCategories() / useCombos() hooks find the cache already full. Zero-spinner first paint. Identical query keys (from keys.ts) are what make the RSC prefetch and the client hooks hit the same cache entries.
  7. 7
    browser — ComboSection joins combo combo_items (ids only) against the cached products query to get names/prices. One source of truth: product data exists once, in the products cache — never duplicated in the combo payload.
  8. 8
    browser — Zustand rehydrates the persisted cart slice from localStorage: {orderNote, activeOrderId, tableId, tableName}. Cart items are session-only by design — a refresh keeps your table identity but empties the tray.

Same 5-minute number on purpose: "how old can what I see be?" has one answer, not two.

t=0     guest loads menu ─ data enters Query cache (fresh)
t<5min  guest scrolls, navigates to detail + back ─ cache is FRESH → no refetch at all
t>5min  data marked stale ─ next mount/focus triggers a BACKGROUND refetch:
        old data stays on screen (isFetching), new data swaps in silently
        → no flash, no skeleton re-entry, ever
focus   refetch-on-focus stays ON → a menu left open on a table tablet
        self-heals ≤ 5 min after an admin 86's a dish ("Hết" overlay appears)

Detail pages get the same treatment early: hovering/touching a card fires prefetchQuery for /menu/product/[id], so tapping through lands on cached data. Each detail segment has its own layout-shaped loading.tsx for the cold-URL case. Card images: buildImageURL(image_path) + native loading="lazy" below the fold + fixed aspect-ratio boxes so cards never reflow.

Tier 3 — mutation doesn't exist for the cart (next section) and shows up exactly once on this page: the pessimistic POST /orders in §06.

03

Add a product — what actually happens (zero network)

Guest taps "Bánh cuốn trứng", picks nhân "Mộc nhĩ", taps add. Everything happens inside the browser — no API call, no Redis, no MySQL:

One writer, three readers, zero requests. Derived numbers are selectors — they can't go stale.

Anatomy of a cart line id — the variant IS the id

product_p12…36_t2=Bánh cuốn trứng + Mộc nhĩ (single-select nhân)
combo_cb1…36_t1t2=Suất đầy đủ + Nhân thịt + Mộc nhĩ (multi-select, ids sorted)
canh_p_canh…_rau=canh có rau — from the OrderSummary stepper, never a card

kind · product/combo id · variant. Add Trứng+Mộc nhĩ twice → same id → one row, qty 2. Add Trứng+Nhân thịt → different id → its own row. Dedup falls out of the scheme — no comparison logic in the store.

  1. 1
    browser — the card's nhân pill selection is plain useState inside ProductCard. Product cards: single-select, first option pre-selected. Combo cards: multi-select, "Nhân thịt" default, last one can't be deselected. Local state — nobody else needs it yet.
  2. 2
    browser — on add, the card computes the line id (anatomy above), which encodes the variant.
  3. 3
    browser — cart.addItem(line) hits the Zustand store: same line id exists → quantity +1; new id → new row.
  4. 4
    browser — every subscribed component re-renders from the store, via selectors: CartBottomBar badge → itemCount() (13) · OrderSummary gains/updates the row + total() (103.000 đ) · the card shows its qty stepper. total()/itemCount() are selectors, never stored numbers (FE rule 11) — they cannot go stale.
  5. 5
    browser — the fly-to-cart animation plays. That's the entire feedback loop. No optimistic-update machinery, no rollback, no spinner — there is nothing to wait for. This is why Tier 3 is "none" for the cart.
What does NOT happen: no component keeps its own copy of a cart line (useState mirrors are banned — that exact pattern caused the reference's 🔴 lost-note bug), nothing is written to the server, and only {orderNote, activeOrderId, tableId, tableName} touch localStorage — the items themselves die with the tab (owner-accepted: table sessions are short).

The same store handles every other cart interaction: the canh stepper in OrderSummary writes canh_* rows bound to the two real soup products; the note field writes cart.orderNote; the qty steppers and row deletes edit lines in place. One store, many surfaces.

04

Inside the cart store — nhân · canh · combos, and how Zustand actually works

§03 showed the fan-out; this section opens the box. The three things a guest builds — a nhân choice, a canh bowl, a combo — all land as rows in one Zustand store, and that store's three mechanics (set(), selector subscriptions, persist) explain every re-render §03 waved at.

1 · Nhân — the ₫0 topping that becomes the line id

Nhân (fillings) are not a separate field — they are the ₫0 toppings joined into each product (PLAN §3.4). The pill widget behaves differently by surface:

Bánh cuốn trứng — product card
single-select · first option pre-selected · plain useState inside the card
Mộc nhĩ ✓Nhân thịtChay
product_p12…36_t2→exactly one nhân id in the tail
Suất đầy đủ — combo card
multi-select · "Nhân thịt" default · the last on-pill can't be deselected
Nhân thịt ✓Mộc nhĩ ✓Chaymin 1 🔒
combo_cb1…36_t1t2→ids sorted so t2+t1 and t1+t2 collapse to one row

The selection lives as useState on the card until you tap add — nobody else needs it yet (FE_STATE §1). On add, the card bakes it into the line id and calls addItem. Because the variant is in the id, dedup is free: same nhân → qty +1; different nhân → a new row. No comparison logic ever runs in the store.

Why sorted ids matter: picking Nhân thịt-then-Mộc nhĩ must equal picking them the other way around. Sorting the id tail (_t1t2) makes the two strings identical, so they merge into one row instead of splitting the tray.

2 · Canh — a stepper, never a card

Soup is filtered out of every list and out of search (isCanhProduct() — by the seeded CANH category_id, never by name: F34; PLAN §4.4-4) — a guest can only pick canh from the stepper in OrderSummary, bound to the two real canh products:

The pill still fires even while dimmed — the canh gate is a guardrail, not a disabled button (PLAN §4.4-5).

3 · Combo — one visible row, many wire items

A combo card's multi-select nhân produces a single combo line (combo_<id>_<sortedNhânIds>) shown as one row with a "Chi tiết" expander. The combo payload is ids-only (PLAN §3.1-3): ComboSection joins names and prices from the already-cached products query — never a second copy. What that one row unfolds into downstream is the §06 four-shape pipeline; inside the store it stays a single deduped line.

4 · How Zustand actually works — one store, outside React

The cart store is created once by create()<persist> as a module-global object. It lives outside the React tree — no <Provider>, no context. A component reaches it by importing the hook and passing a selector; that is the entire wiring.

One store, three doors: actions write through set(), subscribers read through a selector, persist mirrors a four-field slice to localStorage. No provider is anywhere in the picture.

  1. 1
    browser — a component subscribes with a selector: useCartStore(s => s.itemCount()). Zustand remembers that slice for that component.
  2. 2
    browser — an action runs set() with an immutable update (a new lines array, never a mutation). Zustand swaps the state object. Actions are the only writers. A component never assigns to the store — it calls an action, which calls set().
  3. 3
    browser — Zustand re-runs each subscriber's selector against the new state and shallow-compares the result with the previous one.
  4. 4
    browser — a component re-renders only if its own selected slice changed. Add a bánh row and the itemCount() badge + total() update; a different card's stepper — whose slice is untouched — does not re-render. This selective subscription is exactly why "one writer, three readers" in §03 is free: surfaces that didn't change stay still.
  5. 5
    browser — persist writes the partialized slice to localStorage on every change; a reload rehydrates {orderNote, activeOrderId, tableId, tableName} only — the tray starts empty by design (§03).

The mechanism in one line: nhân / canh / combo choices all become line ids in one Zustand store; set() swaps state immutably, per-component selectors decide who re-renders, and persist+partialize keep only your table identity across reloads — derived money (total()) is never stored, so it can never drift.

05

State across components and pages — two hubs, nothing else

Distant components never talk to each other directly — no prop drilling, no context providers, no event buses. Everything shared flows through exactly two hubs:

HUB 1 · TanStack Query cache — server data
  • →MenuCategoryNav, ProductList, ComboSection, FavouritesRail all call the same hooks from queries/catalog.ts
  • →identical keys dedupe to one fetch, one cache entry
  • →the combo join happens here: ids-only combos + cached products
Rule 1's "never": Query data is never copied into a store or useState.
HUB 2 · Zustand stores — cart + favourites
  • →a card deep in a section, CartBottomBar, OrderSummary, TableConfirmModal — all read/write the store only
  • →derived values are selectors (total(), itemCount()) — can't go stale
  • →the reference's 🔴 bug was exactly a hub bypass: the modal kept orderNote in its own useState and dropped what OrderSummary wrote
favourites store: {items[], sets[], suats[]} — device-local, never synced.

Everything else deliberately stays small: modal/expander open = useState in its owning component; search text lives in the URL (?q= — shareable, survives reload); the guest session is an httpOnly cookie the client JS can't even read — there is no auth store.

06

Confirm — POST /orders through every layer

Guest taps "Thanh toán" (canh gate passed), then confirms in TableConfirmModal. This is the page's first and only server write — pessimistic (FE rule 4): nothing on screen pretends success before the 201.

While this runs, the confirm button is disabled with an inline "Đang gửi…" — never a full-page overlay (Tier 3, pessimistic).

The same flow as prose

    browser — the modal reads everything from the cart store: lines, tableId, and orderNote (the note OrderSummary wrote — the reference's silent-loss bug, designed out).
  2. 2
    browser — buildOrderItemsPayload() (the one builder, 3 callers) shapes the wire payload: {product_id|combo_id, quantity, topping_ids} per line. No prices, no names on the wire — the server snapshots both. Canh lines are stripped from combo overrides so the BE never double-expands.
  3. 3
    browser — client.ts fires POST /api/v1/orders; the httpOnly guest-JWT cookie rides along automatically. Online path posts source:"online" + name/phone from /checkout instead.
  4. 4
    gin api — middleware verifies the JWT cookie (guest, unexpired, table binding); handler binds + validates the body. Bad quantity? → 422 envelope with details:[{"field":"items[0].quantity","issue":"min=1"}] and nothing is written.
  5. 5
    gin api — the order service opens one MySQL transaction: fetch current prices/names (never trusts the client) → freeze toppings_snapshot → expand combos → RecalculateTotalAmount → commit, or roll back everything.
  6. 6
    mysql — orders + order_items land atomically. redis is not touched — orders are never cached. Item progress later derives from qty_served — there is no item status column to drift.
  7. 7
    gin api → 201 with the full order object — total_amount: 103000. Lesson 7: thin {id} DTOs caused "Đơn #undefined" in the reference. Never again.
  8. 8
    browser — the handoff: cache the order → clearCart() keeping identity fields → setActiveOrderId(id) → router.replace('/order/<id>'). "Gọi thêm" later re-enters the menu in append mode and posts to POST /orders/:id/items.
07

Redis life — cache-aside, invalidation, fail-open

Redis touches this page in exactly one pattern — cache-aside, inside services only (handlers and repositories never see it). Three scenarios cover everything:

HIT  (the common case, <5 min after any read)
  gin ──GET products:list──▶ redis ──JSON──▶ gin ──▶ browser        mysql: idle

MISS (first read, or TTL expired)
  gin ──GET products:list──▶ redis ─(nil)─▶ gin
  gin ──SELECT … is_available=1, toppings joined──▶ mysql ──rows──▶ gin
  gin ──SET products:list  EX 300──▶ redis          then JSON ──▶ browser

FAIL-OPEN (redis down)
  gin ──GET──▶ redis ✕ error → log it, go straight to mysql ──▶ browser
  the menu never breaks because the cache broke

How stale can a guest's screen get?

admin 86's "Bánh cuốn trứng"
  └─▶ DEL products:list                    (server truth: immediate)
      └─▶ next GET /products = MISS → fresh from mysql → re-cached
guest already on /menu
  └─▶ Query staleTime (≤5 min) or a tab re-focus triggers a background refetch
      └─▶ "Hết" overlay appears on the card, old data never flashes away
worst case ≈ one staleTime window — bounded, self-healing, no reload needed
  • TTL 5 min on every catalog key — even a missed DEL heals itself within one TTL.
  • Orders are never cached — correctness over speed for money-bearing data.
  • The FE's staleTime matches the TTL on purpose: both caches expire on the same clock, so "how old can what I see be?" has one answer, not two (§02's clock bars).
