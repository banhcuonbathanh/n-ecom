Customer menu page — FE + BE build plan

One plan for the highest-traffic surface: the customer-facing /menu of the dine-in bánh cuốn restaurant. The page is a contract — public cached catalog reads, a client-only Zustand cart, and the order-creation handoff on a guest JWT — so frontend and backend are planned together.

source of truthPLAN.md (this folder)

ships acrossC-1…C-5 · T · O rows in TASKS.md

Two ways a guest lands here:

QR path — scanned /table/:id first (T phase): the guest JWT carries table_id; ordering posts straight to the table.

Online path — arrived directly: a table-less guest JWT is minted silently (POST /auth/guest/online); checkout collects name/phone.

Core loop: browse categories/combos → build a cart (client-only) → confirm → POST /orders → live order tracking. The page re-enters in append mode (?add_to_order=<id>) via "Gọi thêm" on /order/:id to add items to an active order.

Cart-line id scheme — every filling combination is its own row

product_<id>_<nhânId|plain>          one nhân, single-select → one row per variant
combo_<id>_<sortedNhânIds|plain>     multi-select set, sorted → stable dedup key
canh_<productId>_<rau|plain>         canh rows bound to the two real soup products

→ addItem dedups naturally on id; the payload builder strips canh lines from
  combo overrides so the BE never double-expands.

Persist rules

  • total() / itemCount() are selectors, never stored (FE rule 11).
  • partialize: persist only {orderNote, activeOrderId, tableId, tableName} — cart items are session-only; versioned storage key with a migrate fn.
  • After POST /orders 201: cache order → clearCart() keeps identity fields + setActiveOrderId(id) → router.replace('/order/<id>') (reference GAP-2 decision).

How state crosses components — two mechanisms, only two

1. server data → Query cache
   MenuCategoryNav ─┐
   ComboSection    ─┼─→ same hooks (queries/catalog.ts) → same keys → ONE fetch
   ProductList     ─┘    combos are ids-only: ComboSection joins names/prices
                         from the cached products query — never a second copy

2. cart/favourites → Zustand stores
   ProductCard.add ──┐
   OrderSummary edit ┼─→ cart store ←── CartBottomBar badge (selectors only)
   TableConfirmModal ┘        └─ modal READS cart.orderNote at POST time
                                 (the reference's own-useState 🔴 bug, designed out)

everything else (modal open, expander open) = useState in its owner — NOT shared

How state crosses pages

  • Menu → product/combo detail: nothing is passed — the detail page reads the route param and hits the same Query cache (prefetched on card hover); the cart store is global, so the cart survives navigation for free.
  • Menu → order page: the post-201 handoff above — order id travels in the URL, table identity in the persisted store slice.
  • RSC → client: page.tsx prefetches the three catalog queries → HydrationBoundary hydrates the same cache — zero spinners on first paint.

New state during C-4/C-5? Run the FE_STATE §1 flow

Zustand is the LAST resort — this page needs exactly two stores,
and Query data is never copied into a store (FE rule 1's "never").

BE contract — endpoints, schema, cache map

Endpoints the page consumes (§3.1 — all under /api/v1)

#RouteAuthTaskBehavior
1GET /categoriespublicC-2 Redis categories:list, 5-min TTL. Fields: id, name, description, sort_order, is_active.
2GET /productspublicC-2 Redis products:list. Only is_available=1, soft-delete filtered; toppings joined via product_toppings. Deliberately no query params in v1 — filtering/search is client-side; manager surfaces use GET /products/all (uncached, role-scoped — FE rule 13).
3GET /combospublicC-3 Redis combos:list. Combo + combo_items: [{id, product_id, quantity}] — ids only; FE joins names/prices from the products query (one source of truth, no duplicated payloads).
4POST /auth/guest/onlinepublic, rate-limitedT No body → sets 2 h guest-JWT httpOnly cookie (table-less, source='online'). Reference's {access_token}-for-memory-store shape not adopted (F-5 cookie decision).
5POST /ordersguest JWTO Cart → order: server-side price/name snapshot, toppings_snapshot JSON, combo expansion (header row unit_price=0 + component rows via combo_ref_id), RecalculateTotalAmount in the same tx. Returns the full order object (lesson 7: thin {id} DTOs caused "Đơn #undefined").
6POST /orders/:id/itemsguest JWTO Append mode; same items payload; recalc total in tx.

Errors ride the Session-0 envelope; codes from BE_STATE.md §4.

06

Behaviors spec (§4.4) — the C-4 / T / O acceptance criteria

B1Anchors, not filters

Every section always renders on one scrollable page. Tab order Tất cả · SUẤT · SUẤT TỰ TẠO · <categories by sort_order> (TRỨNG · BÁNH CUỐN · GIÒ). Tap smooth-scrolls; scrolling spies the active tab (orange text + underline). The nav is the page's only sticky element.

B2Search mode (≥ 2 chars)

Reshapes the page — hides nav/rail/combo sections, renders one flat filtered list (client-side) or the empty state; 1 char shows "Nhập ít nhất 2 ký tự"; ✕ restores the sectioned menu.

B3Nhân pills, inline on cards

Product cards single-select (first pre-selected; Bánh Chay/Giò have none) · combo cards multi-select ("Nhân thịt" default, last one can't be deselected). Each combination = its own cart row. No modal, ever.

B4Canh is never a card

Soup products are filtered out of lists and search (isSoupName) and chosen only via the OrderSummary stepper — two rows bound to the two real canh products (có rau / không rau).

B5Canh gate

Zero canh bowls → "Thanh toán" pill dims but still fires → error toast + auto-open the cart list + scroll-and-shake the canh box.

B6Cart surfaces

On-page OrderSummary (line editor, per-combo "Chi tiết" expander, "TỔNG SỐ MÓN" rollup, table pill) + CartBottomBar's two floating pills (count badge, no total) — one store, selectors only.

B7Order note

Typed in OrderSummary → cart.orderNote → read by TableConfirmModal at POST time (the reference's silent-loss bug, designed out); starts empty (owner-locked — no pre-fill).

B8Two checkout paths

Table bound → TableConfirmModal → POST /orders (source:'qr'); no table → /checkout (source:'online'). Append mode: ?add_to_order=<id> → POST /orders/:id/items.

B9Cards

Price VND via formatVND() in the reference's format ("30.000 đ"), favourite heart (toggle only, stopPropagation), qty stepper, "Hết" overlay when 86'd, fly-to-cart animation on add.

B10VN-first copy

All strings Vietnamese, exactly the reference's (digest §9 set — "Quán Bánh Cuốn", "Tìm món nhanh...", "Thanh toán", …), one constants file, no i18n framework yet.

B11Images

API sends relative image_path; FE builds the URL (FE rule 14) — the reference defined buildImageURL and never called it; ours is called or it doesn't merge.

B12Worked example everywhere

Docs, seeds, screenshots all use the canonical numbers: Bàn 04 · 103.000 đ · badge 13.

09

Decisions + flags (§7)

⚠️ FLAG — customer-surface theme (awaits owner input before C-4). The reference DESIGN_PROMPT (visual source of truth, owner-locked there) is dark + warm orange (#0b0f17 bg, #f97316 the only bright color, Playfair Display serif title) — but our F-7 design system set brand primary #2B59D9 harness blue with "owner may swap hue before C-4".

Plan default: the customer shell adopts the reference's dark/orange as its semantic token values (staff/admin surfaces keep F-7's neutral system); the hue swap lands as one token change in the C-4 task. Say so before C-4 if blue should win. The §01 mockup previews the plan default.

  • ✅ Client-side filtering/search in v1 — a one-restaurant menu is small (~tens of items); one cached products:list + client filter beats param plumbing (the reference sent ?search= and the BE ignored it — we ship no ghost params). 💡 Revisit only if catalog size or multi-branch changes the math.
  • ✅ CartDrawer / MiniCartStrip / recovery banners: not built in v1 — the reference's own final page disabled all of them (drawer has no trigger); cart editing lives in OrderSummary. Re-opened, if ever, by the O-phase recovery task.
  • ✅ Combo payload = ids only, FE joins from the products cache (reference shape kept — it's the no-duplication choice).
  • ✅ Canh gate + ₫0-topping fillings adopted as business rules of the house menu.
  • ✅ Cookie JWT everywhere (F-5) — supersedes the reference's memory token; the "F5 wipes the session, re-scan the QR" defect disappears structurally.
  • ⚠️ Cart-items persistence: reference persists only note + activeOrderId (items die with the tab). Kept — table sessions are short; full-cart persistence is a one-line partialize change if the owner wants it.
  • ❓ Append-mode auth window: adding items to a 1.5 h-old order with a 2 h JWT can expire mid-flow; UX for re-mint lands in the O-phase task (default: toast + re-scan prompt, per reference tracking behavior).
