Customer favourites suite — FE + BE build plan

Three small routes that let a returning guest re-order "the usual" without hunting the menu. The whole suite is BE-read-only: it calls the two cached catalog GETs the menu page already owns, resolves hearted ids into cards, and writes nothing to the backend — favourites and saved sets live in favourites.store.ts (localStorage).

A bridge, not an endpoint. The guest never creates a favourite here — the heart is toggled on /menu (product card, combo card, favourites rail) and lands in the persisted store. This suite reads that back and turns it into a filled cart.

Core loop: open Favourites → (auto-prune anything 86'd) → adjust qty / filter → bulk-add or apply a set → cart badge bumps → check out via the menu's TableConfirmModal / /checkout. It consumes heart toggles from /menu and produces a cart for the order pipeline — it owns neither endpoint of that journey.

Worked example everywhere: 3 favourites — Bánh Cuốn Thịt (Món lẻ, 4.000 đ) · Bánh Trứng Vàng (Món lẻ, 9.000 đ) · Combo Đầy Đặn (Combo, 42.000 đ) — and 2 saved sets — "📋 Sáng thứ 7" (3 món · 88.000 đ) · "📋 Cả nhà" (5 món · 152.000 đ).

BE contract — the whole story is "there isn't one"

The single most important alignment: this suite adds no new BE surface.

2 public GETs  ·  0 writes  ·  0 new endpoints  ·  0 new tables  ·  0 new cache keys  ·  0 invalidations

Every endpoint, cache key, and wire shape it uses is already owned by the menu plan. This plan only points at them: see customer_menu_PLAN.md §3 for the endpoints, the cache map, and the frozen wire shapes — one shape, one home, not restated here.

Endpoints the suite consumes (§3.1 — all under /api/v1)

#RouteAuthPhase/TaskBehavior
1GET /productspublicC-2 already built Redis products:list, 5-min TTL; only is_available=1, soft-delete filtered; toppings pre-joined. Identical call the menu page makes.

There is no favourites endpoint — no POST /favourites, no "save to account". The full trace (menu plan §3.1) proves both routes are bare .GET("") on their groups with no middleware; a token-less guest can open the whole suite.

06

Behaviors spec (§4.4) — the C-6 acceptance criteria

B1Read-only, local-first

The suite issues only GET /products + GET /combos and zero writes; every action (un-heart, qty, save set, add-to-cart) is a local store mutation. Opening it with no guest token still renders resolved cards.

B2Resolve + auto-prune

Store items[] ids are joined against the catalog; any id absent from the payload is dropped and a single toast fires (after both queries settle). Combo component names resolve from the products list; a missing referenced product shows Món không rõ tên (cosmetic, not an error).

B3Filter tabs

Tất cả / Món lẻ / Combo, each with a live count; active = solid orange; filters the list client-side. Counts are selectors over resolvedItems.

B4Card = remove + re-add control

The filled orange heart un-favourites and removes the card (it is not add-to-cart, stopPropagation); a qty stepper (default 1) edits the line; [+ Giỏ] adds that card to the cart.

B5Bulk add

Footer 🛒 Thêm tất cả vào giỏ turns every resolved favourite into a CartItem and pushes it additively (existing cart survives). Cart badge updates.

B6Save a set

TopNav [Lưu bộ] → /menu/favourites/save; RHF+Zod (name min 1) and resolvedItems.length > 0 gate the submit; addSet(name) snapshots items[] into the persisted sets[] → routes to /menu/favourites/sets.

B7Apply / manage a set

SetCard 🛒 Áp dụng resolves the snapshot against the live catalog (silently skipping removed items) and adds all to cart; ✏ renames inline; 🗑 deletes. Empty sets page → dedicated empty state (gated on sets.length, not query).

B8Footer clears the shell nav

The CTA / live-total row sit above the fixed 5-tab ClientBottomNav (~72px bottom padding) — never hidden behind it (reference bug fixed).

B9Distinct load / empty / error

The three are never conflated (§05): a network failure shows the error branch, not the empty state.

B10VN-first copy

All strings Vietnamese, exactly the reference's ("Yêu Thích", "Thêm tất cả vào giỏ", "Bộ đã lưu", "Nhấn ♥ trên món ăn bất kỳ để thêm", …); one constants file, no i18n framework yet.

B11Currency

formatVND() → 35.000 đ (dot thousands, lowercase đ, space); zero-price items show 0 đ. Same helper as the menu.

B12Worked example everywhere

Docs, seeds and screenshots all use: 3 favourites — Bánh Cuốn Thịt (Món lẻ, 4.000 đ) · Bánh Trứng Vàng (Món lẻ, 9.000 đ) · Combo Đầy Đặn (Combo, 42.000 đ) — and 2 saved sets — "📋 Sáng thứ 7" (3 món · 88.000 đ) · "📋 Cả nhà" (5 món · 152.000 đ).

09

Decisions + flags (§7) + verify plan (§8)

Decided ✅

  • ✅ BE-read-only, a strict subset of the menu contract — no favourites endpoint, no new cache keys, no schema. The plan cross-links the menu plan §3 rather than restating it.
  • ✅ Favourites persisted, cart session-only — the deliberate asymmetry (a set outlives the tab; the cart it fills does not). Adopted from the reference.
  • ✅ Read-only ⇒ loading/error branches are the whole FE story — the reference's blank states are the one real thing to fix; §05 is this plan's highest-value slice.

Flags ⚠️

⚠️ FLAG — favourites are device-local, not account-synced. Clearing storage or switching device loses them; there is no "save favourites to my account" (no accounts in v1). If the owner wants cross-device favourites, that needs a favourites table + a POST/GET /favourites pair + auth — a new BE surface, out of this plan's read-only scope.
