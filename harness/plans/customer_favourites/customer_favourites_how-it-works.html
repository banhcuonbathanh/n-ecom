01

Big picture — the whole machine on one screen

The favourites suite is a bridge. It does not create favourites (the heart lives on /menu) and it does not place orders (that is /checkout → POST /orders, owned by the menu plan). It reads a persisted list back, resolves it against the shared catalog cache, and hands a filled cart to the order pipeline.

Hearts flow in from /menu, a cart flows out to the order pipeline. The suite owns neither end of the journey — only the middle.

All of it is a Zustand write. The favourites store persists to localStorage; the cart store does not persist its items. Tier-3 mutation UI never appears — there is nothing to be pending about.

The first server write in the whole journey happens after the guest leaves this suite. Drawn greyed everywhere on this page: we hand off, we don't call it.

The one-sentence version: favourites is a read-only, local-first bridge — it consumes hearts written on /menu, resolves them through the menu's own cached catalog GETs, and produces a cart for the menu's own order pipeline. It adds no endpoint, no cache key, no table.

02

First load — the warm arrival and the cold open

There are two ways in, and they look completely different at runtime. The common one is warm: the guest was just on /menu and taps [Yêu Thích] in the fixed bottom nav.

Warm arrival — the common path, zero spinner

Because queries/catalog.ts is shared with the menu, warm arrival is a cache hit. This is why the named loading branch below matters only on a cold direct open.

Cold open — someone pastes /menu/favourites straight into the browser

Tier 1 is the route skeleton (step 2); Tier 2 is the four-branch decision (step 8). Tier 3 (mutation UI) never fires — nothing here mutates the server.

The /sets page's empty state is gated on sets.length, not on query state — saved sets exist whether or not the catalog answered.
This is the reference defect we designed out. The old build rendered the blank EmptyState whenever resolvedItems was [] — which is also true while the queries are in flight and true when they failed. Cold-load, genuinely-empty and fetch-failed were pixel-identical. Gating on isPending / isError instead of on array length is the whole fix, and it is the highest-value slice of this plan.

Worked example, warm path: the store holds three ids; within 5 min of leaving /menu the DATA branch paints immediately with Bánh Cuốn Thịt 4.000 đ, Bánh Trứng Vàng 9.000 đ and Combo Đầy Đặn 42.000 đ — tabs read Tất cả (3) · Món lẻ (2) · Combo (1).

03

The resolve + auto-prune pass

The store holds ids, not dishes. Names, prices and availability come only from the catalog payloads. So every mount runs a join — and anything that no longer joins is a favourite for a dish the kitchen has taken off the menu.

storeitems[]
4 ids · qty · toppingIds
×
query cacheproducts[] + combos[]
available dishes only
→
lib/favourites-resolve.tsresolvedItems
3 cards — id #4 found nothing
→
useEffect (guarded)removeItem + 1 toast

Pull-based, not push-based: nothing tells the browser a dish was 86'd. Its absence from the payload is the message.

Why the guard exists

Two queries resolve independently. Without productsLoaded && combosLoaded, the effect could run in the window where one payload has arrived and the other has not — and every id belonging to the missing payload would look "absent":

UNGUARDED (the bug we avoid)
  t0   mount ─ products: pending      combos: pending
  t1   products 200 ✓ ────────────▶  effect runs: combos[] is still []
                                      → "Combo Đầy Đặn" resolves to nothing
                                      → removeItem('c_7')  ✗ WRONG — and PERSISTED
  t2   combos 200 ✓                   too late; the favourite is already gone from localStorage

GUARDED (ours)
  t0   mount ─ both pending           effect body short-circuits
  t1   products 200 ✓                 productsLoaded=true, combosLoaded=false → still no-op
  t2   combos 200 ✓                   BOTH true → run once: drop only genuinely-absent ids
                                      → removeItem('p_88') + ONE toast

Why this is worth a guard rather than a retry: the prune writes through the persist middleware, so a wrong prune is not recoverable by refreshing — it has already been committed to disk. The guard is the only thing standing between a slow combos request and a silently emptied favourites list.

Applying a saved set runs the same join with a softer policy: resolveSet() silently skips items no longer on the menu and adds the rest — no toast, no prune of the stored set. "📋 Sáng thứ 7 — 3 món · 88.000 đ" can add two lines and stay a 3-món set.

The domain model — three slices, none of which ever crosses the wire

Prices are not stored — every number above is re-derived from the catalog payload at render time, so an ₫-change on the menu shows up here within one cache cycle (§07).

How Zustand persist actually works

Persist is just a write-through wrapper around set() plus a read on boot. There is no sync, no server, no conflict resolution — which is exactly why favourites are device-local.

FLAG — device-local, not account-synced. Clearing site data or switching phones loses every favourite and every saved set. Cross-device favourites would need a favourites table + a POST/GET /favourites pair + auth — a new BE surface, outside this read-only plan. Raise it before C-6 if it belongs in v1.
05

State across components & pages

Two mechanisms carry state here, and only two: server data → the Query cache and favourites/cart → the Zustand stores. Nothing travels in the URL between these routes — each page simply re-reads the store on mount.

HUB 1 — TanStack Query cache
  • →useProducts() — the list, the save summary, the sets page
  • →useCombos() — combo cards + set component lines
  • →identical keys to /menu: three routes + the menu dedupe to one fetch
The combo → product-name join happens FE-side against this same cache. Product data exists once, never a second copy.
HUB 2 — Zustand stores
  • →favourites (persisted): items[] · sets[] · suats[]
  • →cart (items session-only): the hand-off target
  • →writes are additive — "Thêm tất cả" / "Áp dụng" append, never replace
No prop drilling, no context, no params. A route change is not a state transfer — it is a second reader of the same store.

06

The read path through every layer — and the write path that isn't here

One request, followed all the way down and back. GET /combos is the same trip with a different key.

Six hops down, one dashed hop back. The bottom band is the whole BE-write story of this suite: it doesn't have one.

HopWhat the data is at this point
store items[]ids only — {id, type, qty:1, toppingIds:[]}. No name, no price, ever.
Gin handlerno params, no body, no auth header — the response is the same for every guest
Redis hitthe serialized list, already toppings-joined; MySQL stays idle
Query cache entrythe payload, shared verbatim with /menu and both sibling routes
resolvedItemsthe join result — 3 cards with live names/prices; unresolved ids already dropped
cart linea CartItem built by buildCartItemsFromFavourites() — still client-side
POST /ordersids + quantities; the server re-snapshots names and prices inside one transaction — menu plan territory

Read-only has a real consequence: there is no optimistic update, no rollback, no pending spinner on any button, and no error envelope to render for a guest action. Every failure mode on this page is a read failure — which is why §02's error branch is the entire resilience story.

07

Cache life — two clocks, one bound, zero invalidations

The suite piggybacks entirely on the menu's cache keys. It reads them; it never sets them, never deletes them.

browser · Query staleTime
FRESH — no refetch at allSTALE — background refetch, old cards stay visible…
server · Redis TTL (EX 300)
CACHED — mysql idleEXPIRED — next read repopulates from mysql…
0 min5 min10 min

Same 5-minute number on both sides, on purpose. Stacked worst case: a favourite's name, price or availability can be up to ≈ 10 minutes behind the kitchen.

t=0      staff 86's "Bánh Cuốn Thịt"  ─▶  admin write DELs products:list   (server truth: immediate)
                                          ↳ this suite plays NO part in that fan-out
t≤5min   a guest's Query cache still holds the old payload
         → the card is still on screen, still addable          (worst case #1: staleTime window)
t=5min   staleTime lapses ─▶ background refetch on next mount / tab focus
         → if Redis had also just re-cached a stale list, add up to one more TTL   (worst case #2)
t≈10min  the dish is absent from the payload ─▶ resolve drops it
         ─▶ guarded useEffect fires removeItem + ONE toast (§03) ─▶ persisted
bounded, self-healing, no reload needed — and no code on these routes had to know about it
  • TTL 5 min on both catalog keys — even a missed DEL heals itself within one TTL.
  • Fail-open: Redis down → the service skips to MySQL, logs, and still answers 200. The suite survives a Redis outage with no visible difference beyond latency.
  • No invalidation from here — the suite has nothing to invalidate with. Every one of its writes stops at localStorage.
  • Refetch-on-focus keeps a favourites page left open on a phone self-healing: an 86'd favourite is pruned the next time the tab is focused and the data comes back changed.
  • The staleness bound is the honest answer to "why did my favourite disappear a few minutes after it was 86'd?" — ≈ 10 min, bounded, never unbounded.
