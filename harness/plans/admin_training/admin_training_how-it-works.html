Đào tạo nhân viên — how it actually works

Not what we will build — what happens when it runs. Seven numbered end-to-end sequences for /admin/training: first load, the authoring loop, the roster query that makes the tracking table honest, the staff write path the reference never built, the notes write, and why nothing here is cached.

🔴 The defect this whole runtime designs out. In the reference system, no API path ever wrote training_progress or quiz_attempts. The sqlc queries and repository methods existed — UpsertStaffProgress, InsertQuizAttempt — but nothing called them: no service method, no route.

So three of the page's seven endpoints could only ever return empty or 404. The completion table always read "Chưa có nhân viên nào được giao hướng dẫn này.", the detail modal always read "Nhân viên này chưa bắt đầu hướng dẫn.", and manager notes 404'd silently on save. The page implied it tracked training and recorded nothing. Sections 04 and 05 are the two mechanisms that fix it.

01

Big picture — the whole machine on one screen

Two managers, one Gin server, one MySQL. That is the entire cast. There is no Redis in the training domain and no SSE/WS — both deliberate (§07).

The one-line mental model. Training is a plain REST CRUD domain over MySQL whose only interesting mechanic is a join: the completion table is driven from the staff roster, not from the progress rows. Everything else is ordinary.

02

First load — three tiers, five branches

  1. RSC page.tsx runs on the server and prefetches the guides query for the role in ?role= (default all). While it awaits, loading.tsx streams a skeleton that mirrors the real layout — header, tab strip, card grid, table — never a centered spinner.
  2. HydrationBoundary hands the cache over. The client mounts with the guides already in the TanStack cache, so a warm server paints with zero spinner.
  3. The client picks one of five branches (below). They are mutually exclusive and every one is built — the reference documented three and shipped a fourth by accident.
  4. The tracking table fires its own query for the guide in ?guide= (defaulting to the first guide of the active role), enabled: !!guideId.
  5. Covers lazy-load. Relative cover_image_path → buildImageURL(), fixed aspect-ratio box, so cards never reflow as images arrive.

Two upgrades over the reference, both in this sequence.

① It had no loading.tsx at all and was 'use client' top to bottom — first paint was a bare shell, then a fixed 4-card pulse regardless of how many guides were coming.

② Its guide-list error replaced the entire page, header and tabs included, and the only recovery offered was window.location.reload(). Ours scopes the error to the content region and retries the query, not the document.

03

The authoring loop — create / edit a guide

This is the half the reference got right, and we keep its shape: a modal that always submits the whole guide object, then invalidate-and-refetch. No optimistic update.

  1. Validate client-side first. Zod enforces title + role required, pass_threshold 50–100 (default 75), max_attempts 1–10 (default 3), responsible_roles ≥ 1, URLs valid-or-empty. One threshold range, one constant — the reference's Zod said 1–100 while its own decision log said 50–100.
  2. Submit is pessimistic. The save button goes disabled with an inline “Đang lưu…”. Nothing optimistic — a guide is not a cart line.
  3. Server stamps identity. created_by comes from the JWT and is passed into the service as a parameter, never dug out of a context inside the service. It is never accepted from the request body.
  4. The verb is honest. Edit is PUT, because the body is a full replace. The reference called it PATCH while requiring every field and overwriting every column — a lie in the method name.
  5. On success the mutation invalidates ['training','guides'] (all role variants), the list refetches, and the card appears — carrying the “Nháp” overlay if published is false.
  6. On error the modal stays open, showing the envelope's message with the form intact. The reference had no onError on any of its four mutations, so create, edit, delete and notes all failed silently.

Delete, and why it stopped being a trap. The reference gated DELETE at admin while create/edit needed only manager — and rendered the 🗑 action unconditionally. A manager clicked, got a 403, saw nothing, and the guide stayed. We lower delete to manager and drive the button from one can('training.delete', role) helper, so visibility and the server gate cannot drift apart. Delete is a soft delete: the row keeps its deleted_at and its history.

04

The roster query — the mechanism that makes tracking honest

This is the most important sequence on the page. The question the table answers is “who of my chefs has done this guide?” — and the reference asked it backwards.

✗ Reference — progress-first (INNER JOIN)

SELECT …
FROM training_progress tp
INNER JOIN staff s
  ON s.id = tp.staff_id
WHERE tp.guide_id = ?
ORDER BY s.full_name

Starts from progress rows. A staff member with no row is invisible. Combined with the fact that nothing ever wrote a progress row, the result set was always empty — forever, by construction.

✓ Ours — roster-first (LEFT JOIN)

SELECT …
FROM staff s
LEFT JOIN training_progress tp
  ON tp.staff_id = s.id
 AND tp.guide_id = ?
WHERE s.role = :guide_role
  AND s.is_active = 1
  AND s.deleted_at IS NULL
ORDER BY s.full_name

Starts from the roster. Every assigned staff member appears whether or not they have started — the un-started ones simply carry NULL progress, which renders as Chưa bắt đầu.

Why this matters beyond correctness. Assignment on this page is implicit: a guide is assigned to every active staff member whose role matches guide.role. There is no assignment UI. The LEFT JOIN is the assignment rule, executed at read time — so the table is useful and truthful on day one, before a single staff member touches anything.

The reference's INNER JOIN would have returned zero rows here. Ours returns twelve: 1 hoàn thành · 1 đang học · 10 chưa bắt đầu.

Status is computed, never stored

No table has a training status column. The four states fall out of two raw fields at render time, in one deriveTrainingStatus() feeding one <TrainingStatusBadge>:

quiz_passed === true                        → Hoàn thành      ✅  --done
quiz_passed === false                       → Chưa qua quiz   ❌  --red
quiz_passed === null && watched_percent > 0 → Đang học        🟡  --amber
quiz_passed === null && watched_percent === 0 → Chưa bắt đầu  ⬜  --muted

The fix hiding in that table. The reference rendered Chưa qua quiz (failed) and Đang học (still watching) with the same 🟡 icon, separated only by a title= tooltip. Those are precisely the two states a manager needs to tell apart — one needs coaching, the other needs time. Ours gives failure its own colour.

Pagination is server-side, 10 per page, and the quiz verdict is aggregated in the same query — the reference ran ListQuizAttempts once per row (an N+1) with no cache underneath it.

05

The staff write path — the half that never existed

Section 04 makes the table honest; this section makes it move. These two endpoints are not part of the admin UI at all — they are its truth condition. Without them every row stays “Chưa bắt đầu” forever, which is exactly what the reference shipped.

RULE 1Identity is never client-supplied

staff_id comes from the JWT. A staff member can only ever write their own progress row, no matter what the body says.

RULE 2Watch % is monotonic

POST …/watch upserts a high-water mark. Re-sending a lower percent (a replay, a second device) never walks progress backwards.

RULE 3The server owns pass/fail

The client posts a score only. passed = score >= pass_threshold is computed server-side — the FE never decides, so it can never disagree.

RULE 4Attempt + progress are atomic

Both writes ride one transaction. There is no window where an attempt exists but the progress row disagrees with it.

RULE 5Exhaustion is a 409, not a silent no-op

Past max_attempts the write is rejected with TRAINING_ATTEMPTS_EXHAUSTED in the standard envelope.

RULE 6Staff never see drafts

GET /training/my-guides is published-only. Drafts stay in the manager's authoring view.

🚨 The scheduling consequence. Because this path is what gives the tracking half any data at all, the plan blocks AD-T6 (the tracking UI) on AD-T4 (these endpoints). If AD-T4 slips, AD-T6 ships without the tracking half rather than with an inert one. Shipping a table that cannot fill is how the reference got here.

The notes write path — through every layer

A manager types a coaching note in the progress modal. Small feature, three separate reference bugs designed out.

  1. Debounce 500 ms, then flush on modal close. The reference debounced 800 ms with no flush — closing the modal quickly threw the note away. The flush is the whole fix.
  2. The handler only shapes. Bind {manager_notes}, resolve identity, hand staffId/guideId to the service as parameters. No business logic in the handler (BE_STATE.md layering).
  3. The service upserts. This is the second fix: the reference did a existence check then update, so leaving a note on someone who had not started 404'd. A note about a staff member who has not begun is exactly when coaching matters, so the row is created on demand.
  4. Success invalidates ['training','staffProgress',staffId,guideId] and the guide's progress page, so the table's “Cập nhật” column stays truthful.
  5. Failure is visible. An onError surfaces the envelope message — the third fix; the reference's notes mutation had none, so saves failed silently and invisibly.

An HTTP 404 means this URL addresses nothing. A staff member who exists and simply has not started is a perfectly valid resource with zero progress. Conflating the two is what forced the FE to render an error path as its normal state.

07

Why nothing here is cached, and what that costs

Training reads go straight to MySQL. No Redis at any layer — this is live admin data, on the do-not-cache side of the Redis policy, which reserves Redis for the catalog cache and auth rate-limiting. The reference made the same call and we keep it.

What that costs: eventual convergence between managers

09:03  Manager A  create guide  →  POST  →  MySQL row written
09:03  Manager B  (no push — nothing arrives)
09:07  Manager B  window refocus  →  refetch  →  sees the new guide
09:09  Manager A  delete (soft)   →  vanishes from A immediately
09:12  Manager B  staleTime 5 min expires  →  refetch  →  vanishes from B

This is an accepted trade, not an oversight. A restaurant has one or two managers touching training, and a guide edit is not time-critical the way an order is. Paying for an SSE channel to make two admin screens agree within seconds buys nothing. If the AD phase later opens an admin-wide realtime channel for the live floor, training is a candidate consumer of it, never a reason to build it.
