Admin combos page — FE + BE build plan

The combo ("suất") builder at /admin/combos — where products are composed into the bundles the customer menu sells. Planned on top of the products contract: it cross-links admin_products rather than re-deriving it. Snapshot of the plan; the .md wins.

statusplan · F-28
source of truthadmin_combos_PLAN.md
depends onF-27 products contract
ships acrossC-1 · AD-C1…C5

A combo is built from products — the picker, every chip name, the retail sum and the savings figure all come from GET /products/all. That one dependency is this page's defining risk, and the reference handled it badly: a deleted product turned into a raw UUID chip and a wrong Giá lẻ, with no warning anywhere.

Two sources in the corpus. The traced 08_pages set is the running code (audit found zero doc-vs-code contradictions); the wireframes draft was never built. Where they disagree the traced set wins — §07 lists the draft's inventions.

03

State ownership + the product dependency

DataKindOwner
combo listserverQuery ['admin','combos'], staleTime 30 s
product list (picker + join)serverQuery ['admin','products'] — the same key F-27's page uses
picker search / list searchURL?pq= / ?q= — FE rule 2
scalar form fieldsformRHF + Zod
selected items (product_id → qty)forma Record<string,number> map — not useFieldArray

Why a map, not useFieldArray: toggling/de-duping is O(1) and duplicate product_ids become structurally impossible. The reference made this call and it's correct; we keep it and validate the map at submit against the shared Zod schema.

What happens when a referenced product changes (the reference's biggest hole)

EventOur behavior
renamedchips follow (server joins the name at read)
price changedcombo price untouched; retail_total moves → warn when price ≥ retail_total; product write DELs combos:list
soft-deletedserver marks the item product_deleted:true, excludes it from retail_total; a warning chip names it, not a raw UUID; the modal shows it for explicit removal
hard-deletedblocked by combo_items.product_id RESTRICT → surfaced as 409 PRODUCT_IN_USE with a combo count

BE contract

#RouteAuthTaskBehavior
1GET /combos/allmanager+AD-C1New. Management read — all combos incl. is_active=0, uncached, items server-joined.
2GET /combospublicC-3Customer read — active only, cached; ids-only (owned by customer_menu plan).
3GET /products/allmanager+AD-P1Picker + join. Owned by F-27.
4POST /combosmanager+AD-C2Create; header + items in one transaction; full object 201.
5PATCH /combos/:idmanager+AD-C2Update; items replace-all in tx; omitted scalars preserved.
6PATCH /combos/:id/activemanager+AD-C3New. Hide/show — the ref had no way to take a combo off the menu.
7DELETE /combos/:idadmin+AD-C3Soft delete; existence check (404) + 409 COMBO_IN_USE.
8POST /files/uploadmanager+AD-P4Combo image — same endpoint as products (owned by F-27).

One shared validator for create + update: name required, price ≥ 1, items min 2. The reference let create/update drift, accepting a price-0 zero-item combo it then refused to edit.

06

Behaviors — the AC

B1Manager builds; only admin deletes

Xoá hidden/disabled below admin.

B2A combo needs ≥ 2 items

Client and server, on both create and update (ref: update only).

B3Picker de-dups by id, not name

Searchable (?pq=); ref de-duped by name so same-named products were unpickable.

B4Unavailable products marked

Still selectable, with a warning; ref offered them indistinguishably.

B5Price manual; system advises

Tổng giá lẻ + 90% suggestion + green ✓ Tiết kiệm when > 0; warn when price ≥ retail.

B6Active toggle

Hide a combo from the menu in one click — a capability the ref lacked entirely.

B7Delete guarded twice

Typed dialog + 409 COMBO_IN_USE; unknown id → 404, not a cheerful 204.

B8Edit: replace-all items, preserve scalars

Editing the name must not erase image/category (ref's bug 2).

B9Chips tell the truth

Tên ×qty from the join; a deleted product renders a warning chip, never a raw UUID.

B10Prices via formatVND()

Normalised to 30.000 đ; corpus was inconsistent.

B11VN-first copy

"+ Thêm combo", "(N món đã chọn)", "Tổng giá lẻ", "Giá combo *", "Lưu combo", "Tiết kiệm".

09

Decisions & flags

✅ Management read carries a server-side join while the customer read stays ids-only — the admin table must stay correct even when a product is gone.

✅ Combo price stays manual with a 90%-of-retail suggestion.

✅ Items map, not useFieldArray — duplicates impossible.

✅ Delete = admin+, everything else manager+.

✅ RESOLVED (owner) — is_active; and worked example = Suất Đầy Đủ at 30.000 (= retail; savings shows — on seed, column kept for discounted combos).

⚠ Follow-up (F-15-owned): the menu plan's illustrative combo (55000) should be corrected to 30000 so the one worked example stays consistent across every doc.

💡 no per-id combo endpoint exists (/menu/combo/:id pulls the whole list) — belongs to the C-5 detail task. ❓ combo images: contract supports them, owner to confirm intent.

Verify plan

BE: role matrix; create/update in a tx with a forced item failure proving rollback; PATCH-name-only preserving image_path; the validator rejecting items<2 on both paths; active toggle + guarded delete + PRODUCT_IN_USE; DEL proof for combos:list on product writes. FE: a screenshot per branch (table + picker), the deleted-product warning chip, the price ≥ retail warning, a ≥2-item round-trip.
