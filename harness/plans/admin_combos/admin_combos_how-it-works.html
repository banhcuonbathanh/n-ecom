03

The create path — one transaction

Tick ≥ 2 products, set quantities, type the price, save. Header + every item write in one transaction — a bad item rolls the whole thing back.

request/stepcommitrollback

The reference looped item inserts, slog.Warn-ing and continuing on failure — so a combo could return 201 with items missing, and a failed edit (items deleted before re-insert) left the combo with fewer items than it started with. The transaction ends both.

04

The deleted-product edge case — the reference's worst hole

A product referenced by a combo is soft-deleted elsewhere. combo_items survives (no cascade on soft delete). What the two systems then show:

ReferenceOurs
chipraw UUID p4f9…36warning chip "⚠ Sản phẩm đã xoá"
Giá lẻunder-counted (item = 0)server excludes it from retail_total, marks product_deleted
in the modalinvisible → silently re-savedshown explicitly for removal

The fix is server-side: GET /combos/all joins product_name + unit_price and computes product_deleted — the one place that can be correct even when the product row is gone. The FE just renders the truth it's handed.

06

Cache life & combo → order expansion

Cache-aside; a write DELs combos:list. The new cross-rule: a product write DELs combos:list too, because a combo shows that product's name/price.

product price change ──► DEL products:list + combos:list ──► /menu rebuilds both
combo write          ──► DEL combos:list                 ──► /menu ComboSection rebuilds
staleness bound: admin instant (uncached read) · customer ≤ staleTime + 5-min TTL

Why edits are safe. At order time a combo expands into order_items: one header row (combo_id, unit_price = 0) + N child rows via combo_ref_id, each carrying a price the server snapshots from the catalogue — never trusted from the client. A later combo edit or delete never rewrites a placed order. That snapshot is why this whole page can be cache-based instead of realtime.
