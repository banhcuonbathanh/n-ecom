05

The status model — this plan's central correction

What the reference does. staff_tasks.status is written once as pending and never changes. There is no UPDATE query anywhere in the codebase, no route that advances status, and no job that flips overdue rows. Yet the stats queries, the KPI row, the completion percentages and the overdue highlight are all built on the three values that are unreachable.

Consequence on a live database: "Hoàn thành" = 0, "Đang thực hiện" = 0, "Quá hạn" = 0, every completion rate 0 %, the overdue badge never appears — permanently. The board cannot show progress.

                 ┌──────────────► cancelled ◄──────────┐
                 │                                      │
   (create) → pending ──────► in_progress ──────► completed
                 │                                  │
                 └──────────────────────────────────┘   direct complete allowed

   STORED    pending · in_progress · completed · cancelled        4 values
   DERIVED   overdue = status IN ('pending','in_progress')
                       AND due_at < NOW()
                       ↑ computed in the read query
                       no job · no write · cannot drift · correct the instant a deadline passes
From → ToNote
pending → in_progress / completed / cancelledallowdirect complete is fine for short tasks
in_progress → completed / cancelled / pendingallowback to pending = "un-start"
completed → in_progress / pendingallowcorrecting a mis-click; clears completed_at
completed → cancelled409cancelling what is already done is meaningless
cancelled → anything409terminal; assign a new task instead
any → itself200 no-opidempotent, so a double-click is harmless

Transition validation lives in the service layer — handlers shape, services decide (BE_STATE.md).

09

Decisions, flags & verify plan

Decided

  • overdue is derived, never stored — 4-value stored enum.
  • qualityScore does not ship. A real metric needs a real captured signal, not a rescaled percentage.
  • No Redis for the task domain — ARCHITECTURE.md §4 applied.
  • v1 ships create + status-change; edit and delete deferred rather than half-shipped.
  • Neutral F-7 tokens — admin surface, not the customer dark/orange shell.
  • This plan owns the /admin/tasks contract; the sibling admin_todo_list plan links to it.

⚠️ FLAG — task-id collision. Two TASKS.md rows both claim F-20, and two both claim F-24 — parallel sessions sharing one working tree, the same failure as the three-way F-10 collision in STATE.md Session 7. This plan took F-22 to stay clear; someone must renumber the duplicates.

💡 SUGGESTION — the assignee has no surface. Both pages reading staff_tasks are manager+ gated, so a manager can assign work the staff member cannot see in the system. A product gap, not a bug — out of scope here, but it wants a ruling before the AD phase closes. Cheapest fix: GET /me/tasks plus a panel on the staff surface.

❓ CLARIFY — who marks a task done? Behavior 08 lets a manager move a status. If staff should mark their own work complete, endpoint 4's auth widens to "manager+ or the assignee" — and that needs the surface above. Defaulting to manager+ only until the owner rules.
