The big picture

┌──── MANAGER'S BROWSER ─────────────────────────────────────────────┐
│                                                                     │
│  page.tsx  (coordinator: filters · expandedStaffId · modalOpen)      │
│     │                                                               │
│     ├─ Query ['admin','tasks','stats', date]   30s stale · 60s poll  │

│     └─ Query ['admin','staff']                5min · modal only     │
│                                                                     │
└─────────────────────────────┬───────────────────────────────────────┘
                              │  HTTPS · cookie JWT · manager+ gate
  ════════════════════════════╪═══════════════ THE WIRE ══════════════
                              │
┌──── GIN (BE) ───────────────▼───────────────────────────────────────┐
│  1 GET  /admin/tasks/stats?date=      day rollup + per-staff rows  │
│  2 GET  /admin/tasks?staffId=&date=   one member's tasks           │
│  3 POST /admin/tasks                  create                       │
│  4 PATCH /admin/tasks/:id/status      advance status               │
│  5 GET  /staff?limit=100              dropdown (staff domain)       │
└─────────────────────────────┬───────────────────────────────────────┘
                              │  sqlc
                    ┌─────────▼──────────┐
                    │  MySQL staff_tasks │  the ONLY durable artifact
                    └────────────────────┘

  NO Redis · NO SSE · NO WebSocket · NO localStorage
  Every browser polls independently. Nothing is pushed. Max staleness ≈ 60 s.

Everything above the wire is throwaway. Filters, expanded row, the Query cache, the toast — all in-memory, all lost on F5. The one fact that survives is the staff_tasks row. Every page that wants it asks the database again.

This is deliberate, not a gap: ARCHITECTURE.md §4 scopes Redis to the catalog cache and auth rate-limiting, and a manager-only board with a 60-second poll has no realtime requirement worth a channel.

  the lazy query's  enabled: !!expandedStaffId  flips false → true
        │
        ├─ key ['admin','tasks', 'cash02', '2026-07-19'] fresh (<15 s) ─▶ cache hit, renders instantly
        │
        └─ stale or absent ──▶ GET /admin/tasks?staffId=cash02&date=…
                                    │
                                    ├─ in flight ─▶ skeleton rows
                                    ├─ error ─────▶ inline message + [Thử lại]  ← retry refetches in place
                                    ├─ empty ─────▶ "Không có công việc nào trong ngày này"
                                    └─ ok ────────▶ task rows: name · description · priority · giờ · [status ▾] · notes

Two rules that fall out of the single-value state

  • One open at a time. expandedStaffId is a single string | null, so expanding a second row collapses the first automatically — no bookkeeping needed.
  • Collapse does not evict. The old key stays in the cache with its 15 s staleTime, so re-expanding the same row within 15 s is instant and silent.

Reference gap closed. Its expanded-row error is a dead-end string with no retry — the only recovery is to collapse the row and expand it again. Ours retries in place.

The 422 pre-check is the fix for a real reference bug. The reference has no pre-check: it lets the insert run, the fk_tasks_assigned_to constraint rejects it, the repo returns a driver error, and the service maps every non-ErrNoRows error to ErrInternalError. So assigning a task to a staff id that does not exist returns 500 and a generic "thử lại" toast — caller error reported as server error, with no field-level message. (The sql.ErrNoRows branch sitting above it is dead code: an INSERT never returns it.)

The status transition — PATCH /admin/tasks/:id/status

This endpoint does not exist in the reference at all. It is the single most important addition in this plan, because without it the entire stats half of the page is dead.

                 ┌──────────────► cancelled ◄──────────┐
                 │                                      │
   (create) → pending ──────► in_progress ──────► completed
                 │                                  │
                 └──────────────────────────────────┘   direct complete allowed
  manager picks "hoàn thành" in the expanded row
        │  control disables (pessimistic — no optimistic update; that is cart-only)
        ▼
  PATCH /admin/tasks/3a7e…/status   { "status": "completed" }
        │
        ▼
  SERVICE: look up current status
        │
        ├─ id unknown / soft-deleted ─────────────▶ 404 NOT_FOUND
        ├─ same status ───────────────────────────▶ 200 no-op  idempotent — double-click is harmless
        ├─ completed → cancelled ─────────────────▶ 409 CONFLICT
        ├─ cancelled → anything ──────────────────▶ 409 CONFLICT  terminal
        └─ legal ─────────────────────────────────▶ UPDATE status
                                                     entering completed → stamp completed_at
                                                     leaving  completed → null completed_at
                                                          │
                                                          ▼
                                                    200 { data: Task }
                                                          │
                              invalidate both keys ───────┘
                                    │
                                    ▼
                    KPI row moves: "Đang thực hiện" −1, "Hoàn thành" +1
                    staff row completionRate recomputed on the refetch

  on 409 ──▶ toast "Không thể chuyển trạng thái này" ──▶ control reverts to its previous value

Why the transition rules live in the service. A handler's job is shape; a service's job is whether the change is legal (BE_STATE.md). Putting the machine in the service means the todo-list page — which writes the same table through the same service — inherits the identical rules for free, with no second implementation to drift.

06

How "quá hạn" is computed — the key insight

The reference stores overdue as a status value. That means something must write it: a background sweeper that periodically flips pending rows whose due_at has passed. No such job was ever written. So the stored value is never set, the "Quá hạn" KPI reads SUM(CASE WHEN status='overdue') = 0, and the overdue badge never appears — forever.

We do not store it. overdue is not a status; it is a question you can ask of a row at any moment:

  isOverdue  =  status IN ('pending','in_progress')  AND  due_at < NOW()
                ↑ evaluated inside the read query, every read

A task therefore becomes overdue the instant its deadline passes — with no write, no job, and no window in which the database is wrong:

  task "Kiểm kê nguyên liệu"   due_at = 21:00   status = pending   (never written again)

  wall clock   20:58        20:59        21:00        21:01        21:02
               │            │            │            │            │
  our read     not overdue  not overdue  OVERDUE      OVERDUE      OVERDUE
               ↑ the row is byte-for-byte identical at every tick — only NOW() moved

  reference    not overdue  not overdue  not overdue  not overdue  not overdue
               ↑ waiting for a sweeper job that does not exist

  KPI "Quá hạn" updates on the next stats poll (≤60 s) with zero writes in between.

Derived beats stored whenever the value is a pure function of data you already have. Stored state needs a writer, and a writer can be forgotten, crash, lag, or run twice. due_at < NOW() has none of those failure modes. This is the same instinct as FE_STATE.md's "derived, not stored" rule, applied on the server.

The cost is honest and small: overdueCount and hasOverdue are computed per read rather than looked up. Both are indexed range comparisons on due_at over one day of rows for one restaurant — nothing that needs a cache.

07

Cache & invalidation life

  ['admin','tasks','stats', date]    stale 30 s   poll every 60 s   KPIs + staff table
  ['admin','tasks', staffId, date]   stale 15 s   lazy, no poll     expanded row
  ['admin','staff']                  stale 5 min  on modal open     dropdown

  a write (POST or PATCH) fans out:
        ├──▶ stats key @ task.dueDate      the day the task belongs to
        ├──▶ stats key @ filters.date      the day on screen  ← the fix
        └──▶ tasks key @ (task.staffId, date)

The stale-KPI bug we designed out. The reference invalidates using task.dueDate only. Watch what that does when a manager is looking at today and assigns work for tomorrow:

  REFERENCE
  viewing filters.date = 19/06          KPI "Tổng việc" shows 18
  create task for       20/06   ──▶  invalidate stats@20/06
                                          stats@19/06 never invalidated
                                          KPIs on screen still say 18 — for up to 60 s
                                          manager sees no feedback that the task landed

  OURS
  viewing filters.date = 19/06          KPI "Tổng việc" shows 18
  create task for       20/06   ──▶  invalidate stats@20/06  (the task's day)
                                     ──▶  invalidate stats@19/06  (the day on screen)
                                          visible KPIs refetch immediately
                                          counts for 19/06 are confirmed correct, not assumed

Invalidating the on-screen date even when nothing about it changed is deliberate: one cheap refetch buys the guarantee that what the manager is looking at is true after a write. Assuming it is unchanged is exactly the assumption the reference got wrong.
