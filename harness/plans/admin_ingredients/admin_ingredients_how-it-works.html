01

The big picture

Two roles reach this page, three doors write to the database, and exactly one of those doors is allowed to move stock.

        manager ─┐
                 ├──▶ authMW ──▶ AtLeast("manager") ──▶ /admin/ingredients
        admin ───┘                       │
                                         │  DELETE only:
                                         └──▶ AtLeast("admin") ──▶ 403 for manager

   ┌──────────────────── three write doors ────────────────────┐
   │                                                            │
   │  ① POST /admin/ingredients      create + opening movement  │──┐
   │  ② PATCH/DELETE …/:id           metadata + soft delete     │  │ tx
   │  ③ POST /admin/stock-movements  the ONLY stock door        │──┘
   │                                                            │
   └────────────────────────────┬───────────────────────────────┘
                                ▼
              ┌──────────────────────────────────┐
              │ ingredients.current_stock        │ ◀── running total
              │ stock_movements  (append-only)   │ ◀── ledger of record
              └──────────────────────────────────┘
                    they must never disagree
The whole page exists to keep one invariant true: Σ movements == current_stock. Every design choice below — the transaction, the rejected PATCH key, the opening movement at create time — follows from it. The reference broke this invariant in three separate places.
02

First load — and the five branches

GET /admin/ingredients→ page.tsx (RSC)→ prefetch ['admin','ingredients']→ dehydrate→ HydrationBoundary→ IngredientsClient

Because the query is prefetched on the server and hydrated into the client cache, the first paint is data, not a spinner. The loading branch below is what a cold client refetch shows, not what the operator sees on navigation.

Defect D-6 — the conflation. The reference collapsed loading, empty, and no-match into one state, so an operator searching for a typo saw the same screen as an operator whose inventory was genuinely empty — and both looked like the page was still loading. These are three different situations needing three different next actions: wait, add your first ingredient, clear the search. Hence five named branches, each with its own copy and its own CTA.

Note the split. empty is decided on the server rows; no-match is decided on the filtered rows. Search never touches the network, so a search miss can never be mistaken for an empty inventory.

03

The core action — posting a stock movement

The operator opens StockMoveModal on Bột gạo (25.500 kg), picks Xuất (−), types 2.500, notes "Hỏng do ẩm", and confirms.

The rejection path — an over-draw

Same modal, but the operator types 30.000 against a stock of 25.500.

Stock is unchanged, the modal stays open, the input is preserved. Because mutations are pessimistic, nothing on screen ever showed the movement as having happened. The reference instead clamped silently — GREATEST(0, stock - qty) — which recorded a 30 kg withdrawal against 25.5 kg of stock, landed on zero, and told the operator it worked. The 4.5 kg discrepancy then lived in the data forever.
⚠ F-2. This reject is DB_SCHEMA.md §6 ruling 8's default and is flippable while AD is open. If real kitchen practice is "count later, log now", it becomes a clamp plus a visible warning — never a silent one.
04

Inside the transaction

This is the section the rest of the plan exists to justify.

✗ The reference — two statements, no tx

INSERT INTO stock_movements (…)
VALUES (…);
-- ← crash / timeout / rollback here
UPDATE ingredients
SET current_stock =
      GREATEST(0, current_stock - ?)
WHERE id = ?;

Two independent round-trips. If the process dies between them, the ledger records a withdrawal that never left the count. Under concurrent writes, two reads of the same starting value both write back, and one update is lost. their flag #2

✓ Ours — one tx, row locked

BEGIN;

SELECT current_stock
FROM   ingredients
WHERE  id = ? AND deleted_at IS NULL
FOR UPDATE;          -- ← lock

-- service checks: stock ≥ qty
INSERT INTO stock_movements (…)
VALUES (…);

UPDATE ingredients
SET    current_stock = ?,   -- computed
       updated_at = NOW()
WHERE  id = ?;

COMMIT;

Ledger and count commit together or not at all. The lock serializes concurrent writers on the same row.

The create path has the same shape

The reference wrote initial_stock straight into current_stock with no movement row at all — so an ingredient created with 5 kg had a ledger sum of 0 and a count of 5 from the moment it existed.

POST /admin/ingredients { name: "Mộc nhĩ", initial_stock: "5.000", … }

  BEGIN
    INSERT INTO ingredients (…, current_stock = 0)          ← always starts at zero
    if initial_stock > 0:
        INSERT INTO stock_movements (type='in', qty=5.000)  ← the opening movement
        UPDATE ingredients SET current_stock = 5.000
  COMMIT

  Σ movements (5.000) == current_stock (5.000)   ✓ true from birth
Why it matters beyond tidiness. /admin/storage is planned to report total_imported as SUM(quantity) WHERE type='in'. Under the reference's create path that figure silently excludes every ingredient's opening balance — the report would be wrong and nothing would flag it. Recording the opening movement makes the future report correct by construction rather than by patch.

And the third door is sealed: current_stock is not an accepted PATCH key. Sending it returns VALIDATION_FAILED rather than being silently ignored, so a client that tries learns immediately (defect D-2).

05

Concurrency — two operators, one ingredient

The kitchen manager and the storeroom lead both withdraw from Bột gạo (25.500 kg) at the same moment: 2.500 kg and 1.000 kg.

✗ Without the lock (the reference)

A reads 25.500 ┐
B reads 25.500 ┘  both see the same
A writes 23.000
B writes 24.500   ← overwrites A

ledger:  out 2.500 + out 1.000 = 3.500
count:   25.500 − 1.000 = 24.500
truth:   22.000

2.500 kg vanished from the count
but survives in the ledger.

✓ With the lock

A: 25.500 → 23.000
B: 23.000 → 22.000  (re-read!)

ledger:  out 2.500 + out 1.000 = 3.500
count:   25.500 − 3.500 = 22.000
truth:   22.000

Σ movements == current_stock ✓
This is the AD-INV-4 receipt. Two simultaneous out calls, then assert Σ movements == current_stock. It is the only receipt that proves the transaction is real rather than decorative, so it is not optional.
06

Status derivation at read time

Neither expiry_date nor status is a column — both are computed on the way out, per row (DB_SCHEMA.md §1: derived facts get no column).

row from MySQL→ expiry_date = import_date + shelf_days→ status = ladder(stock, min, expiry)→ JSON→ badge 
for each row, first match wins:

  ① current_stock == 0          ──▶ out_of_stock
  ② expiry_date < today + 7d    ──▶ expiring_soon
  ③ current_stock <= min_stock  ──▶ low_stock
  ④ otherwise                   ──▶ in_stock

Two seed rows exist specifically to prove this is a ladder, not a set (today = 2026-07-19):

The FE renders this value and never recomputes it (FE_STATE.md §9 rule 11). The reference had the BE send a status and the FE re-derive one — two implementations of the same ladder, guaranteed to drift the first time either threshold changed. One source, one ladder.
07

Why there is no cache here

Every other read-heavy surface in this project sits behind Redis. This one deliberately does not.

Catalog (customer menu) — cache-aside

GET /products
  │
  ├─ Redis HIT ──▶ return (fast path)
  │
  └─ MISS ──▶ MySQL ──▶ SETEX ──▶ return

writes DEL the keys; a stale
menu for a few seconds is fine.

Inventory — no server cache at all

GET /admin/ingredients
  │
  └─ MySQL ──▶ return

No Redis client is imported by
the ingredient service. A stale
stock figure is a miscount, and
a miscount is a wrong decision.

OVERALL_PLAN.md lists inventory under "never cached", and ARCHITECTURE.md §4 scopes Redis to catalog + auth rate-limit only. The rule is enforced structurally: only platform/cache may import the Redis client, and the ingredient service does not.

The only cache is on the client

mutation 201→ invalidate ['admin','ingredients']→ refetch→ fresh stock + recolored badges
WriteInvalidates
POST / PATCH / DELETE /admin/ingredients['admin','ingredients']
POST /admin/stock-movements['admin','ingredients'] + ['admin','ingredients',id,'movements']

The staleTime: 60s bound applies only to background refocus refetches — after any write the list is invalidated immediately, so the operator never reads their own stale number. That closes the loop opened in §03 step 11.
