Why search is local, not a URL param. The list is one unpaged fetch of a small table, and the server never sees the filter. Putting it in the URL would create a ghost parameter that looks like server state but isn't — the same call the menu plan made.

04

BE contract

The eight endpoints

All under /api/v1/admin, all behind authMW + AtLeast("manager"). One exception, and it matters:

#RouteAuthTaskBehavior
1GET /admin/ingredientsmanager+AD-INV-2All non-deleted, name ASC; each row carries derived expiry_date + status
2GET …/low-stockmanager+AD-INV-2current_stock <= min_stock * 1.2, most-critical first. Consumed by /admin/summary, not this page
3GET …/:idmanager+AD-INV-2Single row or 404
4POST /admin/ingredientsmanager+AD-INV-3Creates the row and its opening 'in' movement — one tx
5PATCH …/:idmanager+AD-INV-3Metadata only — cannot touch current_stock
6DELETE …/:idadmin onlyAD-INV-3Soft delete; 422 when a recipe references it
7POST /admin/stock-movementsmanager+AD-INV-4Appends a movement and updates stock in one locked tx
8GET …/:id/movementsmanager+AD-INV-5Ledger tail, newest first, LIMIT 50

Schema depended on

Full column specs live in DB_SCHEMA.md §4.6 — not repeated here.

ingredients

  • id CHAR(36) PK
  • name VARCHAR(150)
  • unit VARCHAR(30)
  • import_date DATE
  • shelf_days INT
  • current_stock DEC(10,3) ⚠
  • min_stock DEC(10,3)
  • cost_per_unit DEC(10,0)
  • + timestamps, deleted_at

stock_movements (ledger)

  • id CHAR(36) PK
  • ingredient_id FK
  • type ENUM(in,out,adjustment)
  • quantity DEC(10,3) > 0
  • note TEXT NULL
  • created_by FK→staff
  • created_at
  • no UPDATE · no DELETE

product_ingredients

  • product_id FK
  • ingredient_id FK
  • qty_used DEC(10,3)
  • read-only here —
  • only the delete guard
  • consults it
⚠ current_stock is derived via movements and never written directly. Only endpoints 4 and 7 move it, and only inside a transaction. Everything in this plan follows from that one rule.

Status derivation — the precedence ladder

Computed by the BE on the way out. First match wins:

  current_stock == 0 ?            ──yes──▶  out_of_stock     ← Hành lá (0.000)
        │ no
  expiry_date < today + 7d ?      ──yes──▶  expiring_soon    ← Thịt heo xay (2026-07-21)
        │ no
  current_stock <= min_stock ?    ──yes──▶  low_stock        ← Nước mắm (4.000 ≤ 5.000)
        │ no
        ▼
     in_stock                                                 ← Bột gạo, Mộc nhĩ

Two seed rows exist to prove the ladder: Thịt heo xay is both low (8 ≤ 10) and expiring — it reports expiring_soon. Hành lá is both empty and expiring — it reports out_of_stock. Precedence, not a set.

Wire shapes

Success is unwrapped; only errors ride the envelope. Field spellings are frozen by curl receipts at AD-INV-2/3/4 (gate 8).

GET /admin/ingredients → 200 — decimals are strings, money is a bare VND integer:

[ { "id": "ing1…36", "name": "Bột gạo", "unit": "kg",
    "import_date": "2026-07-12", "shelf_days": 90,
    "expiry_date": "2026-10-10",
    "current_stock": "25.500", "min_stock": "10.000",
    "cost_per_unit": 18000, "status": "in_stock",
    "created_at": "2026-07-12T08:00:00Z", "updated_at": "2026-07-19T09:14:02Z" } ]

POST /admin/stock-movements → 201 — quantity always positive, direction lives in type:

→ { "ingredient_id": "ing1…36", "type": "out",
    "quantity": "2.500", "note": "Hỏng do ẩm" }

← { "id": "mv1…36", "ingredient_id": "ing1…36", "type": "out",
    "quantity": "2.500", "note": "Hỏng do ẩm",
    "created_by": "stf1…36", "created_at": "2026-07-19T09:14:02Z",
    "resulting_stock": "23.000" }

resulting_stock is the value the transaction actually committed — it lets the FE confirm the new figure without a refetch race.

Dataflow

Read path

page.tsx (RSC)→ prefetch ['admin','ingredients']→ HydrationBoundary→ IngredientsClient→ client filter (search)→ IngredientTable

First paint is data, not a spinner — the list is prefetched server-side and hydrated.

Write path — a stock movement

StockMoveModal→ POST /admin/stock-movements→ Gin handler→ service (validate type + over-draw)→ tx: lock row → INSERT ledger → UPDATE stock → COMMIT→ 201 + resulting_stock→ invalidate

Cache map

Nothing here is cached server-side. OVERALL_PLAN.md lists inventory under "never cached", and ARCHITECTURE.md §4 scopes Redis to catalog + auth rate-limit only. No platform/cache import belongs in the ingredient service. FE invalidation is the entire cache story:

Mutations are pessimistic — no optimistic stock UI. Stock is money-adjacent and the server is the arbiter; a rejected over-draw must never flash as success.

Behaviors — the acceptance criteria

01Page loads with the list already rendered from RSC hydration.
02Rows sort name ASC; each shows name, unit, stock, min, expiry, status badge.
03Search filters client-side by name, diacritic-insensitive.
04A search matching nothing shows no-match — distinct from empty.
05[+ Thêm nguyên liệu] opens the form modal in add mode.
06Add posts POST; on 201 modal closes, list invalidates, toast "Đã thêm nguyên liệu".
07Duplicate name → 409 inline "Nguyên liệu này đã tồn tại"; modal stays open, input preserved.
08Row ✎ opens the same modal in edit mode, pre-filled; submits PATCH.
09The edit form exposes no stock field.
10Row 🗑 renders only for admin; managers never see it.
11Delete confirms, then DELETE; on 204 list invalidates + toast.
12Deleting an in-use ingredient → 422 "…đang được sử dụng trong công thức"; row stays.
13Row [Nhập/Xuất] opens StockMoveModal showing current stock as context.
14Modal takes type, a positive quantity in the ingredient's unit, and an optional note.
15Quantity ≤ 0 is blocked client-side by Zod before any request.
16An out > stock is server-rejected → "Không đủ tồn kho (còn 23.000 kg)"; stock unchanged.
17On 201 modal closes, list invalidates, row shows resulting_stock; toast names the delta.
18A movement never edits or deletes an earlier one — corrections are new adjustment rows.
19Status badges recolor after any movement, from the refetched server value.
20All six actions are keyboard reachable; modals trap focus and close on Esc.
07

Reference defects designed out

The big one — D-1. The reference inserted the movement row and then issued a separate UPDATE on current_stock, with no transaction. It also wrote initial_stock straight into current_stock at create time with no movement row at all. The first lets ledger and count drift under crash or concurrency; the second guarantees they disagree from day one.
#Reference findingCountermeasure
D-1Non-transactional stock update (their flag #2); create wrote stock with no movement (their flag #1)Both paths run one tx with the row locked; create inserts the opening in movement. Σ movements == current_stock becomes a testable invariant
D-2current_stock reachable as a plain update fieldNot an accepted PATCH key — rejected with VALIDATION_FAILED. Stock has exactly one door
D-3🗑 shown to managers; BE 403s themButton gated on role === 'admin' (FE_STATE §9 rule 13)
D-4Delete had no in-use guard; the FK would surface as MySQL 1451 → 500Service checks product_ingredients first → clean 422. Load-bearing: the FK is CASCADE, so without the guard a delete silently destroys recipe rows
D-5created_by scanned but dropped from the JSON (their flag #3)Serialized, plus created_by_name joined for display
D-6loading / empty / search-miss rendered through one conflated stateFive named branches, each with its own copy and CTA
D-7409 toast for an error the BE could never emit (no UNIQUE on name)Either the UNIQUE amendment lands and 409 becomes real, or the toast is cut. No UI for imaginary errors
D-8StockMoveModal inlined in page.tsxOwn file with its own Zod schema — the over-draw rule needs a testable home
D-9cost_per_unit stored but silently withheld (their flag #4)Deliberately exposed at manager+ (ruling 7) — never accidentally invisible

⚠️ Flags for the boss

F-1 — schema amendment requested. UNIQUE(name) on ingredients, with the soft-delete rename rule from DB_SCHEMA.md §4.4. DB_SCHEMA.md owns tables, so this plan requests rather than declares it. Until ratified, behavior 07 and the 409 path do not exist and the duplicate-name toast must be cut.
F-2 — over-draw default is flippable. Ruling 8 rejects a below-zero out and says "lock in when AD opens". This plan builds the reject. If real kitchen practice is "count later, log now", say so before AD-INV-4 and it becomes a clamp + warning instead.
F-3 — admin palette unresolved. The live admin shell used orange/dark while our tokens are neutral. This set's mockup uses the neutral F-7 tokens per PAGE_PLAN_GUIDE §7. One answer should cover every admin plan.
❓ C-1 — batch vs ingredient dating. One import_date per ingredient row means re-importing silently moves expiry_date for stock that is physically older. Real batch tracking needs per-movement dating. Out of scope here; worth a ruling before /admin/storage designs expiry reporting.
