02

First load — prefetch then five branches

RSC page.tsx prefetches products + categories + toppings → HydrationBoundary → the client hydrates the same cache (zero-spinner first paint on a warm server; loading.tsx streams a table-shaped skeleton while the RSC awaits). Then the table renders exactly one branch:

The reference had a bare <p>Đang tải...</p> and conflated empty with filtered-empty; its modal sub-queries degraded silently so "loading" and "none exist" looked identical. All five branches here are named and built.

03

The create path — two decoupled steps

Upload first (own progress + errors), then submit the form. A product can be saved with no image; a failed upload never blocks the form.

requestresponse

Header + topping junction + orphan flip + cache DEL all ride one transaction; the response is the full object, so the table doesn't refetch just to learn what it wrote.

04

The availability toggle — broken vs fixed

The reference's headline bug. The route pointed at the full-update handler, which required name/price/category_id and never wrote the column — so a {is_available}-only body was a permanent 400, and a sold-out dish kept showing to customers.

requestreference: 400ours: 200

Dedicated route → dedicated handler binding only {is_available} → the existing ToggleProductAvailability query → invalidate. The badge is pessimistic: it shows pending, then the server's truth, and the dish drops out of products:list.

The delete path — guarded

> 0409 PRODUCT_IN_USE

= 0soft-delete · 204

Why historical orders are safe. order_items snapshot name/unit_price at order-create time. A soft-delete only affects the live menu — never a placed bill. That durability boundary is why this whole page can be cache-invalidation-based instead of realtime.

The reference soft-deleted unconditionally and its FE 409 branch was dead code. Ours emits the 409, so the branch that shows "Sản phẩm đang có đơn hàng đang xử lý, không thể xoá" is reachable.

06

Cache life

Cache-aside in services only. A read fills a key with 5-min TTL; a write DELs it outright (never rewrites), so the next downstream read rebuilds from MySQL.

WriteDELWho rebuilds
product create/update/delete/toggleproducts:list · categories:list · product:<id> · combos:list/menu, /pos, /admin/combos on next fetch
topping writetoppings:list · products:list + joined product:<id>/menu (products embed toppings)

Staleness bound: admin table = instant (uncached read + query invalidate); customer menu ≤ its staleTime + 5-min TTL. Redis failures fail open — a Del error is logged, not fatal; a get miss falls through to MySQL.
