Admin products page — FE + BE build plan

The manager's catalog console at /admin/products — the write side of the catalog whose read side the customer menu consumes. A table, a form modal, and a cache-invalidation fan-out. Snapshot of the plan; the .md wins on any conflict.

One page, three moving parts: a manager-gated table that shows every product (hidden ones included), a create/edit form modal with image upload + topping assignment, and a write path that DELs Redis keys so /menu, /pos and the combo builder rebuild on their next fetch.

This plan's whole job is to design out the reference's headline defect — a product availability toggle that was a permanent 400 — and its twelve siblings.

modal open / edit targetlocaluseState in page.tsx

No Zustand on this page. Nothing here is client state that outlives a route — FE rule 1's decision flow lands on Query / URL / RHF for every row. State crosses components only through the Query cache (the modal's dropdowns read the same keys the customer menu uses) and the form.

State crosses pages purely server-side — MySQL row + Redis eviction. No SSE, no localStorage. A manager's own writes are instant (uncached read + query invalidate); a customer's menu can lag a price change by ≤ 5 min — safe, because orders snapshot name/unit_price at order time, so a catalog edit never rewrites a placed order.

06

Behaviors — the AC

B1Role gating is visible

Manager+ page; 🗑 disabled-with-reason for a manager, not a 403. Below manager: "Không có quyền truy cập trang này".

B2Table shows everything

Incl. is_available=0 rows, dimmed — the one surface where hidden products are visible.

B3Availability toggles in one click

The headline fix. Pessimistic — badge shows pending, then the server's truth.

B4Create is two-step

Upload image (own progress/errors) then submit. A failed upload never blocks the form; a product can save with no image.

B5Edit pre-fills all fields

Incl. toppings. Omit topping_ids = leave the join; send [] = clear it.

B6Delete is guarded twice

Typed dialog naming the product + BE 409. Dialog states: dish leaves the menu; existing orders unaffected.

B7Server validation maps to fields

409 name-conflict → inline on the name input; VALIDATION_FAILED details map by field.

B8Search + filter, client-side, URL-owned

?q=, ?cat= — same call as the customer menu; the ref shipped no search.

B9Prices via formatVND()

Display 35.000 đ; entered as an integer with a đ affix. One formatter (FE rule 12).

B10Images relative + buildImageURL()

FE rule 14; fixed-aspect thumb so rows never reflow.

B11VN-first copy

One constants file; the reference's strings kept ("Còn hàng"/"Hết hàng", "Đã thêm sản phẩm", "Đang lưu…").

B12Worked example everywhere

"Bánh cuốn trứng vịt lộn" · BÁNH CUỐN · 55.000 đ · thứ tự 5 · topping "Hành phi".

09

Decisions & flags

✅ Admin list stays uncached — a manager must see their own write instantly.

✅ Writes are pessimistic (FE rule 4). The availability toggle is the tempting optimistic case and deliberately isn't — a silently-reverted toggle is the exact bug the ref shipped.

✅ Delete = admin+, everything else manager+, button disabled-with-reason.

✅ Client-side search/filter, URL-owned.

⚠ Delete guard is a business decision. Plan default: guard + 409 PRODUCT_IN_USE (the FE was already written for it; a dish vanishing mid-service is worse). Say so before AD-P3 if delete-anytime should win.

⚠ file_attachments is still a stub — AD-P4 promotes its columns to full spec in the same task.

💡 bulk availability toggle (out of v1 scope) · ❓ who deletes orphaned images (OPS phase).

Verify plan

BE: role matrix + cache DEL proof + the availability column flipping + guarded-delete 409 + upload rejects. FE: a screenshot per render branch + a create→edit→toggle→delete round-trip + one server-validation-to-field mapping. Cross-page: a toggle here removes the dish from GET /products.
