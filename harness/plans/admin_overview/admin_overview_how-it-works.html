Admin overview — how it works at runtime

Not what we will build (that is admin_overview_plan.html) but what happens when it runs: the board's first paint, a chef's status change repainting this screen with zero HTTP calls of its own, what the SSE hook actually does to the cache, the write path through every layer including the rejection rail, cash payment, and what happens when the stream drops.

The one sentence that explains this page: the board is a reactive projection of a single TanStack Query cache entry — ['orders','live']. One SSE stream mutates that entry; every zone reads a filtered slice of it. Understand that sentence and the rest is detail.

01

Big picture — the whole machine on one screen

Four humans, four surfaces, one database, one message bus. The overview board is the manager's window; the chef has his own; the guest has a third. Nobody talks to anybody directly — every arrow between two browsers goes through the BE and Redis.

Traffic kinds on this page — only three

KindWhatHow often
ReadsGET /tables · /orders/live · /orders/history (lazy) on mount, then a 15 s/60 s safety-net poll; history only on accordion open
The streamGET /sse/admin — one long-lived connection continuous; a delta per write anywhere in the restaurant
WritesPATCH /orders/:id/status · POST /payments only on a manager click
02

First load — from URL to a painted board

This is a CSR page (unlike the customer menu's RSC prefetch): there is no SEO reason to server-render a staff dashboard behind an auth gate, and the data is realtime anyway. Everything hinges on the three queries defaulting to [] so the shell paints immediately.

What is deliberately not fetched on mount

  • ['orders','history'] — enabled: accordionOpen. Zones E and F stay collapsed and cost zero bytes until a manager actually opens one. They share the key, so opening Paid also warms Cancel.
  • Order line items for the logs — GET /orders/history returns items: [] by design; drill-down is a separate GET /orders/:id.

The upgrade over the reference. The old system had a single centered spinner at /admin/loading.tsx covering every /admin/* route, an AuthGuard that returned null (a genuinely blank window on hard reload), and StatCards that snapped from 0 with no skeleton. We render a layout-mirroring skeleton so the board's shape is on screen before its numbers are.

03

The live update — a chef's click repaints this board

This is the sequence that justifies the whole architecture. The chef advances an order on a different device, in a different app, and the manager's board updates — with no HTTP request originating from the manager's page at all.

The five delta types and what each does to the cache

DeltaEffect on ['orders','live']Visible result
new_orderfetch GET /orders/:id, prepend (dedup by id)row appears in Zone B + NewOrderPopup opens
order_status_changedpatch the order; drop it if the new status leaves ACTIVEbadge changes, or the row leaves the board
item_progresspatch item.qty_servedthe "Còn lại" column counts down live
order_cancelleddrop from the arrayrow vanishes; reappears in Zone F on next open
payment_successdrop from the arraytable flips to Trống; Zone E gets a row

Why this list is exactly five. The reference's WS hook also had case 'order_updated' and case 'order_completed' — live switch branches for events the backend never published (defect D2). Dead branches are worse than missing ones: they read like working features. Our hook handles the five types the BE actually emits, and a delta type the hook does not know is logged, not silently swallowed.

04

Inside the stream hook — how SSE + setQueryData actually work

If you have not wired a server-push cache before, this is the mechanism. It is much simpler than it looks, and it is the same shape as Zustand's set(): an external event writes to a store, and everything subscribed to the changed slice re-renders.

Server-Sent Events in one paragraph

SSE is a plain HTTP GET that never finishes. The server holds the connection open and writes text frames (event: … / data: …) whenever it has something to say; the browser's built-in EventSource parses them and fires an onmessage handler. It is one-directional (server → client), it rides ordinary HTTP so cookies and role middleware "just work", and the browser reconnects automatically when it drops. That one-directionality is exactly right here: this board only ever receives deltas — it sends its writes over normal PATCH/POST calls.

Why setQueryData and not invalidateQueries

invalidateQueries(['orders','live'])     setQueryData(['orders','live'], patch)
  → marks stale                            → writes the new array straight in
  → triggers a REFETCH                     → NO network call
  → full list over the wire, every delta   → O(1) patch of one order
  → on a busy floor: a request per event   → the board scales with the restaurant

We use setQueryData for all five deltas. invalidateQueries is used exactly ONCE,
and correctly: after a successful payment, to refresh ['orders','history'] —
a list we do not have a delta for and rarely read.

The mental model. ['orders','live'] is a little in-memory database of the restaurant's current state. The SSE hook is a replication log applying deltas to it. The zones are views over it. Once you see it that way, every question ("where does this number come from?", "why did that row vanish?") has one place to look.

Across pages — the durable handoff is the database row

The customer menu persists a cart slice, an activeOrderId, an order cache. This page persists nothing: no localStorage, no persisted store, no id in the URL. It is an always-on dashboard, not a wizard. Everything in memory dies on F5 and is rebuilt from two GETs.

06

The write path — a status advance through every layer

The manager clicks Xác nhận on #BC-43. Two rails are drawn: the happy path (blue/green) and the rejection path (red) that the reference got wrong.

The reference's two mistakes on this exact path — both designed out.

1. The unreachable-by-design button. The old UI rendered a red Huỷ ✕ on delivered rows. delivered → cancelled is not in the transition map, so that button 409'd 100% of the time — and the catch printed one generic toast ("Không thể cập nhật trạng thái"), so nobody could tell a business rule from a network blip. It was flagged in three separate reference docs and still shipped. Ours: the row offers only valid transitions, so the button does not exist; and the error carries a specific message when a race does produce a 409.

2. Optimism with no rollback. The old handleAction applied the optimistic patch and, on failure, did nothing but toast — leaning on the 15 s refetch to fix the board. So a rejected transition flashed the wrong status for up to fifteen seconds. Ours rolls the patch back in the catch block.

07

Cash payment — the one place money moves

Payment is the only pessimistic write on the page (FE rule 4: money is never optimistic). It is also where the reference leaked a false impression of client authority.

manager clicks "Đã thanh toán 💰" on Bàn 03 / #BC-43
  │
  ├─ PaymentModal opens — total 70.000 đ
  │    ☐ Khách đã đưa tiền          ← two independent human confirmations
  │    ☐ Nhân viên đã nhận đủ tiền
  │    [Xác nhận]  ← ENABLED ONLY WHEN BOTH ARE TICKED
  │
  ├─ POST /payments { "order_id": "ord43…", "method": "cash" }
  │        ▲ NO amount field. Ever. The server reads order.total_amount.
  │          (The reference sent amount: 42000 and the BE silently ignored it —
  │           a phantom field implying the client sets the price. Defect D3.)
  │
  ├─ BE: order must be ready|delivered → insert payment (pending)
  │      → completePayment() → status=completed, paid_at=NOW()
  │      → MarkOrderPaid()   → orders.status: delivered → paid
  │      → PUBLISH payment_success
  │
  └─ 201 { id, pay_url: null, qr_code_url: null }        ← cash completes inline
       │
       ├─ drop the order from ['orders','live']   → Bàn 03 flips to ● Trống
       ├─ invalidateQueries(['orders','history']) → Zone E will show it on open
       └─ toast "Đã thu tiền thành công"; modal closes

  on error: modal STAYS OPEN with the envelope message, order untouched,
            confirm re-enabled. Never a full-page overlay.

Why the amount never crosses the wire. It is the same rule as the customer menu's order payload carrying no prices: the server is the only authority on money. A client-sent amount is either ignored (confusing) or trusted (a vulnerability). The reference chose ignored; we choose absent.

⚠️ flag  v1 is COD, so only the cash branch is built now — pay_url/qr_code_url stay null until the P phase wires gateways. See admin_overview_PLAN.md §7.

08

When the stream drops — degradation, not silence

A restaurant's wifi is not a datacentre's. The design assumption is that the stream will drop, and the board must degrade honestly rather than quietly showing stale numbers.

stream state   UI                                   data freshness
────────────   ──────────────────────────────────   ─────────────────────────────
null           skeleton zones (connecting)          queries may still be in flight
(connecting)   NOT a blank window

true           Live dot green, no banner            realtime — deltas apply instantly
(open)

false          sticky red ConnectionBanner:         the 15 s / 60 s staleTime polls
(dropped)      "Mất kết nối thời gian thực —        are now the ONLY freshness source;
               đang thử lại…"; Live dot grey;       the board is at most ~15 s stale
               board KEEPS its last data            and it SAYS SO on screen

reconnect      EventSource auto-reconnects with     on reopen, the next poll reconciles
               exponential backoff (cap ~30 s)      any delta missed while down

The two-layer safety net. The SSE stream is the primary freshness source and the short staleTime poll is the backstop — neither alone is trusted. If the stream dies, the poll keeps the board within ~15 s of truth and the banner tells the manager to treat it as degraded. If the poll is slow, the stream has already pushed the delta. The reference had this poll too, but used it to paper over a missing rollback (§06) rather than as an explicit fallback tier.

One gap we inherit and must close at build time. The reference's SSE hook had no disconnect UI at all — only the WS channel drove the banner. So when the doorbell channel died, new orders silently stopped popping up and nothing on screen said so. Consolidating to one stream (§04) makes this structurally impossible: there is a single connected flag, and it drives both the banner and the Live dot. The AD-2 AC must include a stream-down screenshot.

What a hard refresh (F5) recovers

F5 → AuthGuard re-verifies the cookie → RoleGuard(manager)
   → GET /tables + GET /orders/live      ← full board state, rebuilt from the DB
   → GET /sse/admin reopens              ← deltas resume
   ✓ recovered: every order, status, table, total
   ✗ lost (by design): 🔍 kiemTraIds, checkedTableIds, prepPreviewIds,
                       searchQuery, viewMode, accordion open state

Nothing important lives in the browser, so nothing important can be lost there.
