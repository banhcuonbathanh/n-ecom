Admin overview page — FE + BE build plan

One plan for the staff-facing /admin/overview — the live floor command centre and the default admin landing page. Where the customer menu is cached reads + a client cart, this page is a realtime projection of active orders: one SSE delta stream mutating one TanStack Query cache, read by every zone on the board.

The core loop — one order's life, watched live:

① New order arrives over the stream → NewOrderPopup → manager confirms (pending → confirmed). ② Chef advances confirmed → preparing → ready on /staff/kds — this board updates with no HTTP call of its own. ③ Manager marks ready → delivered. ④ Cash payment (delivered → paid) frees the table to Trống. ⑤ The order drops off the live board into the Zone E paid log.

Worked example used in every doc + screenshot: floor 3 / 8 bàn · Bàn 03 · order #BC-43 · Bánh cuốn trứng ×2 + Trứng ×1 = 70.000 đ · 3 món.

No Zustand, no Redux, no context on this page. Server-state (3 queries) + page.tsx useState + one SSE hook is the whole model. page.tsx is the single bridge: it owns every query, derives the filtered arrays, and passes them down as props; writes flow up through onAction callbacks. Zero peer-to-peer widget state.

Contrast the customer menu, which needs two Zustand stores — because there the cart is client-owned state with no server twin. Here every fact on screen is server state or a pure derivation of it, so Zustand would be copying Query data into a store (FE rule 1's "never").

Realtime propagation — ONE path, not two

GET /sse/admin  (cookie JWT · AtLeast manager)  ──→  useAdminStream
   new_order            → GET /orders/:id → prepend to ['orders','live'] + setPopupOrder
   order_status_changed → patch, or DROP when the new status leaves ACTIVE
   item_progress        → patch item.qty_served (the "Còn lại" column counts down live)
   order_cancelled      → drop from ['orders','live']
   payment_success      → drop from ['orders','live']

the reference ran TWO channels: an SSE doorbell (manager, Bearer header) AND
a WS board (?token=, NO role gate — any customer JWT connected). Collapsed to
one cookie-JWT SSE stream: the SEC hole and the 2 dead switch cases both vanish.

Write path — optimistic with rollback (the reference had none)

handleAction(id, next)
  ① add id → loadingIds            (buttons disable, row shows '…')
  ② await PATCH /orders/:id/status
       success → setQueryData patch, reconciled by the RETURNED order object
       error   → ROLLBACK the optimistic patch + toast the SPECIFIC ApiError.message
  ③ finally → remove id from loadingIds

reference: optimistic write, NO rollback — it leaned on the 15 s refetch, so a
rejected transition flashed the wrong status on the board. We roll back — and
buttons offer only VALID next transitions, so the reject path is unreachable
in normal use anyway (§06 · B3).

Cross-page: the durable handoff is the DB row, not the browser

  • Nothing persists locally. No localStorage, no persisted store, no URL id — the page is an always-on dashboard, not a wizard. All three caches are memory-only and die on F5, rebuilt from GET /orders/live + GET /tables.
  • The handoff is the order row. PATCH …/status and POST /payments write MySQL, then publish one delta to Redis, which fans out to every other surface — /staff/kds, other overview tabs, and the customer's /order/:id. Browser → BE → Redis → browser; never browser → browser.
  • Does not survive F5 (by design): kiemTraIds, checkedTableIds, prepPreviewIds, searchQuery, viewMode, accordion open state.

BE contract — endpoints, schema, and why there is no cache map

Endpoints the page consumes (§3.1 — all under /api/v1)

#RouteAuthPhaseBehavior
1GET /tablesstaff (AtLeast cashier)AD id, name, capacity, status, is_active, qr_token; soft-delete filtered, ORDER BY name. Not cached.
2GET /orders/livestaff (AtLeast cashier)AD Active orders status IN (pending,confirmed,preparing,ready,delivered), ORDER BY created_at ASC, enriched with items[] + table_name in one batched fetch — no N+1 (the reference did N+1 per order). Feeds the shared bus ['orders','live'].
3GET /orders/historystaff (AtLeast cashier)AD status IN (paid,cancelled) AND DATE(updated_at)=CURDATE(), newest first. Returns items: [] deliberately — the logs show totals only (§09).
4GET /orders/:idstaff; customer restricted to own table_idO Full order + items with derived item_status (pending/preparing/done from qty_served vs quantity — never stored). Fired on the new_order delta before the popup.
5PATCH /orders/:id/statusstaff (AtLeast chef)R Body {status}; validates the transition map below; writes in a tx; publishes one delta. Invalid → 409 INVALID_STATUS_TRANSITION with a specific message (the reference's generic toast hid the cause). Returns the updated order.
6POST /paymentsstaff (AtLeast cashier)P Body {order_id, method} only — no amount (server reads order.total_amount). Order must be ready/delivered. cash: synchronous complete → MarkOrderPaid (delivered → paid) → publishes payment_success.
7GET /sse/admin (SSE)cookie JWT + AtLeast managerR The single realtime channel. Emits every board delta — new_order, order_status_changed, item_progress, order_cancelled, payment_success — plus a 15 s keep-alive. Replaces the reference's SSE doorbell and its ungated WS /ws/orders-live?token=.

The order state machine (§3.4) — the contract both sides validate against

pending ──→ confirmed ──→ preparing ──→ ready ──→ delivered ──→ paid
   │            │             │                                  (terminal)
   └────────────┴─────────────┴──→ cancelled                     (terminal)

cancel is valid from pending / confirmed / preparing ONLY.
delivered → cancelled DOES NOT EXIST — the reference rendered a Huỷ button
that called it anyway, so it 409'd every single time (defect D1, §07).
We do not add the transition; we remove the button.

There is no cache map — and that is the point. Orders are never cached (BE_STATE.md §7). Every read here hits MySQL directly; Redis is pub/sub only on this page — the SSE fan-out bus, not a cache-aside store.

So the customer-menu plan's entire §3.3 invalidation table collapses to one rule. Freshness comes from (a) the SSE push and (b) short staleTime safety-net polls — never from a TTL'd blob that someone forgets to DEL.

06

Behaviors spec (§4.4) — the AD / R / P acceptance criteria

B1Board = a projection of one cache

Every zone reads the same filtered slice of ['orders','live']; the SSE stream mutates that cache via setQueryData. No zone owns its own copy of an order.

B2New-order doorbell

SSE new_order → GET /orders/:id → NewOrderPopup (order #, table, kitchen items, total). Xác nhận PATCHes pending → confirmed; Bỏ qua writes nothing.

B3Transition-validated buttons

Each row offers only the valid next status from the machine; cancel appears only on pending/confirmed/preparing — never a Huỷ on a delivered order (D1 designed out). Delivered rows offer only Đã thanh toán 💰.

B4Search spans B + D

Client-side filter on order_number / id / customer_name / table name — no debounce, no refetch. Header shows "N đơn · M bàn phù hợp".

B5KPIs are derived

From orders with a real table_id + tables + now: occupied/total, Σ pending, Σ preparing, and urgent/warning as a ratio (>20′ / 10–20′), red when urgent > 0. None stored.

B6Zone B is pending-only

A confirmation queue, not a multi-status board (fixes D5). Urgency-bordered row, order #, elapsed, remaining count, 🔍 Kiểm tra + one confirm button. item_status derives from qty_served.

B7Zone D composition

DishSummaryStrip (floor aggregate + +Δ from 🔍) → ConfirmedPrepList (confirmed orders, excl. canh/giò, TỔNG row) → TableList/TableGrid toggle. Both views can pay and cancel — the reference wired only the list.

B8Zone C is conditional

PrepPanel renders only when kiemTraIds.size > 0; aggregates remaining dishes of 🔍-marked pending orders. Canh (♨) pinned last with a có/không-rau breakdown.

B9Cash payment gate

PaymentModal shows the total + two checkboxes ("Khách đã đưa tiền" + "Nhân viên đã nhận đủ"); confirm enabled only when both are ticked. Posts {order_id, method:'cash'} — no amount. On 201: drop from live, invalidate history, table → Trống.

B10Logs are lazy accordions

Zones E/F start collapsed; badge counts populate only after first open (enabled: open). They share one query key, so opening one warms the other.

B11Connection banner

Stream down → sticky red bar; the board holds its last data; auto-reconnect with backoff; the Live dot goes grey. Never a silent failure.

B12Role-gated, one channel

Page is manager+; endpoints are role-scoped (FE rule 13); the realtime feed is a single cookie-JWT SSE stream — no ?token= WS, no second channel.

B13Worked example everywhere

Docs, seeds and screenshots all use: floor 3 / 8 bàn · Bàn 03 · #BC-43 · Bánh cuốn trứng ×2 + Trứng ×1 = 70.000 đ · 3 món — fixing the reference's own numeric drift across its wireframes.

✅ One SSE channel, cookie JWT, manager-gated — supersedes the reference's SSE + WS split and its ungated ?token= WS. A dashboard only receives deltas, so a single server→client SSE stream is the right primitive; WebSocket's bidirectionality was never used.

  • ✅ Orders never cached; Redis is pub/sub only here (BE_STATE.md §7) — there is no cache-invalidation map to get wrong. The menu plan's §3.3 has no analog on this page.

  • ✅ Optimistic status advance + rollback; pessimistic payment (FE rule 4). Status advance is high-frequency floor UX and deserves the snappy path; rollback + valid-only buttons make it safe. Money is never optimistic.
