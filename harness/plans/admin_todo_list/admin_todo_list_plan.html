Admin to-do list — FE + BE build plan

The manager's plan-the-shift screen at /admin/todo-list: read the day's rollup, drill into one person, and — the part no other page owns — author tasks. Manager-only, pull-only, no Redis.

source of truthadmin_todo_list_PLAN.md

The one thing to understand about this plan: this page is the sibling of /admin/staff/task-board (F-22) and shares its entire data layer. The staff_tasks table, endpoints 1–3 + 5, the status machine and the cache map are owned by admin_task_board_PLAN.md and DB_SCHEMA.md §4.7.

This plan links them and re-derives nothing. It owns four things: the two-view shell, the filter bar, the authoring modal, and the one new endpoint authoring requires — PATCH /admin/tasks/:id.

note  Endpoint 4 (PATCH /admin/tasks/:id/status) exists in the contract but this page never calls it — status advancement is the task-board's verb. Numbered here only to match the sibling plan.

Why endpoint 6 lands with this page

F-22 deferred field-editing as "a pure additive follow-up" — correct for a board that only advances status. But this page's modal is create and edit.

The reference shipped that edit button wired to POST, so every edit silently inserted a duplicate — with no delete button to undo it. Shipping the button without the endpoint reproduces the bug; shipping the page without the button leaves it a create-only form the sibling board already outclasses. So the endpoint lands with the page that needs it.

Endpoint 6 spec — owned here

  • Body — partial: name? description? priority? dueAt? dueTimeStart? dueTimeEnd? notes?; absent = unchanged
  • Not settable — status (endpoint 4 owns it) · assignedBy (JWT, immutable) · assignedTo (ruling below)
  • Validation — same tier-1 as create; empty body → 422 VALIDATION_FAILED
  • Missing / soft-deleted → 404 NOT_FOUND
  • Response — 200 + full updated task
  • Side effects — updated_at only; no completed_at write

Schema — adds nothing

staff_tasks was already promoted into DB_SCHEMA.md §4.7 by F-22. Endpoint 6 writes only existing columns.

staff_tasks · written by PATCH
  • title VARCHAR(200)
  • description TEXT NULL
  • priority ENUM
  • due_at DATETIME
  • due_time_start/end TIME NULL
  • notes TEXT NULL
  • updated_at DATETIME

Ruling — assignedTo is not patchable

Re-assigning a task is a different intent from correcting its fields, and it silently moves a row between two people's boards. v1 = cancel + re-create. Revisit if the owner asks for hand-off. Simplicity first — recorded so it is a decision, not an omission.

Still deferred — DELETE /admin/tasks/:id

The deleted_at column exists from day one (F-22 §3.2), so soft delete stays a pure additive follow-up. Not shipped because nothing in this page's flow requires it once edit works correctly — the reference only needed a delete to clean up the duplicates its broken edit created. Fix the cause, not the symptom.

Cache map — one row added

Server cache: none. ARCHITECTURE.md §4 scopes Redis to the catalog cache and auth rate-limiting; task reads are low-volume, manager-only, and must never be stale against a write the same manager just made. Client keys/staleTimes are F-22 §3.3. This plan adds:

WriteInvalidates
PATCH /admin/tasks/:id['admin','tasks',staffId,date] and ['admin','tasks','stats',date] — for both the task's old dueDate and the currently-filtered date

Both dates, always. F-22 designed this rule out of the reference for create; editing makes it sharper — a patch that moves a task's due date changes the KPI rollup on two different days. Invalidating only the new date leaves yesterday's board stale.

06

Behaviors — these become the acceptance criteria

B1Lands on View A

No staff selected → stats board. The tasks query does not fire (enabled: !!staffId).

B2Drill-down flips the view

Clicking a per-staff row sets filters.staffId → View B → tasks query fires for the first time; the dropdown reflects it.

B3Clearing returns to View A

Deselecting the staff member restores the stats board.

B4Create round-trip

+ Tạo công việc (manager+ only) opens create mode; submit POSTs, closes on 201, KPIs and list refresh with no reload.

B5Edit updates in place

✏️ opens edit mode pre-filled; submit PATCHes endpoint 6 and updates the row — the list length does not change. The reference's 🔴 Bug 1, inverted into an AC.

B6Moving a due date updates both days

Editing a task to another date refreshes the KPI rollup on both the old and the new date.

B7Status filter actually filters

Narrows the visible list, including in_progress. Every option changes what renders.

B8Date change refetches both views

Changing the date drives stats and list to that day.

B9RBAC holds on both sides

Non-manager is blocked by RoleGuard; affordances absent; a forged call gets 403.

B10Every branch is distinct

Each loading / empty / error branch in §4.3 is reachable and visually different from the others.

B11description + notes persist

Both are entered in the modal, stored, and rendered back. In the reference notes was permanently undefined.

B12404 surfaces inline

A PATCH against a deleted task shows an inline modal error, not a blank.

09

Decisions, flags & verify plan

✅ Decided

  1. Endpoint 6 ships with this page — F-22 deferred it; authoring is this page's purpose and the button is unshippable without it.
  2. assignedTo is not patchable — re-assignment is a separate intent; v1 = cancel + re-create.
  3. DELETE stays deferred — it existed in the reference's backlog only to clean up duplicates that endpoint 6 now prevents.
  4. Date range removed, not implemented — matches the real single-day server contract.
  5. Status filtering is client-side — the day's rows are already loaded; a server round-trip would add a param to two endpoints for zero benefit at this data volume.
  6. Hooks and TaskStatusBadge are shared with the task-board, not copied.

⚠️ FLAG 1 — this page and admin_task_board are ~80% the same page.

Both are manager+, read the same four endpoints, render the same KPI rollup and the same per-staff drill-down. The only capability unique to /admin/todo-list is task authoring; the only one unique to /admin/staff/task-board is status advancement. Two pages, one dataset, split by verb.

Options — (a) ship both as planned, two entry points split by verb · (b) merge into one /admin/tasks screen with both verbs · (c) demote one to a link.

Recommendation: (b) — one screen, both verbs, half the FE surface, and no "which page do I open?" for the manager. Both plans build the identical contract, so this is a FE-only decision that can wait until AD opens — no work is blocked and nothing is wasted either way. Owner ruling requested before AD-4.

⚠️ FLAG 2 — filters are component state, not URL state. See §03.

💡 SUGGESTION — the KPI cards should link, not just count. Clicking "Quá hạn 2" to filter the board to overdue staff is one setFilters call and turns a static readout into navigation. Deferred to AD-4; noted so it is not lost.

❓ CLARIFY — is overdue a filterable status? It is derived, not stored (F-22 ruling), so filtering by it means isOverdue == true, not status == 'overdue'. The UI should present all 5 chips uniformly; B7 assumes yes. Flagged because it is a contract subtlety a future implementer could get wrong.
