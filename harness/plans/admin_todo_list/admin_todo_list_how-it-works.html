Admin to-do list — how it works at runtime

Not what we will build but what happens when it runs: the morning load, the view switch, the two write paths, and how a page with no realtime and no server cache still stays fresh.

The mental model in one line: /admin/todo-list is a plan-the-shift screen — read a per-staff board, author tasks that land as plain MySQL rows, drill into one person. All manager+, all pull-only.

02

First load — 07:45, the manager opens the page

1
Route tier — loading.tsx

New vs the reference, which had none: navigating in from another admin page showed a blank flash. The route skeleton covers the JS-bundle gap.

2
Two queries fire in parallel, one does not

stats and staff go immediately. The tasks query is interaction-gated — it has no staffId yet, so it never leaves the browser.

3
Gin gates before the handler runs

authMW + AtLeast("manager"). A guest, customer, chef or cashier JWT is rejected 403 before any query. The FE's canCreate is chrome; this is the gate.

4
One day, one query

Every task read is WHERE DATE(due_at) = ?. There is no BETWEEN — which is exactly why the reference's date-range filter was decorative and why this plan removed it.

5
Render View A — and only View A

The two views are mutually exclusive. With no staff selected there is no task list in the DOM at all.

Three branches the reference collapsed into one. Its stats fetch rendered {data ? (…) : null} — so a failed fetch and an empty day both produced a blank page section, and a failed task fetch rendered the empty state, actively lying to the manager. Here: loading → skeleton, error → panel + Thử lại, empty → "Chưa có công việc nào cho ngày này" + a create CTA. Three states, three renders.

03

The core client action — the A/B switch, zero new machinery

The page's defining interaction costs one setState. There is no router navigation, no store, no prop-drilling — the view is a pure function of one field.

07:49 — the manager clicks chef01's row in the stats board

  onClick ──→ setFilters(f => ({ ...f, staffId: 'chef01-uuid' }))
                 │
                 ├─ staffId != null ──────→ VIEW B replaces VIEW A        (render branch)
                 │
                 ├─ useStaffTasks(staffId, date)
                 │     enabled: !!staffId  ── false → true
                 │     └─→ GET /admin/tasks?staffId=chef01&date=2026-07-19   (FIRST fetch)
                 │
                 └─ filter-bar dropdown shows "chef01"
                       └─ DERIVED from filters.staffId — not a second copy of the selection

clearing the selection → staffId = null → VIEW A returns, already cached (staleTime 30s)

Why the dropdown is derived, not synced. The reference kept a local dropdown value and pushed the stats-row click into it with a useEffect — two sources for one truth, kept in step by a side effect. Here the <select> reads filters.staffId. There is nothing to sync, so there is nothing to desync (FE_STATE.md §9 derived-not-stored).

1
First click pays the network cost

The tasks query has never run, so this is a cold fetch → TodoTasksSkeleton.

2
Going back is free

View A's data is still in the Query cache within its 30 s staleTime — instant, no skeleton.

3
Re-selecting the same person is free

['admin','tasks',staffId,date] is already cached for 15 s.

4
Status filtering never touches the network

The day's rows are already in hand, so the filter runs client-side over cached data — including in_progress, which the reference's dropdown omitted entirely.

04

Inside the authoring modal — one form, two verbs

This is the page's reason to exist and the place the reference broke worst. The modal is a single component with a mode; everything else follows from it.

TaskFormModal
├── mode: 'create' | 'edit'          ← the only branch that matters
├── RHF + Zod form state             ← field values live here, never in the parent
│     name*  assignedTo*  priority*  dueAt*  dueTimeStart?  dueTimeEnd?
│     description?  notes?           ← notes NOW HAS AN INPUT (reference had none)
└── submit
      ├── create → POST  /admin/tasks        → 201 → new row
      └── edit   → PATCH /admin/tasks/:id    → 200 → SAME row, updated
                    └── body = changed fields only; absent = unchanged
                    └── assignedTo is LOCKED (not patchable — see below)

🔴 The reference's Bug 1, drawn. Its handleModalSubmit called createTask.mutate(...) regardless of mode — it never read editTask.id:

edit "Kiểm kê"  →  POST /admin/tasks  →  201 Created  →  list now has 2 near-identical tasks  →  no delete button exists

Every edit silently forked the row. The manager's only recourse was to ignore it. That is why PATCH /admin/tasks/:id ships with this page rather than staying deferred — the button is unshippable without it.

Why assignedTo is locked in edit mode

Patching the assignee would silently move a row between two people's boards — a task vanishes from one manager's view of chef01 and appears under cashier01, with no trace of the move. Correcting a typo and re-assigning work are different intents with different blast radii. v1: cancel + re-create. Revisit if the owner asks for hand-off.

What a partial PATCH actually sends

the manager opens ✏️ on "Kiểm kê nguyên liệu" and changes only the time

  form dirty-fields ──→ { dueAt: "2026-07-19T14:00:00Z" }      ← ONLY what changed
                            │
                            ▼
  UPDATE staff_tasks SET
      title       = COALESCE(?, title),          ← NULL → keep
      priority    = COALESCE(?, priority),
      due_at      = COALESCE(?, due_at),         ← 14:00 → change
      notes       = COALESCE(?, notes),
      updated_at  = NOW()
  WHERE id = ? AND deleted_at IS NULL            ← soft-deleted rows are invisible → 404

  NOT touched: status (endpoint 4 owns it) · assigned_by (JWT) · completed_at

06

The write path — PATCH through every layer

0
Tier 0 — the form

Zod blocks an empty name, a bad priority, a malformed date, and an entirely unchanged submit before a request exists.

1
Gin gate, again

AtLeast("manager") runs before the handler. Writes are gated identically to reads.

2
Handler binds, service decides

Per BE_STATE.md: handlers do shape, services do business. The handler never touches SQL.

3
Partial update via COALESCE

Absent fields pass NULL and keep their stored value. No read-modify-write, so two managers editing different fields of the same task do not clobber each other.

4
Zero rows is a 404, not a 500

deleted_at IS NULL is in the WHERE, so a soft-deleted task is simply invisible → 404 NOT_FOUND, surfaced inline in the modal (behavior B12). The reference's habit of returning 500 for caller error is designed out (F-22 §3.4).

5
Success invalidates two keys across two dates

If the edit moved the due date, the KPI rollup changed on both days. Invalidating only the new date leaves yesterday's board stale — the sharpened form of a bug F-22 designed out for create.

6
No optimistic update

Per FE_STATE.md, optimistic is cart-only. The modal disables its button and waits for the 200 — a manager must never see an edit that did not land.

07

Freshness without Redis and without realtime

Most pages in this system have a cache tier or a push channel. This one has neither — and that is a decision, not a gap.

LayerHereWhy
RedisnoneARCHITECTURE.md §4 scopes Redis to the catalog cache + auth rate-limiting. Task reads are low-volume and manager-only, and must never be stale against a write the same manager just made
SSE / WSnoneNothing on this screen is time-critical to a second. Contrast the order pages, which genuinely push
TanStack Querythe only cachestats 30 s · tasks 15 s · staff 5 min
  staleness bound = the staleTime of whichever key you are looking at

  own write ─────→ invalidate ─────→ 0 s stale          (the case that matters)
  someone else's write ──────────→ ≤ 30 s (stats) / ≤ 15 s (tasks)
  tab regains focus ─────────────→ refetch-on-focus → fresh

  worst case: a manager stares at a board for 30 s while a colleague adds a task.
  For a shift-planning screen this is correct; for the kitchen display it would not be.

Skeletons must not flash on a refresh. Background refetches set isFetching, not isLoading. Cached data stays on screen until the new data lands — so a staleTime expiry is invisible, while a genuine cold load still gets its skeleton. Binding a skeleton to isFetching would make the board strobe every 30 seconds.
