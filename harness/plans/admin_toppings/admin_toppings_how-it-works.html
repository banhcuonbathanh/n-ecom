Admin toppings — how it actually works

01

The big picture

The whole machine on one screen. A topping write touches one DB row, optionally N junction rows, and deletes three sets of Redis keys. Nothing is pushed.

      ADMIN BROWSER (this page)              SERVER                    EVERYONE ELSE
 ┌────────────────────────────────┐                              ┌──────────────────────────┐
 │ TanStack (browser-local)       │                              │ customer phones · POS ·  │
 │  ['admin','toppings'] 60s      │                              │ other admin tabs         │
 └───────────────┬────────────────┘                              └──────────────────────────┘
                 │ POST / PATCH / DELETE                                      ▲
                 ▼                        ┌───────────────────────┐           │ next fetch only
        ───────── THE WIRE ─────────────▶ │ toppings row     (1)  │           │ (no SSE / no WS)
                                          │ product_toppings (N)  │           │
                                          ├───────────────────────┤ ──────────┘
        cache fan-out ───────────────────▶│ Redis DEL:            │  served on next read
                                          │   toppings:list       │
                                          │   products:list       │
                                          │   product:<id> × N ✅  │
                                          └───────────────────────┘

Toppings have no realtime channel. The SSE/WS layer carries order events only. Topping changes propagate by pull — the next fetch on each surface, bounded by the 5-minute Redis TTL plus each client's staleTime. That is a deliberate accepted latency, not an oversight: catalog edits are rare and never time-critical.

The ✅ on product:<id> × N is the whole fix. The reference Del'd only the two list keys. Customer product-detail is the only reader of product:<id>, so it served a stale topping price for up to five minutes after every edit. We Del it too, using ids the write already has in hand.

02

First load — one query, four branches

The reference fired two queries here and gated only one of them. We fire one, and the second query's entire class of bug disappears with it.

1
Route enters

loading.tsx paints a table-shaped skeleton — header row plus six placeholder rows — so the layout is already the right shape when data lands. Nothing jumps.

2
Guards resolve

The admin shell's auth guard confirms the session, then the role guard checks manager+. A server/staff role stops here with the access-denied panel and ToppingsPage never mounts.

3
RSC prefetches on the server

page.tsx calls GET /toppings/all during render and hands the result down through HydrationBoundary. The client component receives a warm cache — first paint has real rows, not a spinner.

4
The server does the join

One SQL statement: toppings LEFT JOIN product_toppings LEFT JOIN products, filtered on both deleted_at IS NULL, aggregated per topping. LEFT so an unlinked topping still returns with an empty products array.

5
The table picks exactly one branch

loading · error · empty · ready — named, mutually exclusive, and never conflated. An error shows a retry panel, not an empty state that lies about there being no toppings.

What the reference did instead. Two parallel queries — toppings, and the entire product catalogue. Only the first had a loading gate. When toppings resolved first, every row rendered “Chưa gắn sản phẩm”, then snapped to real product names when the second query landed. The column visibly lied for a moment on every cold load.

03

The save round-trip

Create and edit share one modal and one path. The trip is pessimistic — the UI waits for the server, because a topping edit is not a cart tap and a wrong price is worse than a 200ms wait.

The response is the full object, never a thin {id} — the lesson-7 rule from OVERALL_PLAN.md §6. Because the row comes back complete, the client writes it straight into the list cache and the table updates without a second round-trip.

The modal resets on every open. React Hook Form keeps its values between mount cycles; the reference's own notes call stale-value leakage “the #1 modal bug in this project”. The reset is asserted by behavior 10, so it is a test, not a hope.

04

The delete transaction

Delete is admin-only and does two writes that must not half-happen — so they share one transaction, per BE_STATE.md.

1 readjunction ids
→
2 txsoft delete
→
3 txpurge junction
→
4 commit204
→
5 fan-outDEL × 3 sets
1
Read the linked product ids first

Inside the tx, before anything is destroyed. These ids serve two purposes: they are the rows to purge, and they are the product:<id> cache keys to Del in step 5. Read them after the purge and they are gone.

2
Soft delete the topping

deleted_at = NOW(). The row stays — every read filters deleted_at IS NULL. Past orders are untouched regardless, because they carry their own toppings_snapshot.

3
Purge the junction rows

The FK's ON DELETE CASCADE fires only on a hard delete, so a soft delete alone leaves the links orphaned forever. We delete them explicitly.

4
Commit, return 204

Both writes land or neither does. A crash between them cannot leave a live topping with no links, or dead links pointing at a live topping.

5
Fan out the invalidation

Using the ids captured in step 1. Cache failures are non-fatal — a dead Redis falls through to MySQL rather than failing the delete that already committed.

Why delete unlinks instead of refusing. Retiring a topping is ordinary manager work. Blocking it until every product is edited first would be hostile — and the customer surface is safe either way, since a soft-deleted topping vanishes from every read on the next fetch. So the dialog states the impact (“đang áp dụng cho 2 sản phẩm — xóa sẽ gỡ liên kết”) rather than standing in the way.

05

Cache fan-out — who sees the change, and when

One edit, five surfaces, no push. This table is the honest answer to “when does the customer see my new price?”

SurfaceReadsSees the change
This page/toppings/all (uncached)immediately — the response object is written into the cache
Other /admin/toppings tabssame, browser-local cacheon their own staleTime lapse (60s), refocus, or remount
Product form picker (/admin/products)['admin','toppings']on the shared-key invalidation — same browser, immediately
Customer /menu · POSproducts:listnext fetch after the DEL — cold, so straight from MySQL
Customer /menu/product/:idproduct:<id>next fetch — because we Del it; the reference made this one wait 5 min
Past ordersorder_items.toppings_snapshotnever — frozen at order time, by design

Snapshot safety is why any of this is allowed. Editing a topping's price does not rewrite history: each order froze its own copy of the name and price at the moment it was placed. So the fan-out above only ever affects live catalog reads — never money that has already been counted.

Staleness bound

worst case for a customer already sitting on /menu:

  edit committed ──▶ Redis DEL (instant) ──▶ customer's TanStack staleTime (≤ 5 min)
                                              └─▶ refetch ──▶ MySQL (fresh) ──▶ sees it

  worst case ≈ the client's own staleTime. The Redis TTL never adds to it,
  because the DEL already happened — the next read rebuilds from MySQL.
06

State across components & pages

Two hubs only, exactly as FE_STATE.md prescribes: the TanStack cache for server data and Zustand for the auth identity. Everything else is local useState that dies with the page.

                    ┌──────────────────────────────────────┐

CrossingMechanism
Component → component (header count, table rows, chips)all read the same cache entry — no prop drilling, no duplicated fetch
This page → the product form's pickerthe shared query key — a write here invalidates there
This page → customer surfacesthe server — Redis DEL, then each client's next fetch
Tab → tab, device → devicenothing direct. Caches are per-browser; the DB is the only shared truth

“Sync” here means “everyone re-reads the same row eventually.” There is no coordination channel between two managers editing at once — last write wins, and neither sees the other until a refetch. Acceptable for a low-contention back-office catalog; it would not be for stock levels.
