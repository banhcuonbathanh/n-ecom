Admin toppings — /admin/toppings

The back-office CRUD screen for toppings — the paid add-ons and the ₫0 nhân fillings the customer menu renders as pills. One table, one modal, five endpoints, and six reference defects designed out.

The one move that matters. The reference fetched the entire product catalogue — uncached, with an N+1 topping query per product — for the sole purpose of printing product names in one column. This plan replaces it with a single server-joined GET /toppings/all. That one change kills the N+1, the second query, and the column flash together.

The shared query key is deliberate. ['admin','toppings'] is also read by the topping picker inside the product form on /admin/products. A write here invalidates that picker too — which is exactly what we want. Do not invent a second key.

Cache map — the invalidation AC

WriteDEL keys
POST / PATCH / DELETE /toppings* toppings:list · products:list · every product:<id> joined to that topping

The third target is the fix. The reference Del'd only the two list keys, so customer product-detail served a stale topping price for up to 5 minutes. The joined ids come free from the junction lookup the write already performs.

Wire shape — GET /toppings/all

{ "data": [

DB field names throughout — price / is_available, never the reference's invented extraPrice / isAvailable. Writes return the full object, not a thin {id}.

06

Behaviors — the acceptance criteria

01Guarded entry

Non-manager never renders the page; logged-out redirects to login.

02Five columns

Tên · Áp dụng cho SP · Giá thêm · Trạng thái · actions. Sorted by name.

03Derived count

Badge reads data.length; shows 0 only in the empty branch, never while loading.

04Price by value

0 → “Miễn phí”; else +8.000đ via the one shared formatVND(). Never “+0đ”.

05Status badge

“Có sẵn” (success) / “Hết” (muted).

06Product chips

From server-joined products[]; empty → “Chưa gắn sản phẩm”. Read-only.

07Long names truncate

Tooltip carries the full name; table scrolls in its own box, page never does.

08Client-side filter

Name search + status chips. No refetch, no URL params, no pagination in v1.

09Add mode

Empty form, title “Thêm topping”, toggle defaults to Có sẵn.

10Edit mode

Same modal, pre-filled, title “Sửa topping”. Resets on every open.

11Inline validation

Name required; price ≥ 0 with 0 explicitly valid.

12Duplicate name

409 → field error “Tên topping đã tồn tại”; modal stays open.

13Save closes cleanly

Returned object written to cache; row appears/updates with no flash.

14Real delete dialog

States the linked count from the row's own products[]. Admin-only button.

15Idempotent delete

Row's Xóa disables while in flight — a double-click can't fire twice.

16Surfaces refresh

Menu, POS, product-detail update on next fetch; past orders never change (snapshot).

Reference defects designed out

#Finding (reference)Countermeasure
①Entire product list pulled, uncached, N+1 per product — to render one columnGET /toppings/all joins products server-side
②That query has no loading gate → every row flashes “Chưa gắn sản phẩm”Only one query remains — the flash cannot exist
③Loading = bare <p>Đang tải…</p>; layout jumps. No loading.tsxTable-shaped skeleton at both tiers
④Delete pending state never read — double-clicks re-firePer-row pending disables that row's Xóa
⑤Writes never Del product:<id> → 5-min stale price on product detailCache map Dels every joined product:<id>
⑥FE maps a 409 the BE can never send — duplicates insert silentlyReal UNIQUE(name) + 409 makes the branch live
⑦RHF modals retain stale values — “the #1 modal bug in this project”Reset on every open, asserted by behavior 10
⑧Delete confirm is a JS confirm(), count computed client-sideReal dialog; count from the row's joined products[]
⑨Create hardcodes is_available=1 — the toggle silently does nothingCreate honours the field
⑩Soft delete orphans product_toppings rows foreverJunction purged in the same tx
⑪Availability written by raw ExecContext outside sqlcOne sqlc statement sets all three columns

09

Decisions, flags & verify plan

✅ Decided

  • Topping CRUD stays inside the products domain — shared cache-invalidation logic.
  • Role split: read public · create/update manager+ · delete admin-only.
  • No pagination or server-side search in v1 — client-side, matching the menu plan.
  • “Áp dụng cho sản phẩm” is read-only here — linking stays on /admin/products.
  • Delete unlinks rather than blocks — the dialog states impact instead of refusing.
  • Query key ['admin','toppings'] is shared on purpose with the product picker.
  • Wire fields use DB names — never extraPrice / isAvailable.

⚠️ 1 — UNIQUE(name) is a schema addition this plan does not own. DB_SCHEMA.md §4.1 carries no unique key today, and adding one pulls in the §4.4 soft-delete×UNIQUE rule (rename to <name>#deleted-<id>, as staff.username does). DB_SCHEMA.md must be amended in C-1 — the 409 is unimplementable until it is.

⚠️ 2 — GET /toppings changes meaning. Making it available-only is a contract change; any future reader wanting the full list must use /toppings/all.

⚠️ 3 — the blank AuthGuard window is a shell-level gap every admin page inherits. Belongs to the admin-shell task; recorded so it isn't lost.

💡 Suggestions (deferred)

  • Bulk status toggle — with 23+ toppings, marking a batch “Hết” is one-by-one work today.
  • Stock-driven availability — toppings map to ingredients that /admin/ingredients (F-21) already tracks; auto-flipping is_available at zero would remove the manual toggle.
